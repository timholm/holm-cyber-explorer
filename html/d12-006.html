D12-006 — Backup Verification and Integrity Testing

Document ID
D12-006
Domain
12 -- Disaster Recovery
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
D12-001, D12-005, D6-002, D6-003, SEC-003
Depended Upon By
D12-002 (drill scenarios test verification), D12-003 (rebuild depends on verified backups), D12-008 (corruption recovery depends on known-good backups).

Purpose

This article defines how the institution proves that its backups are actually restorable. D12-005 defines what is backed up, where, and how often. This article ensures that those backups are not merely present but functional -- that they can be decrypted, decompressed, extracted, and used to recover the data they claim to contain.

D12-001 stated the principle bluntly: a backup that has not been tested is not a backup. It is a hope. This article converts that principle into specific, scheduled, repeatable verification procedures. It covers four layers of verification: media readability (can the storage medium be physically read), archive integrity (is the backup file structurally intact), content verification (does the backup contain the expected files with the expected checksums), and restoration testing (can the data actually be restored to a working state).

Verification is not optional. It is not something to do when there is spare time. It is a scheduled maintenance activity with the same non-negotiable status as the backups themselves. A backup system without verification is a system that discovers its failures at the worst possible moment: when the backup is needed for a real recovery.

Scope

In scope:

  • Four-layer verification model: media, archive, content, and restoration.
  • Verification schedules for each backup location and data tier.
  • Bit-rot detection and media degradation monitoring.
  • Test-restore drill procedures (coordinated with D12-002).
  • Verification logging and reporting.
  • Failure response procedures when verification reveals problems.

Out of scope:

  • Backup creation procedures (D6-002).
  • Backup strategy and architecture (D12-005).
  • Primary data integrity verification (D6-003; this article verifies backup copies).
  • Full disaster recovery drills (D12-002; this article provides the verification component that drills test).

Background

3.1 The Four Layers of Backup Failure

A backup can fail at any of four layers, and each layer requires its own verification method. A backup that passes one layer may fail at another. Comprehensive verification requires testing all four.

Layer 1: Media failure. The physical storage medium is unreadable. The drive does not spin up. The USB device is not recognized. The optical disc has surface damage. This is the most obvious failure and the easiest to detect, but it is also the most final: if the media cannot be read, the data on it is inaccessible.

Layer 2: Archive corruption. The media is readable, but the backup archive file is structurally damaged. The tarball is truncated. The compression stream is corrupted. The encrypted container has a damaged header. The data is physically present on the media but the logical structure that makes it usable is broken.

Layer 3: Content corruption. The archive is structurally intact and can be extracted, but the files inside have been corrupted. Checksums do not match. File contents have been altered by bit rot, write errors, or encryption defects. The file names are correct and the file sizes look right, but the actual data is wrong.

Layer 4: Restoration failure. The data is intact, but the restoration procedure fails. Configuration files reference hardware that does not exist. Service definitions assume software that is not installed. Database dumps cannot be imported because the schema has changed. The data is good but the procedures for using it are not.

3.2 The Cost of Undetected Failure

Each backup cycle, the institution's recovery capability is either confirmed or silently degraded. Without verification, degradation accumulates undetected. The operator believes the backup system is healthy because no errors have been reported. But absence of error is not presence of correctness. The media may be degrading. The archives may be subtly corrupted. The procedures may have drifted out of sync with the current system. Verification is the mechanism that converts silent degradation into visible, actionable findings.

System Model

4.1 Layer 1: Media Readability Verification

Media readability is verified by reading every sector of the backup medium and confirming that the device reports no errors.

For hard drives and SSDs:

  • Run SMART health diagnostics: smartctl -a /dev/[device]
  • Check for reallocated sectors, pending sectors, and uncorrectable errors.
  • Perform a full surface read: dd if=/dev/[device] of=/dev/null bs=1M status=progress

For removable USB media:

  • Connect the device and verify it is recognized by the operating system.
  • Mount the device and perform a full directory listing to confirm accessibility.
  • Check for I/O errors in the kernel log: dmesg | grep -i error

Schedule: Location A -- monthly. Location B -- quarterly. Location C -- annually (during the annual operations cycle, or whenever the media is accessed for refresh).

4.2 Layer 2: Archive Integrity Verification

Archive integrity is verified by testing the archive structure without full extraction.

  • For compressed tar archives: tar -tzf /path/to/backup.tar.gz > /dev/null
  • For encrypted containers: attempt to open the container with the correct key and verify the decryption completes without error.
  • For LUKS-encrypted volumes: cryptsetup open --test-passphrase /dev/[device]

Schedule: Every time a backup is created (automated as part of the backup script). Additionally, monthly for Location A, quarterly for Location B, annually for Location C.

4.3 Layer 3: Content Checksum Verification

Content verification compares the actual file contents against recorded checksums.

Procedure:

  1. Extract the backup to a temporary location (or mount it read-only).
  2. Locate the checksum manifest created during backup (per D6-002, every backup includes a SHA-256 manifest).
  3. Verify: cd /tmp/backup-verify && sha256sum -c manifest.sha256
  4. Record any failures. Any checksum mismatch indicates corruption.
  5. Clean up the temporary extraction.

Schedule: Location A -- monthly (spot-check: verify 10% of files selected randomly, plus all files over 100MB). Location B -- quarterly (full verification of all Tier 1 files). Location C -- annually (full verification of all files).

4.4 Layer 4: Restoration Testing

Restoration testing is the ultimate verification: actually restoring data from the backup to a test target and confirming that it is usable.

Procedure:

  1. Select a test target (spare drive, test partition, or temporary directory on a non-production volume).
  2. Follow the restoration procedure defined in D6-002 Section 4.4 exactly as written.
  3. After restoration, verify:
    • All expected files are present.
    • File permissions are correct.
    • Critical files are readable and contain expected content (open and inspect at least five files manually).
    • If database dumps are included, import them into a test database instance and verify data accessibility.
  4. Record the results in the verification log.
  5. Clean up the test target.

Schedule: Monthly for Tier 1 data from Location A. Quarterly for Tier 1 data from Location B. Annually for a full restore test from Location C (this annual test is coordinated with D12-002's annual full recovery drill). At least one full restore from Location C per year is mandatory per D12-001 R-DR-02.

4.5 Verification Logging

Every verification produces a log entry:

BACKUP VERIFICATION LOG

Date:           [date]
Location:       [A / B / C]
Backup Date:    [date of the backup being verified]
Data Tier:      [1 / 2 / 3 / all]

Layer 1 (Media):     [PASS / FAIL / SKIP] -- [notes]
Layer 2 (Archive):   [PASS / FAIL / SKIP] -- [notes]
Layer 3 (Content):   [PASS / FAIL / SKIP] -- [notes]
Layer 4 (Restore):   [PASS / FAIL / SKIP] -- [notes]

Findings:       [any issues discovered]
Action Required: [any corrective actions needed]
Operator:       [identifier]

Verification logs are Tier 2 documents per D6-001 and are stored in the operational records.

Rules & Constraints

  • R-D12-06-01: Backup verification must be conducted on the schedule defined in Section 4. Missed verifications must be rescheduled within 14 days.
  • R-D12-06-02: Every backup at every location must pass at least Layer 1 and Layer 2 verification annually. Layer 3 verification must be completed at least annually for all locations. Layer 4 (restoration) must be completed at least annually for Location C.
  • R-D12-06-03: Any verification failure at Layer 1 or Layer 2 triggers immediate corrective action: the affected backup must be replaced from primary data or from a verified backup at another location within 48 hours.
  • R-D12-06-04: Any verification failure at Layer 3 triggers investigation: determine the scope of corruption, identify the cause, and assess whether the corruption has propagated to other backup generations or locations.
  • R-D12-06-05: Verification logs must be maintained and reviewed during the quarterly operations cycle. Patterns of recurring verification issues must be escalated to a Tier 2 governance decision.
  • R-D12-06-06: The verification procedures themselves must be reviewed annually to ensure they remain compatible with current backup formats and media types.

Failure Modes

  • Verification complacency. Backups pass verification for years. The operator begins to treat verification as a formality, skipping steps or accepting partial results. The one time a backup is actually corrupted, the abbreviated verification misses it. Mitigation: the four-layer model provides structured rigor. The quarterly drill cycle tests actual restoration.
  • Verification-induced damage. The verification process itself damages the backup media (excessive read cycles on aging media, accidental writes to read-only media). Mitigation: verification uses read-only operations. Media is mounted read-only when possible. SMART monitoring detects increasing read errors before they become critical.
  • Verification scope creep. Verification takes progressively longer as data volume grows, eventually consuming more time than the operator can allocate. Verifications are skipped or abbreviated. Mitigation: the spot-check approach for Location A (10% of files) keeps routine verification tractable. Full verification is reserved for quarterly and annual cycles.
  • False confidence from partial verification. Layer 1 and Layer 2 pass, and the operator assumes the backup is good without testing Layers 3 and 4. But Layer 3 and 4 failures are the most common in practice. Mitigation: the verification schedule mandates all four layers on a defined cadence.

Recovery Procedures

  1. If Layer 1 fails (media unreadable): The backup on this media is lost. Assess whether other copies of the same data exist at other locations. Create a replacement backup immediately from primary data or from a verified backup at another location. Investigate the cause of media failure. If the failure is age-related, advance the media refresh schedule for all media of the same type and age.
  2. If Layer 2 fails (archive corrupted): Attempt partial extraction to recover what can be recovered. Create a replacement backup from primary data. Investigate: was the corruption present at creation time (backup script error) or did it develop after creation (media degradation, bit rot)? Address the root cause.
  3. If Layer 3 fails (content checksum mismatch): Identify the affected files. Determine whether the corruption is in the backup or in the primary data (compare against another backup generation). If the backup is corrupted, replace it. If the primary data is corrupted, initiate D12-008 data corruption recovery.
  4. If Layer 4 fails (restoration fails): The data may be intact but the restoration procedure is outdated. Update the restoration procedure to match the current system. Test the updated procedure. Document the discrepancy.
  5. If verification has lapsed for more than 6 months: Conduct an emergency verification of all locations at all layers. Prioritize Location C (the rebuild safety net). Document any gaps discovered. Resume the normal schedule.

Evolution Path

  • Years 0-5: Verification procedures are established and refined. Early verifications will reveal procedural gaps: checksums not generated during backup, archive formats that are difficult to verify non-destructively, media that degrades faster than expected. Use these findings to improve both the verification procedures and the backup procedures.
  • Years 5-15: Verification is routine. The verification log accumulates a history of media health, allowing the operator to identify degradation trends and proactively replace media before failure.
  • Years 15-30: Media technology transitions require updating verification procedures. New media types may require different tools and techniques. The verification log history helps predict which media types are most reliable under the institution's storage conditions.
  • Years 30-50+: The verification discipline, not the specific procedures, is the lasting contribution of this article. The tools and media will change. The principle -- verify everything, trust nothing, test regularly -- will not.

Founder Commentary

This section is reserved for dated entries by current and future operators.

2026-02-17 -- Founding Entry: I have a horror of the discovery that a backup is corrupted at the moment I need it most. I have seen this happen in professional environments: a team confidently initiates disaster recovery, retrieves the backup tape, and discovers that the last successful backup was from three months ago because verification was "someone else's job" and no one was doing it.

In a one-person institution, verification is always my job. There is no one else to blame and no one else to catch the failure. This article is my hedge against my own future complacency -- the version of me who, five years from now, is tempted to skip the monthly verification because "it always passes anyway."

The four-layer model exists because I once saw a backup pass archive integrity testing with flying colors -- the tarball was structurally perfect -- while containing files that were entirely corrupted at the byte level. The archive was intact. The data inside it was garbage. Layer 2 passed. Layer 3 would have caught it. But Layer 3 was never run. That lesson cost someone else their data. I intend for it to save mine.
