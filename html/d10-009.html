D10-009 — Automation Monitoring: Operational Procedures

Document ID
D10-009
Domain
10 -- User Operations
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
OPS-001, D10-002, D10-003, D10-005, D8-001, D8-006, D8-007, D8-009
Depended Upon By
D10-004 (automation maintenance tasks), D10-005 (automation failure as incident type), D8 (operational feedback on automation behavior).

Purpose

This article defines how operators monitor, verify, and intervene with automated systems during daily operations. Automation is the institution's force multiplier: it performs tasks that would be impossibly tedious for a human operator, it maintains vigilance during off-duty periods, and it executes procedures with a consistency that humans cannot match. But automation is also the institution's most dangerous liability if it operates without oversight.

The core principle of D8-001 (Automation Philosophy) is that no automated system operates without human oversight. This article translates that principle into operational practice. It defines what the operator watches, how often they watch it, what anomalies look like, when to intervene, and how to intervene safely. It is the operational complement to the Domain 8 design articles: where Domain 8 says how automation should work, this article says how the operator verifies that it actually does.

An automated system that runs correctly is invisible. An automated system that runs incorrectly may also be invisible -- until the damage it causes becomes apparent. The procedures in this article are designed to make the invisible visible, to catch automation failures before they compound, and to maintain the operator's understanding of what their automated systems are actually doing.

Scope

This article covers:

  • The automation inventory: what automated systems exist and what each one does.
  • Daily monitoring procedures for automated systems.
  • Anomaly recognition: what normal behavior looks like and how to identify deviations.
  • Intervention thresholds: when to let automation self-correct and when to intervene manually.
  • Manual override procedures and their safeguards.
  • Automation performance review: periodic assessment of whether automation is working as intended.
  • The relationship between automation monitoring and incident response.

This article does not cover:

  • The design or architecture of automated systems (see Domain 8 articles).
  • The creation or modification of automated tasks (see D8-006).
  • The automation override decision framework (see D8-009).
  • The maintenance schedule for automation systems (see D10-004 with D8-006 inputs).

Background

3.1 The Automation Paradox

Automation creates a paradox in operational monitoring. The more reliable an automated system is, the less attention the operator pays to it. The less attention the operator pays, the less likely they are to notice when the system begins to fail. And the later a failure is noticed, the more damage it has caused. This is sometimes called the "automation complacency" problem, and it is the central challenge this article addresses.

The solution is not to pay the same attention to automation as to manual tasks -- that would defeat the purpose of automation. The solution is structured, periodic monitoring with clear baselines and anomaly indicators. The operator does not watch automation continuously. Instead, they check automation outputs at defined intervals, compare those outputs against expected baselines, and investigate deviations. This approach scales: whether the institution has three automated tasks or thirty, the monitoring procedure is the same.

3.2 What Automation Monitoring Looks Like

In practice, automation monitoring has three components. First, the operator reviews automation output logs to verify that scheduled tasks ran, completed successfully, and produced expected results. Second, the operator reviews any alerts or warnings generated by the automation itself. Third, the operator periodically spot-checks automation results by independently verifying a sample of the automated work. These three components -- log review, alert review, and spot-check -- form the operational rhythm of automation monitoring.

3.3 The Manual Fallback Principle

Per D8-001, every automated task must have a manual fallback. The operator must know how to perform any automated task manually if the automation fails. This requirement has an operational implication: the operator must maintain competency in the manual procedures even when automation is working perfectly. The quarterly automation review (Section 4.4) includes a manual execution exercise for this purpose.

System Model

4.1 The Automation Inventory

The automation inventory is a living document that lists every automated task in the institution. For each task, it records:

  • Task name and ID.
  • Purpose: What the task does and why.
  • Schedule: When it runs (cron expression, event trigger, or continuous).
  • Expected output: What a successful run looks like (log entries, files created, metrics updated).
  • Expected duration: How long a normal run takes.
  • Failure indicators: Specific log entries, exit codes, or conditions that indicate failure.
  • Alert configuration: Whether the task generates alerts on failure, and how those alerts are delivered.
  • Manual fallback: Reference to the manual procedure for this task.
  • Owner: The Domain 8 article or other documentation that governs this task.
  • Last verified: Date of the most recent manual verification that the task is working correctly.

4.2 Daily Automation Monitoring

Integrated into the D10-002 daily checklist:

  1. Review the automation dashboard or summary (a single view showing all automated tasks and their most recent status).
  2. For each task that ran in the past 24 hours: verify it completed with expected exit status. Note any warnings.
  3. For any task that should have run but did not: investigate immediately. A missed scheduled task is an anomaly.
  4. Review any alerts generated by automation in the past 24 hours.
  5. Record findings in the operational log: "Automation: all N tasks ran successfully" or "Automation: [task] reported [issue]."

Estimated time: 2-4 minutes as part of the daily health check.

4.3 Weekly Automation Review

During the weekly duty period:

  • Review the week's automation logs in detail. Look beyond pass/fail: are durations stable? Are output sizes consistent? Are any warning counts increasing?
  • Perform one spot-check: select one automated task, manually verify its most recent output independently. For a backup task, verify one file from the backup. For an integrity check task, independently checksum one file and compare. For a monitoring task, independently verify one measurement.
  • Record the spot-check result in the operational log.

4.4 Quarterly Automation Review

During the quarterly duty period:

  • Review the automation inventory for accuracy. Have any tasks been added, removed, or modified without updating the inventory?
  • Review anomaly thresholds: are the current thresholds still appropriate, or should they be adjusted based on recent experience?
  • Perform one manual execution exercise: select one automated task, disable it, perform the task manually per the documented manual fallback, then re-enable the automation. This verifies both the manual procedure and the operator's competency.
  • Review automation performance over the quarter: failure rate, false alert rate, tasks that have never failed (are they still useful?), tasks that fail frequently (do they need redesign?).

4.5 Intervention Thresholds and Override Procedures

The operator intervenes when:

  • An automated task has failed and has not self-recovered within its defined retry window.
  • An automated task has produced output that falls outside the expected parameters.
  • An automated task is running significantly longer or shorter than expected.
  • An automated task has generated alerts that indicate a condition requiring human judgment.
  • The operator observes a pattern across multiple automated tasks that suggests a systemic issue (e.g., all tasks running slower, multiple tasks failing simultaneously).

When intervening:

  1. Do not immediately modify the automation. Investigate first.
  2. Review the task's logs and the system logs for the relevant time period.
  3. Determine whether the issue is with the automation itself or with the system it operates on.
  4. If the automation needs to be stopped: use the documented stop procedure, not ad hoc commands. Record the stop in the change log.
  5. If manual execution is required while automation is stopped: follow the documented manual fallback.
  6. If the issue qualifies as an incident per D10-003, invoke incident response.
  7. If override is required per D8-009, follow the automation override decision framework.

Rules & Constraints

  • R-D10AU-01: The automation inventory must be current. Every automated task must be documented per Section 4.1. An undocumented automated task is an unmonitored automated task and must be documented or disabled.
  • R-D10AU-02: Daily automation monitoring per Section 4.2 is part of the mandatory daily cycle.
  • R-D10AU-03: Weekly spot-checks per Section 4.3 are mandatory. The spot-check task should rotate across all automated tasks over time.
  • R-D10AU-04: Quarterly manual execution exercises per Section 4.4 are mandatory. They verify both the manual procedure and the operator's competency.
  • R-D10AU-05: Intervention must follow the procedure in Section 4.5. Ad hoc commands to stop or modify automation without documentation are prohibited except under S1 emergency conditions.
  • R-D10AU-06: An automated task that fails more than three times in 30 days without root cause identification must be escalated per D10-005 for structural review.
  • R-D10AU-07: Every automated task must have a documented manual fallback per D8-001. Tasks without manual fallbacks must have one created before the next quarterly review.

Failure Modes

  • Automation complacency. The operator trusts automation so completely that they stop checking it. Failures go undetected. Mitigation: the daily monitoring and weekly spot-check are integrated into the operational cycle, making them part of the routine rather than an optional extra.

  • Alert fatigue. Automation generates too many alerts, most of which are false positives or low-severity. The operator stops reading them. A genuine alert is missed. Mitigation: the quarterly review includes alert rate analysis. False positive rates above 20% trigger threshold adjustment.

  • Manual competency atrophy. The operator has not performed a task manually in years because automation handles it. When automation fails, the operator cannot perform the manual fallback effectively. Mitigation: R-D10AU-04 quarterly manual exercises prevent competency atrophy.

  • Intervention damage. The operator intervenes in a running automation incorrectly, causing more damage than the original anomaly. Mitigation: Section 4.5 requires investigation before intervention, documented stop procedures, and change log entries. Impulsive intervention is the enemy.

  • Inventory staleness. The automation inventory does not reflect reality. Tasks have been added or modified without updating the inventory. Monitoring is based on outdated expectations. Mitigation: R-D10AU-01 and the quarterly inventory review.

Recovery Procedures

  1. If automation monitoring has been neglected: Review all automation logs from the neglect period. Identify any tasks that failed without detection. Assess the impact of undetected failures. Restore monitoring as part of the daily cycle. If the neglect period was extended, conduct a comprehensive automation health assessment.

  2. If an automation failure has caused undetected damage: Invoke D10-003 incident response. The incident is the undetected failure period. Assess the full scope of the damage. Restore from backups if data was affected. The post-incident review must examine why monitoring did not catch the failure.

  3. If manual competency has atrophied: Schedule immediate manual execution exercises for all critical automated tasks. Revise the manual fallback documentation based on the exercise. Commit to the quarterly exercise schedule going forward.

  4. If the automation inventory is stale: Conduct a full inventory reconciliation. Enumerate all running automated tasks on all systems. Compare with the inventory. Update the inventory to match reality. Document any discrepancies found.

Evolution Path

  • Years 0-5: Automation is being introduced incrementally. Each new automated task is added to the inventory and monitoring procedures are established. The operator is still deeply familiar with the manual procedures because automation is new. This is the easiest period for automation monitoring because the operator has not yet developed complacency.

  • Years 5-15: Automation is mature and reliable. This is the period of greatest complacency risk. The quarterly manual exercises become critically important. The operator may not have performed some tasks manually in years. The automation inventory may have grown significantly.

  • Years 15-30: Automation may need to be rebuilt as underlying systems change. The automation inventory and the manual fallback documentation become the bridge between old and new implementations. A successor may need to understand what each automation does before they can maintain or replace it.

  • Years 30-50+: The automation layer will have been rebuilt multiple times. The monitoring principles and inventory practices defined here should persist even as the specific automations change entirely.

Founder Commentary

This section is reserved for dated entries by current and future operators.

2026-02-17 -- Founding Entry: Automation is seductive. Once a task is automated, it feels done. It is off the list. Someone else -- something else -- is handling it. That feeling is a trap. The automated task is not done. It is delegated. And unlike a human delegate who can raise concerns and exercise judgment, an automated delegate will execute its instructions precisely, including when those instructions have become wrong due to changed conditions.

The weekly spot-check is the most important procedure in this article. It takes minutes. It verifies that automation is producing correct results, not just successful exit codes. An automated backup that runs successfully every night but has been silently backing up an empty directory is producing successful exit codes. Only a spot-check -- opening the backup and verifying its contents -- would catch this failure.

The quarterly manual execution exercise is a discipline I expect to resist. I will not want to manually perform a task that automation handles perfectly. But the point is not the task itself. The point is maintaining the competency to perform it. The day I need that competency is the day automation has failed, and I do not want that to be the day I discover I no longer know how to do it.
