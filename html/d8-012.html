D8-012 — Automation & Data Integrity Contracts

Document ID
D8-012
Domain
8 -- Automation & Agents
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
ETH-001, CON-001, SEC-001, OPS-001, D8-001, D8-002, D8-003, D8-007, D8-011, D6-001, D6-002, D6-007, D6-008
Depended Upon By
D8-010, D8-013, D6-007. All automated agents that read or write institutional data. D10 operational procedures involving automated data handling.

Purpose

This article defines the formal interface between automated agents and the institution's data systems. Every automation that reads, writes, modifies, or deletes data is entering into a contract with the data layer: a set of promises about what the automation will do, what it will not do, how it will handle failure, and what guarantees it provides to the data it touches. This article makes those promises explicit, enforceable, and auditable.

The need for data integrity contracts arises from a fundamental asymmetry. A human operator working with data is slow but adaptive. They notice when something looks wrong. They pause when a file seems too large or too small, when a name is unexpected, when a result contradicts their intuition. Automation has no intuition. It operates at speed, without pause, without doubt. When an automation corrupts data, it does so at the speed of the machine, and it does so without the self-awareness that would cause a human to stop and investigate. By the time a human notices the damage, the automation may have propagated the corruption across multiple data stores, backups, and dependent systems.

Data integrity contracts are the institution's defense against this asymmetry. They impose on every automation a set of obligations that a human operator fulfills intuitively: verify before acting, act within boundaries, confirm after acting, and leave a trail that allows others to understand and reverse what was done. These obligations are not suggestions. They are contractual requirements, verifiable through the testing framework (D8-010) and auditable through the observability standards (D8-011).

In a fifty-year institution, the data is irreplaceable. The automation is replaceable. The contracts defined in this article ensure that every automation treats data with the care that its irreplaceability demands.

Scope

This article covers:

  • The definition and structure of a data integrity contract.
  • Pre-conditions: what an automation must verify before touching data.
  • Transaction boundaries: how automated operations are scoped to prevent partial writes and inconsistent states.
  • Post-conditions: what an automation must verify after completing a data operation.
  • Rollback requirements: how an automation must undo its effects if post-conditions are not met.
  • Data tier constraints: how the data tier system (D6-001) restricts what automation may do at each tier.
  • The contract registry: how data integrity contracts are documented, stored, and audited.
  • Contract violation handling: what happens when an automation violates its contract.

This article does not cover:

  • The data tier system itself (see D6-001).
  • The data model or schema definitions (see D6-002).
  • General data integrity principles independent of automation (see D6-007).
  • The observability standards for logging data operations (see D8-011, which defines the logging format; this article defines what must be logged regarding data).
  • The testing framework for verifying contract compliance (see D8-010).
  • Backup and recovery procedures (see D6-006).

Background

3.1 The Speed Problem

A human operator moving files manually might process ten files in a minute. If they make an error on the third file, they notice, stop, and correct it. The damage is three files. An automation performing the same task might process ten thousand files in a minute. If it encounters the same error condition, it applies the error ten thousand times before a human can intervene. The damage is not ten thousand times worse in a linear sense -- it is categorically different. It may be unrecoverable. It may have propagated to backups. It may have triggered other automations that have acted on the corrupted data.

Speed is the reason automation is valuable. Speed is also the reason automation is dangerous to data. Data integrity contracts exist to ensure that speed does not outrun correctness.

3.2 The Interface Between Domains

Domain 6 defines what data is, how it is classified, how it is structured, and what integrity means. Domain 8 defines how automation operates. This article is the bridge between them. It translates Domain 6's integrity requirements into obligations that Domain 8's automations must fulfill. Without this bridge, each automation author must independently interpret D6-001's data tier constraints and D6-007's integrity requirements. Interpretation varies. Inconsistency results. Gaps appear. This article eliminates interpretation by providing a concrete, enforceable contract structure.

3.3 Lessons from Database Systems

The concept of a data integrity contract borrows from the ACID properties of database transactions: Atomicity (an operation either completes fully or has no effect), Consistency (the operation leaves data in a valid state), Isolation (concurrent operations do not interfere), and Durability (completed operations persist). This institution does not operate a traditional database, but the principles are universal. Every data operation, whether it involves a database, a filesystem, or a simple text file, must exhibit these properties if the data it touches is to remain trustworthy over decades.

The institution's adaptation of ACID is pragmatic. Full database-style transactional guarantees are not always achievable on a filesystem. But the intent -- that data operations are complete, correct, non-interfering, and durable -- is achievable through the contract structure defined in this article.

System Model

4.1 The Data Integrity Contract Structure

Every automation that interacts with institutional data must have a documented Data Integrity Contract (DIC). The contract is part of the automation's specification (D8-007) and is stored in the contract registry alongside the automation's lifecycle record (D8-006). A DIC contains the following sections:

Section 1: Data Scope. What data does this automation access? Specify by data tier (per D6-001), by location (filesystem paths, data store identifiers), and by operation type (read, write, modify, delete, move, copy). An automation that reads from Tier 2 data and writes to Tier 3 data has a different risk profile than one that modifies Tier 1 data in place. The data scope defines the boundaries of the automation's data access. Any access outside the declared scope is a contract violation.

Section 2: Pre-conditions. What must be true before the automation begins its data operations? Pre-conditions are verifiable assertions that the automation checks at startup, before touching any data. Examples: the source data exists at the expected path; the source data's checksum matches the expected value; sufficient disk space exists for the output; no other automation is currently writing to the target location; required dependencies are available. If any pre-condition fails, the automation must halt without modifying data and report the failure per D8-011.

Section 3: Operation Specification. What exactly does the automation do to the data? This is a precise description, not a summary. It specifies: which files or records are read, in what order; what transformations or computations are applied; where results are written; what intermediate states exist during processing. The operation specification must be detailed enough that a human reviewer can predict the automation's effects on a given data set without running the automation.

Section 4: Transaction Boundaries. How is the operation scoped to prevent partial completion? A transaction boundary defines the unit of work that either succeeds completely or has no effect. For a simple automation that copies a single file, the transaction boundary is the file copy. For a complex automation that updates multiple records, the transaction boundary may encompass all updates. The contract must specify: what constitutes a complete transaction; what intermediate states are possible if the automation is interrupted; and how partial transactions are detected and handled.

Section 5: Post-conditions. What must be true after the automation completes its data operations? Post-conditions are the mirror of pre-conditions: verifiable assertions that the automation checks after completing its work. Examples: the output file exists and has a valid checksum; the output record count matches the expected count; no source data was unintentionally modified; the data store is in a consistent state. If any post-condition fails, the automation must invoke its rollback procedure.

Section 6: Rollback Procedure. How does the automation undo its effects if post-conditions fail? The rollback procedure must restore data to its pre-operation state. For write operations, this may mean deleting the output. For modify operations, this means restoring from the pre-modification backup (which the automation must create before modifying). For delete operations -- which are restricted per D8-001, R-D8-08 and require human authorization -- rollback means restoring from the pre-deletion copy. If rollback is not possible (because the operation is inherently irreversible), the contract must state this explicitly, and the automation must operate at a higher human-in-the-loop level per D8-003.

4.2 Data Tier Constraints

The data tier system defined in D6-001 imposes specific constraints on automated data operations. These constraints are non-negotiable and override any convenience that wider access might provide.

Tier 1 -- Institutional Memory (Founding documents, constitutional articles, irreplaceable records): Automation may read Tier 1 data. Automation may not write, modify, or delete Tier 1 data without explicit, contemporaneous human authorization per D8-001, R-D8-08. This authorization must be logged with the authorizer's identity, the timestamp, and the specific operation authorized. No blanket authorizations. No standing permissions. Each operation requires its own authorization.

Tier 2 -- Operational Data (Logs, configurations, working data): Automation may read and write Tier 2 data per its DIC. Modification of existing Tier 2 data requires pre-modification backup and post-modification verification. Deletion of Tier 2 data requires documented justification and follows the data retention schedule (D6-011). Automation operating on Tier 2 data must operate at Level 2 (Supervised Autonomous) or higher per D8-001.

Tier 3 -- Transient Data (Temporary files, processing artifacts, cache): Automation may read, write, modify, and delete Tier 3 data per its DIC with minimal constraints. However, the DIC must still specify what Tier 3 data is created or modified, and the automation must still verify post-conditions. The reduced constraints do not eliminate the contract requirement -- they reduce the severity of contract violations.

Cross-tier operations: An automation that reads from one tier and writes to another must comply with the constraints of the more restrictive tier for the entire operation. An automation that reads Tier 2 data and writes the result to Tier 3 operates under Tier 2 constraints, because the integrity of its output depends on the integrity of its input. An automation that reads Tier 3 data and writes to Tier 1 operates under Tier 1 constraints, because it is modifying irreplaceable data.

4.3 The Contract Registry

All Data Integrity Contracts are stored in the institutional contract registry, a structured collection of DIC documents maintained alongside the automation lifecycle records (D8-006). The registry serves three functions:

Discovery: An operator can look up any automation and immediately understand what data it touches, how it touches that data, and what guarantees it provides. This is essential when investigating data anomalies -- the registry answers the question "what automations could have caused this?"

Audit: During the quarterly automation audit (D8-004) and the annual re-justification (D8-001, R-D8-07), the registry provides the authoritative record of what each automation is permitted to do with data. Actual behavior (as recorded in logs per D8-011) is compared against the contract. Discrepancies are violations.

Succession: A future maintainer encountering an unfamiliar automation can read its DIC to understand its relationship with institutional data without reading the automation's source code. The contract is a human-readable summary of the automation's data obligations.

The registry is organized by data tier, then by automation ID. Each entry links to the full automation specification (D8-007) and the lifecycle record (D8-006). The registry is itself classified as Tier 2 data and is subject to the same backup and retention requirements as other operational data.

4.4 Write-Ahead Principle

Before any automation modifies or deletes existing data, it must first write a record of what it intends to do and create a recoverable copy of the data it will change. This is the write-ahead principle, and it has three components:

Intent log: Before executing a data modification, the automation writes a log entry describing: what data will be changed, how it will be changed, and why. This entry is written before the modification begins, so that if the automation crashes mid-operation, the intent log reveals what was in progress.

Pre-modification snapshot: Before modifying a file or record, the automation creates a copy of the original. The copy is stored in a designated rollback area with a clear naming convention linking it to the automation, the execution ID, and the timestamp. The copy persists until the post-conditions are verified, at which point it may be removed per the automation's DIC.

Completion confirmation: After the modification succeeds and post-conditions pass, the automation writes a completion entry to the intent log, confirming that the operation finished correctly. An intent log entry without a corresponding completion entry indicates an interrupted operation that requires investigation.

Rules & Constraints

  • R-D8-12-01: Every automation that reads, writes, modifies, moves, copies, or deletes institutional data must have a documented Data Integrity Contract (DIC) conforming to the structure defined in Section 4.1. Automation without a DIC must not be deployed.
  • R-D8-12-02: All automation must verify pre-conditions before executing data operations. If any pre-condition fails, the automation must halt without modifying data and log the failure per D8-011.
  • R-D8-12-03: All automation must verify post-conditions after completing data operations. If any post-condition fails, the automation must execute its rollback procedure and log the failure.
  • R-D8-12-04: All automation that modifies or deletes existing data must implement the write-ahead principle (Section 4.4): intent log, pre-modification snapshot, and completion confirmation.
  • R-D8-12-05: Automation data access must not exceed the scope declared in its DIC. Any access outside the declared data scope is a contract violation, must be logged as an ERROR per D8-011, and must trigger an investigation per D8-004.
  • R-D8-12-06: The data tier constraints in Section 4.2 are absolute. No automation may circumvent tier-based restrictions regardless of operational convenience. Tier 1 write/modify/delete operations require contemporaneous human authorization with no exceptions.
  • R-D8-12-07: All DICs must be registered in the contract registry (Section 4.3). The registry must be updated whenever a DIC is created, modified, or retired.
  • R-D8-12-08: Pre-modification snapshots must be retained until post-conditions are verified and for a minimum of 72 hours after successful completion. For operations on Tier 1 data, snapshots must be retained for a minimum of 30 days.
  • R-D8-12-09: During the quarterly automation audit, each automation's actual data operations (as recorded in logs) must be compared against its DIC. Discrepancies must be resolved within 14 days: either the automation is corrected to match the contract, or the contract is amended through the standard change process (D8-010, GOV-001).
  • R-D8-12-10: Automation must not hold exclusive locks on data resources for longer than the duration of a single transaction boundary as defined in the DIC. Long-running operations must be decomposed into smaller transactions to prevent resource starvation.

Failure Modes

  • Silent data corruption. An automation modifies data incorrectly but reports success because its post-conditions are insufficient. The corruption is not detected until a human or another system reads the data much later. Mitigation: post-conditions must be substantive, not perfunctory. Checking that a file exists is necessary but insufficient; checking its checksum, size, and a sample of its content is better. R-D8-12-03 requires post-condition verification. The quarterly audit verifies that post-conditions are meaningful.

  • Rollback failure. The automation's rollback procedure does not work because the pre-modification snapshot was corrupted, not created, or already cleaned up. The data is left in an inconsistent state with no path to recovery. Mitigation: R-D8-12-04 mandates the write-ahead principle. R-D8-12-08 mandates snapshot retention. The testing framework (D8-010) must include rollback testing as part of the Automation Acceptance Test Plan.

  • Contract drift. Over time, the automation evolves but its DIC is not updated. The automation's actual behavior diverges from its documented contract. The registry becomes unreliable. Mitigation: R-D8-12-09 mandates quarterly comparison of actual behavior to contract. Any modification to an automation's data operations triggers a DIC review per the lifecycle change process (D8-006).

  • Cross-automation interference. Two automations operate on the same data concurrently. One overwrites the other's output, or both read stale data because neither is aware of the other's writes. Mitigation: the DIC must declare all data locations. The contract registry enables discovery of overlapping data scopes. R-D8-12-10 constrains lock duration. D8-002 requires minimal inter-automation dependencies.

  • Tier boundary violation. An automation accesses data at a tier above its authorization, either through a bug or through an undeclared dependency. Mitigation: R-D8-12-06 makes tier constraints absolute. File system permissions should enforce tier boundaries where possible. The observability logs (D8-011) record all data access, enabling detection of violations.

  • Snapshot storage exhaustion. Pre-modification snapshots accumulate because cleanup is deferred or the automation runs more frequently than anticipated. Snapshot storage fills, causing the automation to fail when it cannot create the next snapshot. Mitigation: R-D8-12-08 defines retention periods. The DIC must specify snapshot cleanup procedures. The monitoring system (D8-008) tracks snapshot storage consumption.

  • Cascade through data. Automation A corrupts data that Automation B reads as input. Automation B produces incorrect output, which Automation C reads as input. The corruption propagates through the data pipeline faster than human review. Mitigation: each automation's pre-conditions should verify input integrity (checksums, expected ranges, format validation). The write-ahead principle and post-condition verification create checkpoints that limit propagation.

Recovery Procedures

  1. If silent data corruption is discovered: Determine the scope of corruption using the intent logs and execution logs. Identify the automation responsible. Check whether pre-modification snapshots are still available. If yes, restore from snapshots. If no, restore from the most recent verified backup per D6-006. Trace forward from the corruption point to identify all downstream data that may have been affected. Verify each downstream artifact. Update the automation's post-conditions to detect the corruption pattern. Document the incident per SEC-001.

  2. If rollback fails: Do not attempt to fix the data manually under time pressure. Halt the automation. Assess the inconsistency. If the pre-modification snapshot exists but the automated rollback failed, perform manual rollback using the snapshot. If the snapshot is unavailable, restore from backup per D6-006. Investigate why the rollback failed and fix the rollback procedure before re-enabling the automation.

  3. If contract drift is discovered during audit: Immediately compare the automation's actual data operations (from the last quarter's logs) against the DIC. If the automation is doing less than the contract permits, note but do not treat as urgent. If the automation is doing more than the contract permits -- accessing data outside its declared scope, or performing operations not in its specification -- halt the automation. Investigate whether the undeclared operations caused harm. Update the DIC or the automation to restore alignment. Re-test per D8-010.

  4. If cross-automation interference is detected: Halt both automations. Determine which automation's output is authoritative. Restore the non-authoritative data from the appropriate source. Modify the automations to avoid concurrent access: either serialize their execution (D8-009) or partition their data scopes so they do not overlap. Update both DICs.

  5. If a tier boundary violation occurs: Halt the automation immediately. Assess whether the violation resulted in unauthorized data modification. If Tier 1 data was modified without authorization, treat as a security incident per SEC-001. Restore from backup if needed. Fix the automation's access controls. Review the contract registry for similar vulnerabilities in other automations.

Evolution Path

  • Years 0-5: The DIC framework is new. Early automations will have simple contracts -- a backup script that reads files and writes copies, a log rotator that moves files and deletes old ones. These simple cases are the foundation. Write their contracts rigorously, even though the contracts seem obvious for such simple operations. The discipline of writing contracts for simple cases builds the skill to write contracts for complex cases later. The contract registry will be small and easy to maintain.

  • Years 5-15: As automation matures, data pipelines may emerge: sequences of automations where one's output is another's input. The DIC framework must accommodate pipeline contracts -- composite contracts that describe the data flow through multiple automations. The contract registry becomes a map of the institution's automated data flows. This is when the registry's discovery function becomes most valuable: understanding which automations touch which data.

  • Years 15-30: Hardware and platform changes may require rewriting automations. The DICs survive the rewrite because they describe obligations, not implementations. A new automation replacing an old one can inherit its predecessor's DIC (with updates for any changed behavior). The contract provides continuity across implementation changes. This is the period when the write-ahead principle proves its worth: intent logs from years past enable reconstruction of data operations history.

  • Years 30-50+: A successor inheriting the institution's automation finds in the contract registry a complete map of how automation interacts with data. Even if they cannot immediately understand every automation's source code, they can understand every automation's data obligations. The DIC framework gives the successor a way into the system that does not require reading code -- it requires reading contracts, which are written in structured natural language. The registry is the successor's first stop when investigating any data anomaly.

  • Signpost for revision: If the DIC framework feels burdensome for trivially simple automations, consider a lightweight contract template for Tier 3 operations that preserves the essential elements (data scope, pre-conditions, post-conditions) without requiring the full six-section format. But resist the temptation to exempt any automation from the contract requirement entirely. The automation that seems too simple to need a contract is the one that will cause trouble when it changes.

Commentary Section

This section is reserved for dated entries by current and future operators.

2026-02-17 -- Founding Entry: The single most important idea in this article is that the contract is separate from the code. The DIC describes what the automation promises to do with data. The code implements that promise. When these are the same document -- when the only description of the automation's data behavior is the code itself -- the institution has a problem, because reading code requires a programmer, but reading a contract requires only literacy and attention.

I expect the write-ahead principle (Section 4.4) to generate the most resistance. Creating a snapshot before every modification is slow. It consumes storage. It feels redundant when the automation "always works." But the snapshot is not for when the automation works. It is for when the automation fails, or when a future maintainer discovers that the automation was wrong in ways that were not immediately visible. The snapshot is an insurance policy, and like all insurance, its value is invisible until the day you need it.

A practical note on pre-modification snapshots: they need not be full copies in every case. For small files, a full copy is trivial. For large data sets, a checksum plus a diff or a pointer to the last known-good backup may be sufficient, provided the backup is verified per D6-006. The DIC should specify the snapshot strategy appropriate to the data size and the risk level of the operation. What is not negotiable is that some form of snapshot exists. An automation that modifies data without any record of the prior state is an automation that has made its errors permanent.

Future maintainers: if you find the contract registry outdated, do not discard it. Update it. The registry is only as good as its last maintenance, but an outdated registry is still better than no registry. It tells you what someone once believed each automation was doing with data. Start from there.
