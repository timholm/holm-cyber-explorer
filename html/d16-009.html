D16-009 — Error States, Feedback & System Communication

Document ID
D16-009
Domain
16 -- Interface & Navigation
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
D16-001 (Interface Philosophy)
Depended Upon By
D16-011 (Offline & Degraded-Mode Behavior)

Purpose

This article defines how the institution's interface communicates with users about the system's state. It covers three categories of communication: error states (something has gone wrong), operational feedback (the system is confirming or reporting on user actions), and system status messages (the system is reporting on its own condition). Together, these constitute the interface's voice -- the channel through which the institution speaks to the people it serves.

D16-001 establishes that the interface must communicate honestly and that it must never misrepresent system state. This article translates that principle into specific, enforceable rules for every category of system-to-user communication. An error message that is vague, misleading, or absent undermines the user's sovereignty as surely as a locked door. A progress indicator that lies about progress teaches the user to distrust the interface. A confirmation message that arrives too late or not at all leaves the user uncertain whether their action had any effect.

This article is addressed to every person who writes an error message, designs a status indicator, implements a loading state, or decides what the interface should say when something goes wrong -- or right. It is also addressed to the person who, decades from now, must interpret error messages written by people who are no longer available to explain them.

Scope

This article covers:

  • Classification of system-to-user communications (errors, confirmations, warnings, status, progress).
  • Required structure and content for error messages.
  • Feedback timing requirements for user-initiated actions.
  • Progress indication standards for long-running operations.
  • Confirmation and acknowledgement patterns for successful actions.
  • Warning patterns for potentially destructive or irreversible actions.
  • System health status communication.
  • The relationship between user-facing messages and technical logs.

This article does not cover:

  • The technical infrastructure that detects errors (covered by Infrastructure domains).
  • How the interface visually renders these messages (covered by D16-007 and D16-014).
  • Input method-specific feedback (covered by D16-008).
  • Degraded-mode interface behaviour when entire subsystems fail (covered by D16-011).
  • Quality assurance procedures for testing error handling (covered by Domain 19).

Background

The Cost of Bad Error Communication

Most software communicates errors badly. Error messages are written by developers for developers. They contain technical jargon, internal identifiers, and instructions that assume knowledge the user does not have. "Error 0x8007045D: The request could not be performed because of an I/O device error" tells the developer exactly what happened; it tells the user nothing actionable. The user's response to such a message is helplessness, followed by distrust, followed by avoidance of the system.

In a self-sovereign, air-gapped institution, the consequences of bad error communication are more severe than in a connected environment. There is no help desk to call. There is no online forum to search. The operator must resolve the problem using only the information the system provides and the documentation stored locally. If the error message does not help the operator understand and resolve the problem, the operator is stuck.

Honesty as Foundation

D16-001 Principle R-D16-05 (Honest Communication) applies with particular force to error states. The temptation in error handling is to soften bad news -- to say "Something went wrong" instead of "The storage device is full and no new data can be written until space is freed." The softened message is politer. It is also useless. An institution that respects its operators tells them the truth, clearly and completely, even when the truth is alarming. Alarming truth can be acted upon. Soothing vagueness cannot.

The Spectrum of System Communication

System-to-user communication exists on a spectrum from routine to critical. At one end, a brief visual flicker confirms that a keystroke was received. At the other end, a full-screen message warns that the system is about to lose data irreversibly. The communication system must handle this entire spectrum with consistent principles but appropriate urgency.

System Model

All system-to-user communications are classified into five categories, each with distinct requirements:

COMMUNICATION CLASSIFICATION
==============================

Category        Urgency     Persistence    User Action Required    Example
--------        -------     -----------    --------------------    -------
CONFIRMATION    Low         Transient      None                    "Document saved."
                            (3-5 seconds)

STATUS          Low         Persistent     None (informational)    "Search index: 98% built"
                            until resolved

PROGRESS        Medium      Persistent     Wait or cancel          "Importing: 47 of 230 items"
                            until complete

WARNING         High        Persistent     Decision required       "This will delete 12 items.
                            until dismissed                         This cannot be undone."

ERROR           Critical    Persistent     Action required         "Storage full. Free 500MB
                            until resolved                          to continue. See D4-003."


REQUIRED MESSAGE STRUCTURE (for Warnings and Errors):
=====================================================
1. WHAT happened     -- Plain language description of the event
2. WHY it matters    -- Impact on the user's current task or data
3. WHAT to do        -- Specific, actionable steps to resolve or proceed
4. WHERE to learn    -- Reference to documentation (article ID) for deeper context

Example:
  WHAT:  "The search index is corrupted and cannot return results."
  WHY:   "You will not be able to search for content until the index is rebuilt."
  WHAT:  "To rebuild the index, navigate to System > Maintenance > Rebuild Index."
  WHERE: "For details on index maintenance, see D16-004 Section 8."

Communication Channels

Messages are delivered through one of three channels based on urgency and context:

  • Inline: Messages that appear within the content area, near the element that triggered them. Used for confirmations and field-level validations. These do not interrupt the user's workflow.
  • Banner: Messages that appear at the top or bottom of the viewport, overlaying but not replacing the current content. Used for warnings, system status, and non-critical errors. These get the user's attention without blocking their work.
  • Modal: Messages that require acknowledgement before the user can proceed. Used only for critical errors and for confirmations of destructive or irreversible actions. Modals interrupt workflow and MUST be used sparingly.

Rules & Constraints

  • R-D16-09-01: No Silent Failures. Every error detected by the system MUST be communicated to the user through the interface. No error may be silently swallowed, logged only to a technical log, or hidden behind a generic "something went wrong" message. The user is the operator. The operator must know when operations fail.

  • R-D16-09-02: Plain Language. All user-facing messages MUST be written in plain language. No technical jargon, error codes, stack traces, or internal identifiers may appear in user-facing messages. Technical details MUST be available in a separate technical log, accessible from the error message via a "show details" mechanism, but they MUST NOT be the primary communication.

  • R-D16-09-03: Actionable Errors. Every error message that requires user action MUST include specific instructions for resolving the error. "An error occurred" is never acceptable. "The file could not be saved because the disk is full. Free at least 50MB of space and try again." is the minimum standard.

  • R-D16-09-04: Honest Progress. Progress indicators MUST reflect actual progress. If the system cannot measure actual progress, it MUST display an indeterminate progress indicator (e.g., a spinner) rather than a fake progress bar that moves at an arbitrary rate. A progress bar that reaches 99% and stalls teaches the user that progress bars are lies. Once that lesson is learned, it cannot be unlearned.

  • R-D16-09-05: Confirmation of Destructive Actions. Any action that deletes data, modifies data irreversibly, or changes system configuration MUST require explicit confirmation before execution. The confirmation message MUST describe specifically what will happen ("This will permanently delete 12 articles and all their revision history") rather than generically ("Are you sure?"). The destructive option MUST NOT be the default-selected option.

  • R-D16-09-06: Feedback Timing. Immediate feedback (visual acknowledgement of input) MUST occur within 100 milliseconds. Operation acknowledgement (confirmation that a user-initiated action has been received and processing has begun) MUST occur within 1 second. If an operation will take longer than 2 seconds, a progress indicator MUST appear. If an operation will take longer than 10 seconds, the progress indicator MUST include an estimated time remaining or a cancel option.

  • R-D16-09-07: Accessible Communication. All system messages MUST be announced to screen readers via appropriate ARIA live regions or equivalent mechanisms. Visual-only indicators (colour changes, icon swaps) MUST be accompanied by text that conveys the same information. Auditory feedback MUST NOT be the sole communication channel for any message.

  • R-D16-09-08: Message Persistence. Confirmation messages SHOULD auto-dismiss after 3-5 seconds. Warning and error messages MUST persist until the user explicitly dismisses them or until the condition is resolved. No critical message may disappear before the user has had the opportunity to read it.

  • R-D16-09-09: No Blame. Error messages MUST NOT blame the user. "Invalid input" suggests user fault. "The value entered is not in the expected format. Expected: YYYY-MM-DD" describes the problem without attribution. The interface serves the user; it does not judge them.

Failure Modes

  • Error message staleness. The system changes but error messages are not updated. An error message references a menu item that was renamed or a procedure that was reorganised. The user follows obsolete instructions and fails to resolve the error. Likelihood: high over time. Severity: high (undermines trust in all error messages). Detection: automated cross-referencing of error message content against current interface labels and documentation references during each release cycle.

  • Alert fatigue. Too many non-critical messages train the user to dismiss all messages without reading them. When a critical message appears, it is dismissed reflexively. Likelihood: very high if warnings are overused. Severity: critical (a dismissed critical warning can lead to data loss). Detection: logging of message dismiss-times; if messages are being dismissed faster than they can be read, alert fatigue is occurring.

  • Cascade communication failure. A single root error triggers multiple downstream errors, each generating its own message. The user is overwhelmed by a flood of messages and cannot identify the root cause. Likelihood: moderate. Severity: high (obscures the actionable information). Detection: message-rate limiting; if more than three messages appear within two seconds, aggregate them into a single summary with expandable detail.

  • False confidence from fake progress. A progress bar that does not track real progress reaches 90% and stalls. The user waits, believing the operation is nearly complete, when in fact it has encountered an error that has not been reported. Likelihood: moderate. Severity: high (wastes user time and erodes trust). Detection: R-D16-09-04 prohibits fake progress bars; enforcement through code review and automated detection of progress indicators not connected to real metrics.

  • Inaccessible error communication. An error is communicated only through a colour change (e.g., a field border turns red) with no text explanation and no screen reader announcement. Sighted users with colour vision deficiency and screen reader users both miss the error. Likelihood: moderate. Severity: critical for affected users. Detection: automated accessibility testing that verifies all error states have text content and ARIA announcements.

Recovery Procedures

  1. If error messages reference stale information: Conduct an audit of all error messages in the system. Cross-reference every instruction, label reference, and documentation pointer against the current interface and documentation. Update all stale references. Establish a process for flagging error messages whenever the elements they reference are modified.

  2. If alert fatigue is detected: Review the frequency and severity classification of all messages. Reclassify any message that is not truly a warning to a lower severity (status or confirmation). Reduce the visibility of low-severity messages. Ensure that critical messages use a visually distinct channel (modal or high-contrast banner) that the user has not been trained to dismiss by overexposure to non-critical alerts.

  3. If error cascades are overwhelming users: Implement cascade detection in the message system. When multiple errors occur within a short window, group them under a single root-cause message with expandable details. Ensure the root cause is identified and presented first. Downstream errors should be listed as consequences, not as independent events.

  4. If progress indicators are found to be dishonest: Audit all progress indicators in the system. For each one, verify that it is connected to a real metric (items processed, bytes written, steps completed). Replace any indicator that is not connected to a real metric with an indeterminate indicator. Update this article's Commentary Section with the specific instances found and the corrections made.

  5. If critical messages are being dismissed unread: Add a mandatory acknowledgement delay for critical messages: the dismiss button is inactive for 2-3 seconds, ensuring the user has time to read the message. This is a last resort and should not be needed if alert fatigue is properly managed. Document the use of this pattern and monitor whether it improves or annoys.

Evolution Path

  • Years 0-5: Establish the message classification system and the four-part error message structure. Write all initial error messages to the standard. Build the infrastructure for cascading detection and message-rate management. Expect significant revision as real users encounter real errors and report which messages were helpful and which were not.

  • Years 5-15: The message library should be stable and well-tested. Focus shifts to maintaining message accuracy as the system evolves. Establish the cross-referencing process that catches stale references. Begin tracking message effectiveness: do users who encounter error messages successfully resolve their problems?

  • Years 15-30: Language may drift. Terms that were clear in 2026 may become ambiguous or archaic by 2046. Review all user-facing messages for linguistic clarity in the context of the current user population. This is not a technical review; it is a communication review, and it should involve actual users, not just developers.

  • Years 30-50: The error communication system should be one of the most stable parts of the interface, because its principles are rooted in human cognition rather than in technology. The specific messages will have been rewritten multiple times. The principles -- honesty, actionability, clarity, accessibility -- should be unchanged.

Founder Commentary

This section is reserved for dated entries by current and future operators.

2026-02-17 -- Founding Entry: I have spent more time on the "no silent failures" rule (R-D16-09-01) than on any other rule in this article. The temptation to swallow errors is enormous. Every developer has written a catch block that logs the error and moves on, because surfacing the error to the user "would be confusing." This is a failure of imagination, not a failure of the user. If the error is confusing when reported, write a better report. Do not hide the error.

The "no blame" rule (R-D16-09-09) is one I feel strongly about. A system that says "Invalid input" is a system that has decided the user is wrong rather than examining whether its own expectations are unreasonable. In most cases, "invalid input" means "I expected a specific format and did not tell you what it was." The system failed first. The error message should reflect that.

I anticipate that the four-part error message structure (what happened, why it matters, what to do, where to learn more) will feel verbose for simple errors. That is acceptable. Verbosity in error messages is a feature, not a bug. An error message that is too detailed wastes a few seconds of reading time. An error message that is too terse wastes hours of troubleshooting time. The asymmetry is clear.
