D16-014 — Interface Component Catalog & Pattern Language

Document ID
D16-014
Domain
16 -- Interface & Navigation
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
D16-001 (Interface Philosophy), D16-006 (Accessibility Standards), D16-007 (Typography & Layout)
Depended Upon By
D16-010 (Multi-Generational Continuity), D16-013 (User Onboarding)

Purpose

This article defines the vocabulary of interface elements that the institution uses to communicate with its users. It establishes a pattern language -- a named, documented set of reusable interaction patterns -- that ensures consistency across the interface and provides a shared vocabulary for everyone who designs, builds, evaluates, or discusses the interface.

A pattern language is not a style guide, though it serves some of the same purposes. A style guide says "buttons look like this." A pattern language says "when the user needs to make a choice between two mutually exclusive options, use the Binary Choice pattern, which presents both options with equal visual weight and requires explicit selection." The pattern language describes the problem, the solution, the context in which the solution applies, and the relationships between patterns. It is a design grammar, not a design dictionary.

This article is addressed to every person who builds, modifies, or evaluates interface components. It is the reference document that ensures a navigation menu built by one person looks and behaves the same as a navigation menu built by another person, across decades of development. It is also addressed to the person who must add a new pattern to the catalog: it defines the process by which the vocabulary grows.

Scope

This article covers:

  • The structure and format of pattern definitions.
  • The foundational pattern catalog: the initial set of patterns that cover the institution's core interaction needs.
  • Pattern relationships: how patterns compose, substitute, and conflict.
  • The process for proposing, reviewing, and adding new patterns.
  • The process for deprecating and replacing existing patterns.
  • Pattern compliance requirements: when and how existing interface elements must conform to cataloged patterns.
  • Degraded-mode variants: how patterns adapt when resources are constrained (per D16-011).

This article does not cover:

  • The visual styling of patterns (covered by D16-007; this article defines structure and behaviour, not appearance).
  • The implementation technology used to build patterns (the catalog is technology-agnostic).
  • Content strategy or editorial patterns (covered by content domains).
  • Quality assurance procedures for testing pattern compliance (covered by Domain 19).

Background

Why a Pattern Language, Not Just Components

Component libraries are common in software development. They provide reusable visual and interactive building blocks: buttons, forms, menus, dialogs. But component libraries are implementation artefacts. They are tied to specific technologies, specific rendering platforms, and specific code bases. When the technology changes -- and it will change multiple times in fifty years -- the component library must be rewritten.

A pattern language operates at a higher level. It describes abstract interaction patterns that can be implemented in any technology. The "Confirmation Dialog" pattern describes a problem (the user is about to perform a consequential action), a solution (present the consequences and require explicit confirmation), and constraints (the destructive option must not be the default). This description is valid whether the pattern is implemented in HTML, in a native application framework, or in a technology that does not exist yet. The pattern survives technology transitions; only its implementation changes.

The Value of a Shared Vocabulary

When an institution has a named set of patterns, discussions about the interface become precise. Instead of "let's add a way for users to pick from a list," the discussion becomes "this is a Single-Select List pattern; it requires keyboard navigation, a visible selection indicator, and a clear empty state." The name carries the requirements. The name carries the accessibility constraints. The name carries the degraded-mode variant. Every participant in the discussion knows exactly what is being proposed because they share the vocabulary.

Over fifty years, the people designing the interface will change many times. The pattern language provides continuity across personnel transitions. A new designer can read the pattern catalog and understand not just what the interface looks like today, but what principles and patterns underlie it, why specific patterns were chosen, and how to extend the system consistently.

Relationship to Christopher Alexander

The concept of a pattern language in software design originates with architect Christopher Alexander, whose "A Pattern Language" (1977) described recurring solutions to recurring problems in building and urban design. Alexander's insight was that good design is not the product of individual genius but of the disciplined application of solutions that have proven their value across many contexts. This article applies the same insight to interface design: the patterns documented here are not inventions but recognitions -- named solutions to problems that every interface encounters.

System Model

PATTERN DEFINITION STRUCTURE
===============================
Every pattern in the catalog follows this structure:

PATTERN NAME: [Unique, descriptive name]
PATTERN ID:   [P-D16-NNN]
STATUS:       [Active / Deprecated / Proposed]
PROBLEM:      What recurring interaction problem does this pattern solve?
CONTEXT:      When should this pattern be used? When should it NOT be used?
SOLUTION:     How does the pattern solve the problem? Structure and behavior.
CONSTRAINTS:  What rules from D16-001, D16-006, D16-007, D16-008 apply?
ACCESSIBILITY:What specific accessibility requirements does this pattern carry?
DEGRADED:     How does this pattern behave under D16-011 degradation classes?
COMPOSED OF:  What sub-patterns does this pattern include?
RELATED:      What other patterns solve similar problems?
EXAMPLES:     Where is this pattern used in the current interface?


FOUNDATIONAL PATTERN CATALOG
===============================

Navigation Patterns:
  P-D16-001  Primary Navigation       -- Main menu, always visible
  P-D16-002  Breadcrumb Trail         -- Location indicator, hierarchical
  P-D16-003  Table of Contents        -- Document-level section navigation
  P-D16-004  Pagination               -- Sequential content traversal
  P-D16-005  Cross-Reference Link     -- Inline link to related content

Content Display Patterns:
  P-D16-010  Article View             -- Full document display with metadata
  P-D16-011  List View                -- Multiple items in scrollable list
  P-D16-012  Detail-on-Demand         -- Summary with expandable detail
  P-D16-013  Data Table               -- Structured tabular data display
  P-D16-014  Metadata Display         -- Document properties sidebar

Input Patterns:
  P-D16-020  Text Field               -- Single-line text input
  P-D16-021  Text Area                -- Multi-line text input
  P-D16-022  Single Select            -- Choose one from list
  P-D16-023  Multi Select             -- Choose many from list
  P-D16-024  Binary Choice            -- Yes/No, On/Off, True/False
  P-D16-025  Date Input               -- Locale-aware date entry

Feedback Patterns:
  P-D16-030  Confirmation Message     -- Transient success acknowledgement
  P-D16-031  Warning Banner           -- Persistent caution notice
  P-D16-032  Error Message            -- Persistent error with resolution steps
  P-D16-033  Progress Indicator       -- Operation progress (determinate)
  P-D16-034  Activity Indicator       -- Operation in progress (indeterminate)
  P-D16-035  Confirmation Dialog      -- Pre-action destructive confirmation

System Patterns:
  P-D16-040  Degradation Indicator    -- Current system health status
  P-D16-041  Search Interface         -- Query input, results display, filters
  P-D16-042  Help Tooltip             -- Brief contextual explanation
  P-D16-043  Help Expansion           -- Detailed contextual explanation
  P-D16-044  Empty State              -- What to show when there is no content

Pattern Composition

Patterns compose to form more complex interface regions. The Article View pattern (P-D16-010) composes the Breadcrumb Trail (P-D16-002), the Table of Contents (P-D16-003), the Metadata Display (P-D16-014), and potentially the Cross-Reference Link (P-D16-005). The composition is documented in the pattern definition, so a designer implementing the Article View can trace its requirements down to each constituent pattern.

Pattern Registry

The pattern catalog is maintained as a versioned registry. Each pattern has a unique ID that is never reused. Deprecated patterns remain in the registry with their deprecated status, a deprecation date, and a pointer to the replacement pattern. The registry is itself an article in the institution's content system and is navigable through the standard interface.

Rules & Constraints

  • R-D16-14-01: Pattern Compliance. Every interactive element in the institution's interface MUST correspond to a pattern in the catalog. If an element does not match any existing pattern, either it must be adapted to fit an existing pattern or a new pattern must be proposed and approved before the element is deployed. No ad hoc interaction elements are permitted.

  • R-D16-14-02: Pattern Completeness. Every pattern definition MUST include all fields specified in the Pattern Definition Structure. No field may be omitted. The Accessibility and Degraded fields are particularly important and MUST NOT contain placeholders or "to be determined" entries. A pattern without defined accessibility behaviour is not ready for use.

  • R-D16-14-03: New Pattern Proposal Process. A new pattern may be proposed by anyone involved in interface design. The proposal MUST include a complete pattern definition plus a justification explaining why no existing pattern adequately addresses the problem. The proposal is reviewed by the Domain 16 lead and at least one other reviewer (preferably from Domain 19 for quality assurance perspective). If approved, the pattern is added to the catalog with Active status.

  • R-D16-14-04: Deprecation Process. A pattern may be deprecated when a superior alternative is available or when the problem the pattern solved no longer exists. Deprecation MUST include: a deprecation notice with the date, the reason, and a reference to the replacement pattern (if any). Deprecated patterns MUST remain in the catalog indefinitely. Interface elements using a deprecated pattern MUST be migrated to the replacement within 12 months of the deprecation date.

  • R-D16-14-05: Technology Independence. Pattern definitions MUST NOT reference specific implementation technologies, programming languages, frameworks, or rendering engines. A pattern must be implementable in any technology that supports the basic capabilities described in the pattern (text display, user input, visual hierarchy). Technology-specific implementation notes may be maintained separately but are not part of the pattern definition.

  • R-D16-14-06: Accessibility as Pattern Property. Every pattern MUST define its accessibility requirements as an inherent part of the pattern, not as an optional enhancement. The Confirmation Dialog pattern, for example, does not have "optional" screen reader support; it has "required screen reader announcement of the dialog content and required focus trapping within the dialog." Accessibility is structural, not cosmetic.

  • R-D16-14-07: Degraded-Mode Variant Required. Every pattern MUST define how it behaves under each degradation class from D16-011 (D1 through D4). If a pattern cannot function at a given degradation class, it must specify this explicitly and identify what simpler pattern (if any) replaces it. The Search Interface pattern (P-D16-041) may not function at D4 (minimal system); the pattern definition must state this and point to the Table of Contents pattern (P-D16-003) as the D4 alternative for content discovery.

  • R-D16-14-08: Catalog Versioning. The pattern catalog is versioned. Every addition, modification, or deprecation increments the catalog version. The complete catalog at every version is preserved in the institutional archive per D20 standards. This allows future maintainers to determine which patterns were available at any point in the institution's history.

Failure Modes

  • Pattern proliferation. Too many patterns are added without sufficient justification, many of them slight variations on existing patterns. The catalog becomes a sprawling, confusing mess that no one reads. Designers ignore it and build ad hoc elements. Likelihood: moderate over time. Severity: high (defeats the purpose of having a catalog). Detection: periodic review of catalog size versus actual pattern usage; any pattern not used in the interface should be evaluated for removal or deprecation.

  • Pattern rigidity. The catalog becomes so authoritative that designers are afraid to propose new patterns or to argue that existing patterns do not serve a new use case. The catalog, intended to enable consistent design, becomes a constraint that prevents effective design. Likelihood: moderate. Severity: medium (suboptimal design choices are made to fit existing patterns rather than proposing better ones). Detection: tracking of new pattern proposals; if proposals drop to zero for more than 12 months, the catalog may be intimidating rather than enabling.

  • Orphaned implementations. The interface contains elements that were built to match patterns that were later deprecated, but the elements were never migrated to the replacement pattern. Over time, the interface becomes a museum of deprecated interaction styles. Likelihood: high without enforcement of the 12-month migration window. Severity: medium (creates inconsistency and confusion). Detection: automated inventory of interface elements mapped to their pattern IDs; flag any elements using deprecated patterns.

  • Accessibility regression in pattern evolution. A pattern is updated to add new capabilities, and the update inadvertently breaks an accessibility requirement that was met by the previous version. Likelihood: moderate. Severity: critical (accessibility regressions are always critical per D16-006). Detection: accessibility testing of every pattern update, with comparison against the previous version's accessibility profile.

  • Technology-implementation leakage. Pattern definitions gradually accumulate technology-specific language and assumptions, making them unintelligible after a technology transition. Likelihood: moderate (especially if the same technology is used for many years, making tech-specific language feel "normal"). Severity: high during a technology transition (the patterns that should guide the reimplementation are instead tied to the old implementation). Detection: periodic review of pattern definitions for technology-specific language; any reference to a specific framework, library, or runtime should be flagged and removed.

Recovery Procedures

  1. If the pattern catalog has become bloated: Conduct a comprehensive usage audit. Map every interface element to its corresponding pattern. Identify patterns that are defined in the catalog but not used anywhere in the interface. For each unused pattern, determine whether it serves a future need (keep it) or was an overspecification (deprecate it). Merge near-duplicate patterns into single patterns with configurable variants. Aim for the smallest catalog that covers all actual interface needs.

  2. If designers are bypassing the catalog: Investigate why. Common reasons: the catalog is hard to search, existing patterns do not fit real needs, the proposal process is too burdensome, or the catalog is seen as a bureaucratic obstacle rather than a design tool. Address the root cause. Simplify the proposal process if it is too heavy. Add missing patterns if the catalog has gaps. Improve catalog navigation if it is hard to search. The catalog serves the designers; if designers are not using it, the catalog needs to change, not the designers.

  3. If orphaned implementations are found: Create a migration plan that prioritises the most visible and most used orphaned elements. Assign migration tasks with deadlines. Update the interface and the pattern registry as each migration is completed. If the 12-month migration window from R-D16-14-04 has been exceeded, treat this as a quality defect per Domain 19 and escalate accordingly.

  4. If a technology transition reveals technology-dependent patterns: Freeze the current pattern catalog. Review every pattern for technology-specific language or assumptions. Rewrite those patterns in technology-neutral terms, preserving the problem, solution, and constraints while removing implementation-specific language. The rewritten patterns become the authoritative version; the old version is archived. This is a significant effort, but it is the pattern catalog's primary value proposition: it should survive technology transitions.

  5. If an accessibility regression is discovered in a pattern update: Immediately revert the pattern to the previous version in the catalog. Fix the accessibility issue in the updated version. Test the fix against all accessibility requirements defined in the pattern and in D16-006. Only re-deploy the updated version after the fix is verified. Record the regression, its cause, and the fix in the pattern's revision history and in this article's Commentary Section.

Evolution Path

  • Years 0-5: Establish the foundational pattern catalog as defined in this article. Implement each pattern as a reference example. Build the pattern registry infrastructure. Expect rapid growth in the catalog as the interface reveals interaction needs not anticipated in the initial set. This is healthy; the catalog should grow during this period. Resist the urge to over-specify; keep patterns at the level of problem and solution, not at the level of pixel-perfect specification.

  • Years 5-15: The catalog should stabilise. New patterns are added less frequently because the foundational set covers most needs. Focus shifts to refinement: improving existing patterns based on user feedback and accessibility audits. The first deprecations may occur as early patterns are superseded by better solutions. This is also the period when the pattern language proves its value in onboarding new designers who can learn the vocabulary from the catalog.

  • Years 15-30: A technology transition is likely. The pattern catalog should provide the continuity bridge: new implementations of the same patterns on new technology. If the catalog was kept technology-neutral per R-D16-14-05, this transition is straightforward. If technology-specific language crept in, this is when the cost of that leakage is paid. Conduct a full catalog review during any technology transition.

  • Years 30-50: The pattern catalog is one of the institution's most valuable design assets. It contains decades of accumulated knowledge about what interaction patterns work and which do not. New entries in the "Deprecated" section tell the story of the institution's interface evolution as clearly as any historical document. Protect the catalog per D20 standards. It is not just a design tool; it is institutional memory in its most practical form.

Founder Commentary

This section is reserved for dated entries by current and future operators.

2026-02-17 -- Founding Entry: The initial catalog contains approximately 30 patterns. I expect this number to grow to perhaps 50-70 over the first five years and then stabilise. If it grows to hundreds, something has gone wrong -- either patterns are being defined at too granular a level, or the institution's interface has become excessively complex.

I have chosen the name "pattern language" deliberately, invoking Alexander's concept. The key insight from Alexander is that patterns are not isolated solutions; they form a language in which each pattern connects to others. The Confirmation Dialog pattern makes sense only in the context of the Error Message pattern and the Binary Choice pattern. The patterns must be read as a network, not as a list. Future maintainers should resist the temptation to treat the catalog as a flat inventory and should instead maintain the relational structure (Composed Of, Related) that gives the catalog its power.

The technology-independence rule (R-D16-14-05) will be the hardest rule to enforce because it requires ongoing vigilance. When you implement the same pattern in the same framework for ten years, the framework's idioms start to feel like the pattern's requirements. They are not. The pattern is "present a choice and confirm the user's intent." The framework is how you happen to achieve that today. Future maintainers: guard this boundary ruthlessly. It is the pattern catalog's immune system against technological obsolescence.
