D13-009 — Documentation Evolution and Format Migration

Document ID
D13-009
Title
Documentation Evolution and Format Migration
Domain
13 -- Evolution & Adaptation
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
ETH-001, CON-001, D13-001, D13-008, D13-010, D20-001, META-001
Depended Upon By
All domains (documentation is cross-cutting). Referenced by META-001 through META-015 for documentation standards.

Purpose

This article governs how the documentation set itself evolves over time. The holm.chat documentation institution is, at its core, a documentation system. The articles, frameworks, procedures, and records that compose it are not ancillary to the institution -- they are the institution. When the documentation format changes, the institution itself is being transformed. This demands extraordinary care.

Documentation evolution encompasses three distinct activities. First, format migration: changing the file format, markup language, or structural conventions used to store documentation. Second, structural restructuring: reorganizing how documents are arranged, categorized, and cross-referenced. Third, terminological evolution: updating the language and terminology used throughout the documentation set as the institution's understanding and context evolve.

Each of these activities carries risks that are specific to documentation. Format migration risks breaking cross-references and losing formatting semantics. Structural restructuring risks orphaning documents and breaking navigational expectations. Terminological evolution risks creating inconsistencies between updated and not-yet-updated documents. This article addresses all three categories with procedures that preserve the integrity and navigability of the documentation set through every transformation.

Scope

In scope: - Documentation format selection and migration procedures. - Cross-reference integrity maintenance during documentation changes. - Structural reorganization procedures (adding domains, merging articles, splitting articles). - Terminology update procedures and consistency verification. - Documentation version management (complementing D13-006 for the specific case of documentation). - The documentation migration checklist. - Backward compatibility for documentation readers.

Out of scope: - Content authoring standards (META-001 and the meta-documentation domain). - Data format migration for non-documentation data (D13-008). - Institutional memory preservation policies (D20-001). - Specific documentation tooling decisions (these change; procedures endure).

Background

3.1 Documentation as the Institution's Core Asset

In most organizations, documentation is a supporting function -- it describes the real systems. In this institution, documentation is the primary output. The documentation set is the institution's product, its archive, its memory, and its operational manual. Migrating the documentation is not like migrating a database -- it is like migrating the institution's brain. Every neuron must arrive intact, and every connection between neurons must be preserved.

3.2 The Cross-Reference Problem

The documentation set is a web of cross-references. Articles reference other articles by ID. Sections reference other sections. Rules reference principles. Procedures reference rules. When any element moves, renames, or restructures, these references must be updated or they become dead links -- pointing to locations that no longer exist or that now contain different content. Cross-reference integrity is the single greatest challenge in documentation evolution.

3.3 The Terminology Drift Problem

Over decades, the institution's language will evolve. Terms that were precise at founding may become ambiguous as context changes. New concepts will emerge that require new terms. Old terms may acquire connotations that were not present at founding. Managing this evolution without creating confusion -- where the same term means different things in different parts of the documentation -- requires deliberate coordination.

System Model

4.1 Documentation Format Selection

The documentation format must satisfy the canonical format criteria from D13-008 Section 4.2, plus additional criteria specific to documentation:

  • Structural markup. The format must support semantic structural markup: headings, sections, lists, tables, and metadata. Purely visual formatting (bold, italic) is secondary to structural semantics.
  • Cross-referencing. The format must support internal links (references between documents) and anchors (references to specific locations within a document).
  • Human authoring. The format must be writable by a human using a text editor. Formats that require specialized authoring tools create tool dependencies that conflict with D13-001's modularity principle.
  • Rendering independence. The format must be readable in its raw form (without rendering) and renderable into a human-friendly presentation. This ensures that even if the rendering tool is lost, the content remains accessible.

The current documentation format (semantic HTML) satisfies all of these criteria. Future format changes must demonstrate superiority on these criteria through the Evolution Decision Framework (D13-001) before they are adopted.

4.2 The Documentation Migration Checklist

When the documentation format changes, every document in the set must be migrated. The checklist ensures completeness:

  • [ ] Complete inventory of all documents in the current format, including article count, total cross-reference count, and metadata structure.
  • [ ] Target format selected and validated against the criteria in Section 4.1.
  • [ ] Conversion tool selected and validated on a representative sample (minimum 10% of documents).
  • [ ] Cross-reference mapping created: for every cross-reference in the current format, the equivalent reference in the new format is documented.
  • [ ] Conversion executed for all documents.
  • [ ] Post-conversion verification: every document opens correctly in the new format, every cross-reference resolves correctly, every metadata field is preserved.
  • [ ] Navigation structure (sidebar, index, domain maps) updated for the new format.
  • [ ] Original-format copies archived per D13-008 R-DFM-04.
  • [ ] Documentation version incremented per D13-006.
  • [ ] Migration recorded in institutional timeline (D20-003).

4.3 Cross-Reference Integrity Procedures

Cross-reference integrity is maintained through three mechanisms:

Stable identifiers. Every document has a stable identifier (e.g., D13-009) that does not change even if the document's filename, location, or format changes. Cross-references use stable identifiers, not filenames or paths. When the documentation format changes, a mapping table from stable identifiers to new locations is created and verified.

Reference auditing. After any structural change (format migration, reorganization, document addition or removal), a reference audit is performed. The audit checks every cross-reference in the documentation set and verifies that it resolves to a valid target. Broken references are flagged and corrected before the change is considered complete.

Redirect or alias support. When a document moves or is restructured, the old location includes a pointer to the new location for a minimum of 12 months. This ensures that any external notes, bookmarks, or mental models that reference the old location can still reach the content.

4.4 Structural Reorganization Procedures

When the documentation structure must change -- adding a new domain, merging two articles into one, splitting one article into two, or reorganizing the domain hierarchy -- the following procedure applies:

  1. Document the proposed reorganization: what is changing and why.
  2. Impact assessment per D13-007: which cross-references are affected, which navigation elements must be updated, which index documents must be revised.
  3. Execute the reorganization in a staging copy of the documentation set, not in the production copy.
  4. Perform a reference audit on the staging copy.
  5. Verify that every document in the staging copy is accessible through the navigation structure.
  6. If verification passes, promote the staging copy to production. Archive the previous structure.

4.5 Terminology Evolution Procedures

When a term needs to be changed across the documentation set:

  1. Document the change: old term, new term, reason for change, scope of change (all documents, specific domain, specific articles).
  2. Perform a complete search for the old term across the entire documentation set. Record every occurrence.
  3. For each occurrence, determine whether the replacement is appropriate. Not every occurrence of a word should be changed -- context matters.
  4. Execute the replacements. For each replacement, verify that the surrounding context remains coherent.
  5. Add the terminology change to a terminology changelog that records all terminology evolution events with dates and rationale.
  6. If the old term appears in cross-references, stable identifiers, or external-facing names, establish aliases or redirects per Section 4.3.

Rules & Constraints

  • R-DEF-01: Documentation format changes are Tier 2 decisions under GOV-001. They must follow the Evolution Decision Framework and the documentation migration checklist (Section 4.2).
  • R-DEF-02: Cross-reference integrity must be verified after any documentation change that affects document locations, identifiers, or structure. Broken references must be corrected before the change is considered complete.
  • R-DEF-03: Stable document identifiers (e.g., D13-009) may not be reassigned. Once assigned, an identifier permanently refers to its original document, even if the document is archived or retired. New documents receive new identifiers.
  • R-DEF-04: Terminology changes affecting more than 10 documents must be treated as a Tier 3 decision with documented justification. Terminology changes affecting foundational terms (defined in ETH-001 or CON-001) are Tier 1 decisions.
  • R-DEF-05: All documentation evolution activities must preserve the anti-revisionism principle from D20-001. Past commentary, decision records, and historical entries are not modified, even when terminology changes. If an old term appears in a historical entry, it remains -- a footnote may be added for clarity.
  • R-DEF-06: A terminology changelog must be maintained that records every terminology change with the date, old term, new term, reason, and scope. The changelog is a permanent institutional record.

Failure Modes

  • Cross-reference rot. Over time, small documentation changes accumulate broken cross-references that are never audited. The documentation becomes a web of dead links. Mitigation: R-DEF-02 mandates verification after changes. Periodic full-set reference audits (at least annually) catch accumulated breakage.
  • Format lock-in. The documentation format proves harder to migrate than expected. The institution delays migration and the format ages. Mitigation: the criteria in Section 4.1 favor formats that are inherently migratable (text-based, structured, tool-independent). The raw content should always be accessible even if the rendering layer changes.
  • Terminology inconsistency. A terminology change is applied to some documents but not others. The same concept is referred to by different terms in different parts of the documentation. Mitigation: the complete-search requirement in Section 4.5 Step 2 and the terminology changelog (R-DEF-06) provide systematic tracking of changes.
  • Structural fragmentation. Repeated reorganizations fragment the documentation set into a structure that no longer reflects the institution's actual organization. Documents are hard to find. Related content is scattered across distant parts of the hierarchy. Mitigation: structural reorganizations follow the procedure in Section 4.4, which includes impact assessment and verification. The documentation structure is reviewed during the annual institutional review.

Recovery Procedures

  1. If cross-reference rot is detected: Perform a full reference audit across the entire documentation set. For each broken reference, determine the correct target and repair it. If the target no longer exists, create a redirect or update the reference to point to the most relevant surviving content. Document the audit results and the repairs made.
  2. If a documentation format migration has failed: Restore the pre-migration documentation set from archive. Investigate the failure: was the conversion tool defective, the target format unsuitable, or the migration procedure inadequate? Address the root cause before reattempting.
  3. If terminology inconsistency is discovered: Consult the terminology changelog to determine the intended state. Perform a fresh search for both old and new terms. Correct all inconsistencies. Update the changelog with the correction.
  4. If structural fragmentation has occurred: Conduct a documentation architecture review. Map the current structure against the institutional domain model. Propose a reorganization that restores alignment. Execute the reorganization through the procedure in Section 4.4.

Evolution Path

  • Years 0-5: The documentation format is new and well-supported. Evolution activity is focused on content growth, not format change. Use this period to establish cross-reference discipline and begin the terminology changelog. Small structural adjustments may be needed as the documentation set grows beyond its initial plan.
  • Years 5-15: The first documentation format questions arise. The original format may be aging, or superior alternatives may have emerged. The Evolution Decision Framework is applied to evaluate whether a format migration is justified. If a migration proceeds, it is the first major test of this article's procedures.
  • Years 15-30: The documentation set has undergone at least one format migration and multiple structural adjustments. The terminology changelog has grown into a linguistic history of the institution. Cross-reference integrity procedures are well-practiced. The documentation format may have changed again, and the migration procedures have been refined by experience.
  • Years 30-50+: The documentation set has survived multiple format paradigm shifts. The earliest documents have been migrated through several formats. The format genealogy (maintained per D13-008) traces each document's journey. The documentation set is recognizably continuous -- the same institution, the same principles, the same knowledge -- even though its physical format is entirely different from the founding era.

Founder Commentary

This section is reserved for dated entries by current and future operators.

2026-02-17 -- Founding Entry: I chose semantic HTML for the documentation format because it is the closest thing to a universal document format that exists today. It is text-based, human-readable in raw form, renderable in any web browser, and structurally expressive enough to support the full range of documentation needs. But I am under no illusion that HTML will be the format fifty years from now. What I am confident of is that when the time comes to migrate, the structured nature of HTML will make migration tractable. The content is clearly separated from presentation. The cross-references are explicit. The metadata is embedded. These properties are more important than the specific format -- they are what I will look for in any successor format. The most delicate aspect of documentation evolution is the tension with anti-revisionism. Old documents must not be silently altered. When terminology changes, when structure changes, when the format changes, the history of what was written and when it was written must be preserved. This article's procedures are designed to honor that principle while still allowing the documentation to evolve.
