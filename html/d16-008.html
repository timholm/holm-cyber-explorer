D16-008 — Input Method Abstraction Layer

Document ID
D16-008
Domain
16 -- Interface & Navigation
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
D16-001 (Interface Philosophy), D16-006 (Accessibility Standards)
Depended Upon By
D16-009 (Error States), D16-013 (Onboarding), D16-014 (Component Catalog)

Purpose

This article defines the abstraction layer that separates the institution's interface logic from any specific input method. Keyboard, pointer device, touchscreen, voice command, assistive switches, screen readers, and methods that do not yet exist must all be treated as interchangeable pathways to the same set of institutional actions. No input method may be privileged. No input method may be excluded.

The air-gapped, off-grid nature of this institution means that the hardware environment will change unpredictably. A machine equipped with a full keyboard and mouse today may be replaced by a tablet with only a touchscreen tomorrow, or by a device with an input mechanism that was not commercially available when this article was written. The abstraction layer exists to ensure that the institution's interface remains fully operational across all these transitions without requiring a redesign of the interaction model each time the hardware changes.

This article is addressed to every person who implements interactive elements, writes event handlers, designs input flows, or evaluates whether an interface component is accessible across input modalities. It is equally addressed to the future maintainer who must integrate an input technology that the original authors could not have anticipated.

Scope

This article covers:

  • The architecture of the input method abstraction layer.
  • The canonical set of abstract input actions that all interface components must support.
  • Requirements for keyboard operability as the universal baseline.
  • Requirements for pointer, touch, and voice input support.
  • Requirements for assistive technology compatibility.
  • The process for integrating new input methods into the abstraction layer.
  • Focus management, tab order, and spatial navigation principles.
  • Input feedback requirements across modalities.

This article does not cover:

  • The visual appearance of interactive elements (covered by D16-007).
  • Error messages and system feedback content (covered by D16-009).
  • The specific hardware abstraction layer at the operating system level (covered by Infrastructure domains).
  • Authentication input flows (covered by Security domains, though this article's principles apply to their user-facing presentation).

Background

The History of Input Assumptions

Every generation of interface design has been shaped by assumptions about how users provide input. Mainframe interfaces assumed a keyboard. Early graphical interfaces assumed a mouse. Mobile interfaces assumed a touchscreen. Each transition broke interfaces built on the previous assumption. Keyboard shortcuts became unreachable on touchscreens. Hover states became meaningless on devices without pointers. Drag-and-drop operations became impossible with keyboard-only navigation.

The institution cannot afford to build on any single input assumption because it cannot predict which input devices will be available in twenty or forty years. The abstraction layer described in this article addresses this by defining interface actions in terms of abstract intents -- "activate," "navigate," "select," "dismiss" -- rather than in terms of specific physical gestures. Each input method provides its own mapping from physical gestures to abstract intents, and the interface responds to the intents, not to the gestures.

Keyboard as Universal Baseline

Among all input methods, the keyboard occupies a special position: it is the only input method that can express every possible abstract intent. A pointer can click but cannot easily express "go to the next item in a list" without a visible target. A touchscreen can tap but cannot easily express "open the context menu for this item" without a long-press convention that many users do not know. Voice input can command but relies on the user knowing the available commands. The keyboard, through combinations of keys, can express any action without requiring spatial targeting, gestural knowledge, or discovery of hidden affordances.

For this reason, the keyboard is the universal baseline: every function the institution exposes through its interface MUST be achievable via keyboard alone. This is not merely an accessibility requirement (though it serves that purpose); it is an architectural requirement that ensures the interface can function even when the most constrained input method is the only one available.

The Air-Gap Implication

In a connected environment, input method detection and adaptation can rely on external libraries, browser capabilities, and cloud-based accessibility services. The institution has none of these. Input method support must be self-contained. Detection of available input methods must be done locally. Adaptation must be handled by locally stored code and configuration. This reinforces the need for a clean abstraction layer: the fewer assumptions the interface makes about input, the less it breaks when assumptions prove wrong.

System Model

The input abstraction layer consists of four components:

INPUT ABSTRACTION ARCHITECTURE
================================

+-------------------+     +-------------------+     +-------------------+
|  PHYSICAL INPUT   | --> |  INPUT ADAPTER    | --> |  ABSTRACT INTENT  |
|  (hardware event) |     |  (per-method)     |     |  (canonical action)|
+-------------------+     +-------------------+     +-------------------+
                                                            |
                                                            v
                                                    +-------------------+
                                                    |  INTERFACE LOGIC  |
                                                    |  (responds to     |
                                                    |   intents only)   |
                                                    +-------------------+

ABSTRACT INTENT CATALOG:
========================
Intent              Keyboard        Pointer         Touch           Voice
------              --------        -------         -----           -----
Activate            Enter/Space     Click           Tap             "Open/Select"
Navigate-Next       Tab / Arrow     (n/a)           Swipe-Right     "Next"
Navigate-Prev       Shift+Tab       (n/a)           Swipe-Left      "Previous"
Navigate-Into       Enter / Right   Click           Tap             "Open"
Navigate-Out        Escape / Left   Back-btn        Swipe-Down      "Back"
Select              Space           Click           Tap             "Select [name]"
Multi-Select        Shift+Click     Ctrl+Click      Long-press+Tap  "Also select"
Context-Menu        Shift+F10       Right-click     Long-press      "Options"
Dismiss             Escape          Click-outside   Swipe-Down      "Close/Cancel"
Search              Ctrl+F / /      Click-search    Tap-search      "Search for..."
Scroll              Page/Arrow      Scroll-wheel    Swipe           "Scroll down"
Confirm             Enter           Click-confirm   Tap-confirm     "Confirm/Yes"
Cancel              Escape          Click-cancel    Tap-cancel      "Cancel/No"

FOCUS MANAGEMENT:
=================
- Exactly one element has focus at any time
- Focus MUST be visible (minimum 2px outline, 3:1 contrast against background)
- Focus order MUST match reading order (DOM order in web contexts)
- Focus MUST NOT be trapped in any component (Escape always exits)
- Modal dialogs MUST trap focus within themselves until dismissed
- Focus MUST be restored to the triggering element when a modal is dismissed

Adapter Registration

Each input method is supported by an adapter that translates physical events into abstract intents. Adapters are registered at system startup and can be added or removed without modifying the interface logic. This registration model allows future input methods to be integrated by writing a new adapter and registering it, without touching any existing code.

Rules & Constraints

  • R-D16-08-01: Keyboard Completeness. Every function exposed through the interface MUST be achievable using only the keyboard. There are no exceptions. If a function cannot be expressed through keyboard interaction, it must be redesigned until it can. This rule is the foundation of input method neutrality.

  • R-D16-08-02: No Input-Exclusive Features. No feature or function may be available exclusively through a single input method. If a context menu is available via right-click, it MUST also be available via keyboard (Shift+F10 or equivalent) and via touch (long-press). If a drag-and-drop reordering is available via pointer, an alternative reordering mechanism MUST exist for keyboard and touch users.

  • R-D16-08-03: Visible Focus at All Times. The currently focused element MUST be visually indicated at all times when keyboard or sequential navigation is in use. The focus indicator MUST have a contrast ratio of at least 3:1 against adjacent colours. The focus indicator MUST NOT be suppressed, hidden, or made invisible for aesthetic reasons.

  • R-D16-08-04: Predictable Focus Order. The order in which elements receive focus when navigating sequentially MUST match the visual reading order of the content. Focus order MUST NOT jump unpredictably between distant areas of the screen. Custom tab-index values SHOULD be avoided except where the natural DOM order cannot be made to match the visual order.

  • R-D16-08-05: No Time-Dependent Input. No interface action may require the user to perform input within a specific time window. Double-click timing, long-press duration thresholds, and gesture speed requirements MUST be configurable or MUST have a time-independent alternative. Users with motor impairments may operate at speeds that the default timing does not accommodate.

  • R-D16-08-06: Input Feedback Within 100ms. Every input action MUST produce perceptible feedback within 100 milliseconds. For keyboard input, this means the focused element must visually respond to the keypress. For pointer input, this means the target must acknowledge the click. For touch input, this means a visual response must appear at the touch point. If the action triggered by the input requires longer than 100ms to complete, the feedback must indicate that processing is underway (see D16-009 for detailed feedback requirements).

  • R-D16-08-07: Adapter Extensibility. The input adapter registration system MUST support the addition of new input method adapters without modification to existing adapters or to the interface logic layer. New adapters MUST conform to the abstract intent catalog. If a new input method introduces actions that cannot be mapped to existing abstract intents, the intent catalog MUST be extended through the standard amendment process before the adapter is deployed.

  • R-D16-08-08: No Gesture-Only Actions. No action may be available exclusively through a gesture (swipe, pinch, rotate, shake, or similar). Every gestural action MUST have an equivalent non-gestural alternative. Gestures are convenient shortcuts, not primary interaction paths.

Failure Modes

  • Focus loss. The keyboard focus becomes invisible or is moved to an element the user did not expect, leaving the user unable to determine where they are in the interface. Likelihood: high during interface modifications if focus management is not explicitly tested. Severity: critical for keyboard-only users. Detection: automated focus-tracking tests that verify focus visibility and logical order after every navigation action.

  • Input method bias. A developer optimises an interaction for one input method (typically pointer) and the alternatives for other methods are either missing or degraded. The interface technically supports all input methods but practically favours one. Likelihood: very high without enforcement. Severity: high (creates a tiered user experience). Detection: cross-input-method testing as part of every component review; require testing with keyboard-only, pointer-only, and touch-only before approval.

  • Adapter incompatibility. A new input method adapter is added that produces abstract intents at a rate or in a pattern that the interface logic does not handle correctly. For example, a voice adapter that emits rapid-fire "navigate-next" intents overwhelms a list component. Likelihood: moderate. Severity: medium. Detection: adapter integration testing with stress scenarios.

  • Timing assumption violation. An interface component relies on a timing assumption (e.g., distinguishing a click from a long-press based on a fixed 500ms threshold) that does not accommodate users with motor impairments. Likelihood: moderate. Severity: high for affected users. Detection: accessibility testing with adjusted timing settings; automated scan for hardcoded timing values.

  • Platform-level input API change. The underlying operating system or rendering platform changes its input event model, causing adapters to stop receiving events or to receive them in an unexpected format. Likelihood: moderate over 50 years. Severity: critical (can render the entire interface non-interactive). Detection: input event monitoring in automated test suites that fail loudly when expected events are not received.

Recovery Procedures

  1. If focus is lost or invisible: Immediately audit all interface states for focus visibility. Restore focus indicators to all interactive elements. Verify that every state transition (opening a dialog, closing a panel, navigating between pages) explicitly manages focus placement. Treat focus loss as a critical accessibility defect and prevent deployment of changes until resolved.

  2. If an input method is discovered to be non-functional: Determine whether the failure is in the adapter or in the interface logic. If the adapter is not receiving events, check the platform-level input API for changes. If the adapter is emitting intents but the interface is not responding, audit the intent handler. Restore function through the adapter layer without modifying the interface logic if possible. If interface logic changes are required, ensure they do not break other input methods.

  3. If a new input method cannot be mapped to existing intents: Do not force the new method into an inadequate mapping. Document the unmapped capabilities. Propose an extension to the abstract intent catalog through the D16 amendment process. Until the extension is approved, provide the new input method with a partial mapping that covers all critical actions (activate, navigate, dismiss, confirm, cancel) and document the limitations.

  4. If timing assumptions are excluding users: Audit all time-dependent interactions. Replace fixed timing thresholds with configurable values. Provide a system-wide timing sensitivity setting that adjusts all thresholds proportionally. Ensure that a fully time-independent alternative exists for every time-dependent interaction.

  5. If a platform input API change breaks all adapters: This is a critical infrastructure event. Freeze interface deployments. Write a compatibility shim that translates the new API events into the format the existing adapters expect. This shim buys time for a proper adapter rewrite. Document the API change, the shim, and the planned adapter migration in the D20 decision log.

Evolution Path

  • Years 0-5: Implement the abstraction layer with adapters for keyboard, pointer, and touch. These three cover the current hardware landscape. Build the adapter registration system from the start, even if only three adapters exist, to establish the pattern for future extension. Write comprehensive input tests that exercise every abstract intent through every adapter.

  • Years 5-15: Expect new input modalities to emerge. Voice input may become practical for institutional use. Gesture-based spatial input (hand tracking, eye tracking) may mature. Each new modality requires only a new adapter and possibly an extension to the intent catalog. The interface logic should not need to change.

  • Years 15-30: The original hardware platforms will have been replaced at least once. The adapter layer will have proven its value (or its inadequacy) through at least one major platform transition. If the abstraction has held, this is validation. If it has not, this is the moment to redesign the abstraction based on the actual failure patterns observed, rather than the theoretical failure patterns predicted in this article.

  • Years 30-50: Input paradigms may have shifted in ways that are not foreseeable today. The abstract intent model -- based on actions like "activate," "navigate," and "dismiss" -- is grounded in the structure of human-computer interaction rather than in any specific technology. If future input methods are so radically different that these intents no longer map naturally, this article should be rewritten. But the principle from D16-001 -- input method neutrality -- should persist regardless.

Founder Commentary

This section is reserved for dated entries by current and future operators.

2026-02-17 -- Founding Entry: The abstraction layer described here is intentionally simple. There are only thirteen abstract intents. A more complex system could model more nuanced interactions, but complexity in an abstraction layer defeats its purpose. The abstraction must be simple enough that a new adapter author can understand the entire intent catalog in an hour and write a mapping in a day.

I am particularly firm on the keyboard-completeness rule (R-D16-08-01). Every accessibility expert I have consulted agrees: if it works with a keyboard, it almost certainly works with assistive technology. The keyboard is not just one input method among many; it is the Rosetta Stone that ensures all other methods can function. A keyboard-complete interface is, by construction, an interface that can be operated by almost any assistive device, because assistive devices typically emulate keyboard events as their lowest-common-denominator output.

The intent catalog will need to grow. I have resisted the temptation to anticipate every possible future intent. Better to start minimal and extend deliberately than to start comprehensive and discover that half the intents are never used.
