CON-003 — The Refusal Registry

Document ID
CON-003
Domain
1 -- Constitution & Philosophy
Version
1.0.0
Date
2026-02-17
Status
Draft

Purpose

This article is a registry of things the holm.chat Documentation Institution will never do. It is not a list of limitations. It is a list of commitments -- things the institution has examined, understood, and deliberately rejected. Every entry exists because there is a plausible reason someone might propose it, a coherent argument in its favor, and a principled reason the institution refuses.

Refusals are more durable than capabilities. A capability can be added at any time. A refusal, once established, should only be removed through the most rigorous deliberative process the institution possesses. This asymmetry is intentional. It is easier to add than to subtract, easier to permit than to forbid. The Refusal Registry exists to ensure that certain boundaries are crossed only after extraordinary deliberation, never through convenience or incremental erosion.

This document is addressed to every future maintainer who will face pressure -- internal or external, rational or emotional -- to do something the institution has decided it will not do. Before acting, consult this registry. If the proposed action appears here, the default answer is no. The burden of proof is on the person proposing to override the refusal, not on the refusal itself.

Scope

This article covers all categories of institutional refusal: technical, operational, ethical, and strategic. It applies to every domain and every system within the institution.

This article does not cover temporary restrictions, conditional limitations, or operational guidelines. Those belong in their respective domain articles. The refusals documented here are intended to be permanent -- or as close to permanent as any human commitment can be.

Each refusal entry includes: the refusal itself, the reasoning behind it, the scenarios under which it might be challenged, and the process for overriding it if circumstances demand. No refusal is absolute in the mathematical sense -- reality can produce situations no founder can anticipate. But every refusal documented here carries the highest possible burden of proof for override.

Background

Most institutions define themselves by what they do. This is insufficient. An institution that only defines what it does is vulnerable to mission creep, scope expansion, and the gradual accumulation of activities that individually seem reasonable but collectively transform the institution into something its founders would not recognize.

Defining what an institution refuses to do is a form of negative architecture. Just as a building is defined as much by the spaces it encloses as by the walls it erects, an institution is defined as much by its refusals as by its capabilities. The Refusal Registry is the institution's negative space -- the boundaries that give it shape.

The registry also serves as an institutional memory of temptation. Many of these refusals represent paths that are genuinely attractive. Connecting to the internet would be convenient. Using cloud services would be easier. Adopting proprietary formats would unlock features. The registry acknowledges these attractions honestly while documenting the reasons they are refused. This honesty is important: a refusal that pretends the refused option has no benefits is a refusal that will be overturned the first time those benefits become urgent.

System Model

Refusals are organized into categories and ranked by severity. Each refusal has a unique identifier for reference in other documents.

Category A: Network and Connectivity Refusals

REF-A01: The institution will never connect to the public internet during normal operation. The internet is an attack surface, a dependency, and a surveillance vector. Air-gapped operation is not a feature of this institution; it is the institution's fundamental security posture. Limited, controlled, temporary connections for specific data import purposes are governed by Domain 18 (Import & Quarantine) and require a documented justification, a threat assessment, and immediate disconnection upon completion. "Normal operation" means everything that is not an explicitly authorized import session.

REF-A02: The institution will never depend on any external network for any core function. Core functions include: data storage, data retrieval, computation, documentation access, backup, and system administration. None of these functions may require a network connection that exits the institution's physical boundary. Internal networking between institutional systems is permitted and expected.

REF-A03: The institution will never transmit data to any external endpoint without explicit, per-transmission authorization from the operator. There is no standing permission for outbound data transmission. Every transmission is a discrete event requiring discrete approval.

Category B: Dependency Refusals

REF-B01: The institution will never depend on a commercial vendor for any critical function. A critical function is any function whose failure would compromise data integrity, system availability, or institutional operation. Vendor products may be used for non-critical functions, but only with a documented alternative that can be deployed if the vendor relationship ends. The institution does not assume any vendor will exist in ten years, let alone fifty.

REF-B02: The institution will never depend on a subscription service. Subscriptions create recurring dependency. If a subscription lapses -- whether through financial hardship, vendor decision, or infrastructure disruption -- the capability it provides disappears. Every capability must be owned, not rented.

REF-B03: The institution will never use software that requires external activation, license verification, or phone-home capability. Software that must contact an external server to function is software that can be disabled remotely. This is incompatible with sovereignty.

Category C: Data Handling Refusals

REF-C01: The institution will never store data in a proprietary format as the sole copy. Every piece of data must exist in at least one open, documented format that can be read without proprietary software. Proprietary formats may be used for working copies if they provide genuine operational advantage, but the canonical archive must use open formats.

REF-C02: The institution will never store data on behalf of an external party. The institution exists to serve its operator. It is not a hosting service, a data repository, or a custodial platform for others. Data from external sources may be imported through the quarantine process defined in Domain 18, but only for the operator's use.

REF-C03: The institution will never implement automated data destruction without explicit operator confirmation. Automated systems may flag data for destruction. Only the operator may confirm it. This protects against both software errors and social engineering attacks that target automated deletion systems.

Category D: Operational Refusals

REF-D01: The institution will never sacrifice documentation completeness for development speed. No system is considered operational until its documentation is complete. This is defined in OPS-001 and reinforced here because the temptation to ship first and document later is universal and universally destructive on a long time horizon.

REF-D02: The institution will never grow beyond the architectural comprehension of a single operator. The operator must be able to hold a mental model of the entire institution's architecture. Individual components may be complex, but the overall architecture must be graspable by one mind. When architectural complexity approaches this limit, the institution must simplify before it grows.

REF-D03: The institution will never deploy a system it cannot fully audit. "Fully audit" means the ability to inspect source code, configuration, data flows, network connections, and resource consumption. Binary blobs, obfuscated code, and opaque containers are prohibited in production systems.

Category E: Ethical Refusals

REF-E01: The institution will never be used to cause deliberate harm to others. This refusal is stated in ETH-001 and reinforced here. It includes but is not limited to: attack infrastructure, surveillance tools targeting others, platforms for harassment, and repositories of data acquired through exploitation.

REF-E02: The institution will never misrepresent its own capabilities, limitations, or state. Documentation will never overstate what the institution can do. Recovery procedures will never promise certainty. Status reports will never hide problems. Honest accounting of the institution's condition is a non-negotiable ethical commitment.

Rules and Constraints

  • R-CON-003-01: Every refusal in this registry carries a default answer of "no." The burden of proof for override lies entirely with the person proposing the override, not with the refusal.
  • R-CON-003-02: Overriding a refusal requires: a written justification explaining why the specific circumstances warrant an exception, a risk assessment of the override, a plan for returning to compliance after the override period, and approval through the process defined in CON-011.
  • R-CON-003-03: Temporary overrides are permitted only for Category A and Category B refusals, and only under the specific conditions documented in each refusal entry. Categories C, D, and E refusals have no temporary override provision.
  • R-CON-003-04: New refusals may be added to this registry through the standard amendment process. Existing refusals may only be removed through the highest-tier amendment process defined in CON-011, with a minimum 180-day deliberation period.
  • R-CON-003-05: Every override of a refusal must be recorded in the institutional log with: the date, the refusal identifier, the justification, the approver, the expected duration, and the actual duration. This log is reviewed annually during the principle audit.

Failure Modes

Erosion by exception. A refusal is overridden once for good reasons. Then again. Then again. Each exception is individually justified, but collectively they render the refusal meaningless. This is the primary failure mode of any refusal system. Mitigation: the override log creates a visible record of how often each refusal has been overridden. If a refusal is overridden more than twice in any five-year period, a formal review must determine whether the refusal should be amended or whether the overrides represent a compliance failure.

Registry stagnation. The world changes, but the registry does not. Refusals that once made sense become obstacles to legitimate institutional needs. The registry is treated as sacred text rather than living architecture. Mitigation: the annual principle audit includes a review of every refusal for continued relevance. The Evolution Path section below provides guidance on recognizing when a refusal may need amendment.

Incomplete coverage. A harmful practice is not in the registry because no one anticipated it. The absence of an explicit refusal is taken as permission. Mitigation: the registry is supplemented by the general principles in CON-002. When a situation arises that is not covered by a specific refusal but appears to violate a general principle, the general principle governs until a specific refusal can be drafted.

Performative compliance. The letter of a refusal is observed while its spirit is violated. For example: the institution does not "connect to the public internet" but connects to a private relay that connects to the public internet. Mitigation: each refusal is accompanied by its reasoning. Compliance must be evaluated against the reasoning, not just the text.

Recovery Procedures

  1. If a refusal has been violated: Stop the violating activity immediately. Document the violation in the institutional log: what was done, when, by whom, and what the consequences were. Assess whether the violation compromised any data, system, or principle.
  2. If the violation was accidental: Determine how it happened and implement safeguards to prevent recurrence. Add an entry to this article's Commentary Section describing the incident and the safeguard.
  3. If the violation was deliberate but unauthorized: This is a governance failure. Escalate to the authority model defined in GOV-001. Document the incident thoroughly. Review whether the governance structures are sufficient to prevent recurrence.
  4. If the violation was deliberate and authorized but not through the proper override process: This is a process failure. The authorization is retroactively reviewed through the proper override process. If the override would not have been approved, corrective action is taken as if the violation were unauthorized.
  5. If a pattern of violations indicates the refusal is impractical: Initiate the amendment process. Do not continue violating the refusal while the amendment is under review. Either comply with the current refusal or formally suspend it through the override process while the amendment proceeds.

Evolution Path

Years 0-5: The registry will be tested by reality. Expect to discover refusals that are too broad (prohibiting legitimate activities) and gaps where refusals are needed but absent. Additions should be made promptly when gaps are found. Amendments to existing refusals should be rare and carefully considered.

Years 5-15: The registry should be stable. The focus shifts to building an interpretive record through the Commentary Section and the override log. Patterns of overrides may suggest that certain refusals need refinement.

Years 15-30: Technology may have changed enough that some refusals need reframing. The core refusals (Categories D and E) should remain stable. Network and dependency refusals (Categories A and B) may need updated language to address technologies that did not exist at founding.

Years 30-50+: The registry must be evaluated as a whole. Does it still define meaningful boundaries? Have any refusals become obsolete? Have new threats emerged that require new refusals? The institution's identity depends on maintaining meaningful boundaries, but those boundaries must be relevant to the actual threats and temptations the institution faces.
