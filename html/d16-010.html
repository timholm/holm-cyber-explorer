D16-010 — Multi-Generational Interface Continuity

Document ID
D16-010
Domain
16 -- Interface & Navigation
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
D16-001 (Interface Philosophy), D16-014 (Component Catalog)
Depended Upon By
D16-012 (Localization), D16-013 (Onboarding)

Purpose

This article addresses the central challenge of a fifty-year interface: how to evolve the interface over time without disorienting the users who depend on it. The institution will outlive every design trend, every hardware generation, and every aesthetic sensibility of its founding era. Its interface must change. But change, handled badly, is indistinguishable from destruction. A user who opens the institution one morning to find that everything has moved, been renamed, or been reorganised will experience not an upgrade but a loss.

This article defines the strategies, processes, and constraints that govern interface evolution. It ensures that change happens deliberately, gradually, and with full documentation of what changed, why, and how users can adapt. It also ensures that the institution retains the ability to explain its own interface history -- why the interface looks the way it does, what it used to look like, and what decisions led to the current state.

This article is addressed to every person who will propose, design, approve, or implement a change to the institution's interface. It is also addressed to the user who experiences a change and needs to understand what happened and why.

Scope

This article covers:

  • The classification of interface changes by scope and impact.
  • The change governance process: who may propose, approve, and implement changes.
  • Transition strategies for major interface changes.
  • User notification requirements for interface changes.
  • Interface versioning and the preservation of interface history.
  • The relationship between interface evolution and the four-layer model defined in D16-001.
  • Criteria for when interface change is necessary versus when it is merely desired.
  • Protection against both stagnation (refusing to change when change is needed) and churn (changing when stability is needed).

This article does not cover:

  • The specific visual design of any particular interface version (covered by D16-007).
  • The content or structure of the information architecture (covered by D16-002; changes to IA are the most impactful and are governed by that article's own evolution process).
  • Localization changes that adapt the interface for different languages or cultures (covered by D16-012, though this article's transition strategies apply).
  • Quality assurance procedures for validating interface changes (covered by Domain 19).

Background

The Problem of Interface Entropy

Interfaces decay in two ways. The first is stagnation: an interface that never changes becomes progressively more alien as the conventions around it evolve. Users who grew up with different interaction paradigms find it confusing. Metaphors that were obvious at launch become obscure. The interface fossilises while the world around it moves on. The second is churn: an interface that changes too frequently never allows users to develop fluency. Each change resets the user's learning curve. Muscle memory is disrupted. Landmarks are removed. The user who was expert yesterday is a novice today.

The institution must navigate between these two hazards. It must change -- but only when change serves the user, and only at a pace the user can absorb. This requires explicit governance, because without governance, the pace of change is set by whoever has the ability to modify the interface, not by what the users can absorb.

Lessons from Long-Lived Institutions

Physical institutions offer useful precedent. Libraries, churches, and government buildings evolve their physical interfaces over decades. The principles they follow are instructive: change the interior while preserving the entrance (users must always be able to find the front door). Change one section at a time, not everything simultaneously. Mark changes clearly (new signage, temporary notices, guided maps). Preserve the core wayfinding structures (main corridors, central desks, stairways) even as peripheral areas are remodelled.

The institution's digital interface should follow the same principles. The primary navigation structures are the main corridors. The search system is the central desk. The information architecture is the building layout. These change only when the institution's purpose changes. Peripheral elements -- visual styling, interaction refinements, new convenience features -- can change more frequently, because their change does not disorient the user's fundamental understanding of where things are.

The Layer Model Revisited

D16-001 defines four layers: philosophical, structural, interaction, and presentation. Each layer has a different rate of acceptable change. The philosophical layer should change only in response to fundamental institutional shifts. The structural layer should change only when the institution's content scope or organisation genuinely requires it. The interaction layer should change when new input modalities or user research justify it. The presentation layer may change every five to ten years to remain visually current. This article governs the process by which changes at each layer are proposed, evaluated, approved, and communicated.

                          mission changes (GOV framework)  ETH-001 process


TRANSITION PROTOCOL FOR MAJOR CHANGES
=======================================

Phase 1: ANNOUNCE (90 days before)
  - Publish change description accessible from every interface page
  - Describe what will change, what will not change, and why
  - Provide timeline with specific dates

Phase 2: PARALLEL (30 days, overlapping with announcement)
  - Both old and new interface available simultaneously
  - User can switch between them at will
  - All data and state is shared; only presentation/structure differs

Phase 3: DEFAULT FLIP (Day 0)
  - New interface becomes the default
  - Old interface remains available via explicit "use previous version" option
  - Transition guide prominently displayed on first load

Phase 4: SUNSET OLD (90 days after default flip)
  - Old interface is removed
  - Change documentation is archived per D20 standards
  - Any user feedback received during transition is logged in Commentary

Interface Changelog

The institution maintains a permanent, append-only changelog of all interface modifications. Each entry records: the date, the change class, a description of what changed, the rationale for the change, the articles or standards that authorised the change, and any user feedback received. The changelog is itself navigable through the interface and is never purged.

Rules & Constraints

  • R-D16-10-01: No Surprise Changes. No interface change that affects user workflow may be deployed without advance notification. Minor changes (visual polish, bug fixes that do not alter behaviour) may be deployed with changelog entries only. Moderate changes require 30 days' advance notice. Major changes require 90 days' advance notice and a parallel operation period.

  • R-D16-10-02: Change Justification Required. Every proposed interface change MUST include a written justification that answers three questions: (1) What problem does this change solve? (2) Who is affected by the current state? (3) Why is this change the right solution, and what alternatives were considered? Changes motivated solely by aesthetic preference or trend-following MUST be rejected unless they demonstrably improve usability.

  • R-D16-10-03: One Major Change at a Time. No more than one major (structural) interface change may be in progress at any time. This prevents the compounding disorientation that occurs when multiple foundational elements shift simultaneously. If multiple structural changes are needed, they MUST be sequenced with sufficient time between them for users to adapt to each one.

  • R-D16-10-04: Preserve Navigation Landmarks. The primary navigation entry points -- the main menu, the search bar, the home link, and the breadcrumb trail -- MUST NOT be removed or relocated in a major change without a transition period that maintains both the old and new locations simultaneously. These are the user's wayfinding anchors. Moving them without transition is equivalent to rearranging the streets of a city overnight.

  • R-D16-10-05: Interface History Preservation. Screenshots, descriptions, or other records of every major interface version MUST be archived per D20 standards. Future maintainers must be able to see what the interface looked like at any point in the institution's history. This serves both historical understanding and practical debugging (identifying when a specific interaction problem was introduced).

  • R-D16-10-06: User Feedback Integration. Every major interface change MUST include a mechanism for users to provide feedback during the transition period. This feedback MUST be reviewed before the old interface is sunset. If significant usability problems are reported, the sunset MUST be delayed until the problems are addressed.

  • R-D16-10-07: Rollback Capability. For any moderate or major interface change, a rollback plan MUST exist before deployment. If the change causes critical usability problems, it MUST be possible to revert to the previous version within 24 hours. The rollback plan MUST be tested before the change is deployed.

  • R-D16-10-08: Stagnation Review. If no moderate or major interface changes have been made for five years, a mandatory review MUST be conducted to determine whether the interface is still serving its users well or whether it has stagnated. Stagnation is not safety; it is deferred risk. The review must include user testing with participants who represent the current user population, not the founding population.

Failure Modes

  • Redesign shock. A major interface redesign is deployed without adequate transition, and long-time users are suddenly unable to find basic functions. Productivity drops. Trust erodes. Some users refuse to use the system until the old interface is restored. Likelihood: high without the governance process defined in this article. Severity: critical (can effectively remove users from the institution). Detection: user feedback during transition; task-completion metrics before and after the change.

  • Change aversion lock-in. The governance process becomes so burdensome that all interface changes are blocked. The interface stagnates. New users find it increasingly incomprehensible. Maintainers are unable to fix known usability problems because the approval process is prohibitive. Likelihood: moderate (governance processes tend toward increasing friction over time). Severity: high (stagnation is as harmful as churn, just slower). Detection: the five-year stagnation review (R-D16-10-08); tracking proposed changes that were abandoned due to process burden.

  • Incremental drift. Many small "minor" changes accumulate over time, each individually insignificant but collectively transforming the interface without any of the governance safeguards that apply to moderate or major changes. The interface of 2036 bears little resemblance to the interface of 2026, but no single change was large enough to trigger the transition protocol. Likelihood: high. Severity: moderate to high. Detection: periodic comparison of the current interface against archived screenshots from the previous review cycle; automated tracking of cumulative change magnitude.

  • Parallel period abuse. During the parallel operation period for a major change, users overwhelmingly prefer the old interface. Rather than investigating why, the maintainers sunset the old interface on schedule, assuming the preference is mere familiarity. In fact, the new interface has a genuine usability regression. Likelihood: moderate. Severity: high. Detection: R-D16-10-06 requires feedback review before sunset; additionally, task-completion metrics must be compared between old and new versions.

  • History loss. Interface versions are not archived. Future maintainers cannot understand why the interface looks the way it does because they cannot see what it evolved from. Design decisions that were reactions to specific problems appear arbitrary. Likelihood: moderate (archiving is easy to neglect). Severity: high over time (compounds with each lost version). Detection: D20 audit of interface history completeness as part of the standard institutional memory review.

Recovery Procedures

  1. If a redesign has caused user disorientation: Immediately activate the rollback plan. Restore the previous version while the new version's problems are identified and addressed. Conduct user testing to pinpoint the specific elements that caused disorientation. Do not re-deploy the new version until those elements are fixed. Extend the parallel operation period to give users more time to adapt.

  2. If the change process has become a bottleneck: Review the governance process itself. Distinguish between changes that genuinely need governance (structural, navigational) and changes that do not (bug fixes, minor visual adjustments). Streamline the process for lower-impact changes. Consider delegating minor and moderate change approval to a single domain lead rather than requiring committee review.

  3. If incremental drift is detected: Conduct a comprehensive comparison between the current interface and the last major-version archive. Document all differences. Classify each difference as intentional-and-documented, intentional-but-undocumented, or unintentional. For undocumented and unintentional changes, decide whether to formalise them (by documenting them) or to revert them. Update the changelog retroactively for all intentional-but-undocumented changes.

  4. If interface history has been lost: Reconstruct what can be reconstructed from available sources: screenshots in documentation, user-facing changelogs, version control history, and the memories of current participants (captured via D20-003 oral history methods). Document the gap in the historical record. Establish automated archiving to prevent future loss.

  5. If the interface has stagnated for more than five years: Conduct the mandatory stagnation review. Test the interface with new users who have no familiarity with it. Identify metaphors, conventions, or patterns that are no longer comprehensible. Develop a phased modernisation plan that addresses the most critical comprehension gaps first, following the transition protocol for each change.

Evolution Path

  • Years 0-5: The interface is new and will undergo rapid iteration. Most changes in this period will be classified as moderate or minor. Establish the changelog from the first day. Archive the first production version immediately. Begin building the user feedback infrastructure. The governance process during this period should be lightweight to avoid impeding necessary iteration, but every change must still be logged.

  • Years 5-15: The interface should stabilise. Changes become less frequent and more deliberate. The governance process can tighten as the cost of disruption increases (more users, more established habits). Expect the first stagnation review around year 10. This is the period where the discipline of documenting change rationale pays off: new maintainers arriving in this period can read the changelog to understand how the interface reached its current state.

  • Years 15-30: This is the highest-risk period for generational disconnect. Users who joined the institution in its second decade may find the founding-era interface conventions alien. A major presentation refresh is likely and appropriate. The transition protocol defined in this article exists for exactly this moment. The refresh must respect all structural and philosophical principles while updating the presentation and interaction layers.

  • Years 30-50: The interface will have been through at least one major refresh. The governance process itself should be reviewed: does it still serve the institution, or has it become an obstacle? The changelog should contain decades of institutional learning about what works and what does not. This accumulated knowledge is a strategic asset. Protect it per D20 standards.

Founder Commentary

This section is reserved for dated entries by current and future operators.

2026-02-17 -- Founding Entry: The hardest lesson in interface design is that your users' muscle memory is not your enemy. It is your greatest asset. A user who can navigate the interface without thinking is a user who can focus on the content. Every interface change disrupts that muscle memory, at least temporarily. Some disruption is necessary -- the interface must evolve -- but it must be weighed against the cost.

I have set the transition period for major changes at 90 days before and 90 days after, with a 30-day parallel period. These numbers are estimates based on organisational change research suggesting that people need roughly three months to fully adjust to a significant workflow change. Future maintainers should adjust these durations based on observed user behaviour, not based on impatience to ship the new version.

The one-major-change-at-a-time rule will be tested. There will come a moment when multiple structural improvements are urgently needed. Resist the urge to bundle them. Each major change should be experienced, absorbed, and evaluated independently. Sequential pain is recoverable. Simultaneous pain is catastrophic.
