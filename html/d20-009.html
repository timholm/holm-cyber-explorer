D20-009 — Memory Accessibility & Discoverability

Document ID
D20-009
Domain
20 — Institutional Memory
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
D20-011 (Memory Taxonomy), D20-002 (Decision Log)
Depended Upon By
D20-007 (Context Recovery, which depends on findable records)

Purpose

This article defines the requirements and standards for ensuring that institutional memory records are findable, readable, and useful to people who were not present when the records were created. It addresses the gap between possessing records and being able to use them -- a gap that can render an otherwise comprehensive memory system effectively useless.

The most complete decision log in the world is worthless if no one can find the entry they need. The most detailed oral history is wasted if it is buried in an unindexed archive. The most meticulously maintained timeline is invisible if the reader does not know it exists. Accessibility and discoverability are the bridge between memory as stored data and memory as usable knowledge.

This article is addressed to two audiences. The first is the institutional members who create and maintain memory records: they must understand that a record is not complete until it is indexed, cross-referenced, and written in language that a future reader can understand. The second audience is the future reader themselves: this article tells them where to look, how to search, and what to expect when they go looking for institutional memory.

Scope

This article covers:

  • Indexing requirements for all memory record types.
  • Cross-referencing standards: how records link to other records.
  • Search capabilities that the memory system must support.
  • Language and readability standards for memory records.
  • Entry points: how a new user discovers the memory system.
  • Navigation paths: how a user moves through the memory system to find what they need.
  • Accessibility for users with different skill levels, languages, and time horizons.

This article does not cover:

  • The general interface and navigation system (covered by Domain 16). This article defines memory-specific accessibility requirements that Domain 16 must implement.
  • The search system architecture (covered by D16-ART-004). This article defines what memory-specific search capabilities are needed.
  • The memory taxonomy that structures the records (see D20-011).
  • The quality metrics for memory accessibility (covered by Domain 19).

Background

3.1 The Accessibility Paradox

Institutional memory faces a paradox of accessibility: the records that are most needed are often the hardest to find. A current member looking for context on a recent decision can usually find it because they know where to look, when it was made, and what terms to search for. A future member looking for context on a founding-era decision may not know the vocabulary that was used, the organizational structure that existed at the time, or even that the decision was a distinct event rather than an implicit assumption.

This paradox means that accessibility cannot rely solely on the reader knowing what they are looking for. The system must also support discovery: the ability to find relevant records even when the reader does not know the right search terms, does not know the chronological context, and does not know the institutional structure that produced the record.

3.2 Language Drift Over Decades

Over a 50-year horizon, the language used within the institution will change. Terms that are common today may become obscure. Acronyms that are obvious to current members may be opaque to future readers. Concepts that are implicit in today's discourse may need to be explicit in tomorrow's records. Memory accessibility must account for this drift by requiring that records be written in clear, self-contained language that does not rely excessively on contemporary jargon or assumptions.

3.3 Multiple Access Patterns

Users access institutional memory through different patterns, each requiring different support:

  • Known-item search: "I need the decision log entry for Decision #247." This requires precise indexing and ID-based retrieval.
  • Topic search: "What decisions have been made about backup procedures?" This requires keyword indexing, topic tagging, and domain classification.
  • Chronological browsing: "What happened in 2035?" This requires the timeline (D20-003) and date-based navigation.
  • Person-based search: "What did Contributor X work on?" This requires the contributor registry (D20-006) and attribution indexing.
  • Exploratory discovery: "I do not know what I am looking for, but I need to understand the institution's history." This requires curated entry points, guided pathways, and cross-referencing that leads the reader from one record to related records.

System Model

4.1 The Memory Access Architecture

USER NEED
  |
  +-- Known item --> ID LOOKUP --> Direct retrieval
  |
  +-- Topic search --> KEYWORD/TAG INDEX --> Filtered results --> Record
  |
  +-- Chronological --> TIMELINE (D20-003) --> Date-linked records
  |
  +-- Person-based --> CONTRIBUTOR REGISTRY (D20-006) --> Attributed records
  |
  +-- Exploratory --> ENTRY POINTS --> GUIDED PATHWAYS --> CROSS-REFERENCES --> Records

4.3 Cross-Referencing Standards

Cross-references are the connective tissue of institutional memory. They transform isolated records into a navigable graph. Every memory record must include cross-references to:

  • The decision(s) it documents or relates to.
  • The article(s) in the documentation framework that it supports or is supported by.
  • The timeline event(s) it corresponds to.
  • Related records of any type that provide additional context.

Cross-references must be bidirectional where feasible: if Record A references Record B, Record B should reference Record A. If bidirectional linking is not technically feasible, a central cross-reference index must be maintained that maps relationships in both directions.

4.4 Entry Points for New Users

A new user encountering the institutional memory system for the first time needs structured entry points, not just a search bar. The memory system must provide:

  • The Institutional Timeline: A chronological overview that serves as a narrative entry point. "Start here to understand what happened."
  • The Founder's Summary: A concise document (maintained as part of D20-001 or a derivative) that explains the institution's origins, purpose, and major milestones. Updated at least every 5 years.
  • The Domain Map: A structural overview showing how the institution is organized and where different types of memory records live.
  • The "Start Here" Guide: A practical orientation document that tells a new user: "If you need to understand why something is the way it is, here is how to find out."

Rules & Constraints

  • R-D20-09-01: Every Record Is Indexed. No memory record may be considered complete until it is indexed along all dimensions specified in Section 4.2. An unindexed record is a lost record. The indexing must be completed within 14 days of the record's creation.

  • R-D20-09-02: Plain Language Requirement. Memory records must be written in clear, self-contained prose. Acronyms must be expanded on first use within each record. Jargon must be defined or avoided. A reader who is encountering the institution for the first time should be able to understand any individual record without needing to read other records first (though cross-references provide additional context).

  • R-D20-09-03: Cross-References Are Mandatory. Every memory record must include at least one cross-reference to a related record. Decision log entries must reference the relevant domain article. Commentary entries must reference the article they annotate. Oral history transcripts must reference the relevant decision log entries and timeline events. Orphan records -- records with no cross-references -- are a quality defect.

  • R-D20-09-04: Entry Points Are Maintained. The entry points defined in Section 4.4 must be kept current. The institutional timeline must be updated within 30 days of any significant event. The Founder's Summary must be reviewed at least every 5 years. The "Start Here" guide must be tested with each new member: can they navigate the memory system using only the guide?

  • R-D20-09-05: Search Must Support All Access Patterns. The memory system must support all five access patterns described in Section 3.3: known-item, topic, chronological, person-based, and exploratory. If the current implementation does not support all five, the unsupported patterns must be documented as known limitations and addressed in the evolution plan.

  • R-D20-09-06: Accessibility Survives Platform Changes. The indexing, cross-referencing, and entry points must not depend on a specific software platform. If the platform changes, the accessibility infrastructure must be migrated or reconstructed. Memory records that become inaccessible due to a platform change represent an institutional failure.

Failure Modes

  • Index rot. The index falls out of sync with the actual records. Records exist that are not in the index; index entries point to records that have been moved or reorganized. Severity: moderate. Likelihood: high over long time periods. Detection: periodic index audit (compare index against actual records). Mitigation: the 14-day indexing deadline (R-D20-09-01) and annual index audits.

  • Jargon accumulation. Over time, records become increasingly opaque to new readers because they rely on institutional jargon that was never defined. Severity: moderate. Likelihood: high. Detection: new-member onboarding test -- can a new member understand the records? Mitigation: the plain language requirement (R-D20-09-02) and a maintained glossary of institutional terms.

  • Cross-reference decay. Cross-references break when records are reorganized, renumbered, or moved. Severity: moderate (navigability degrades). Likelihood: moderate. Detection: periodic cross-reference validation. Mitigation: stable record IDs that do not change when records are reorganized.

  • Entry point staleness. The "Start Here" guide and Founder's Summary become outdated. New users encounter misleading or incomplete orientation. Severity: moderate. Likelihood: high without active maintenance. Detection: new-member feedback. Mitigation: R-D20-09-04 mandates regular review.

  • Platform lock-in. Accessibility features are implemented in a platform-specific way that cannot be migrated. When the platform changes, the index, cross-references, or search capabilities are lost. Severity: high. Likelihood: moderate. Detection: platform migration planning should include an accessibility migration assessment. Mitigation: R-D20-09-06 requires platform-independent accessibility infrastructure.

Recovery Procedures

  1. If the index is out of sync: Conduct a full index reconciliation. Compare the index against actual records. Add missing entries. Remove or flag entries that point to non-existent records. This is labor-intensive but essential. Prioritize by record significance: Tier 1 permanent records first, then Tier 2 active records.

  2. If jargon has made records opaque: Create a retrospective glossary based on the terms found in existing records. For the most important records (Tier 1 permanent records, foundational articles), add annotations that define terms in context. Do not modify the original records (per D20-008); instead, create companion annotations that are cross-referenced to the original.

  3. If cross-references are broken: Validate all cross-references. Fix broken links by updating the cross-reference to point to the correct record (this is a metadata update, not a content modification, and is permitted under D20-008). Where the target record cannot be found, note the broken reference and investigate whether the target was renamed, reorganized, or lost.

  4. If entry points are stale: Conduct an entry point review. Update the Founder's Summary, the "Start Here" guide, and the timeline. Test the updated entry points with a new user or a member who has not interacted with the memory system recently.

  5. If platform migration has damaged accessibility: Prioritize restoring search capabilities, then index integrity, then cross-references. Reconstruct what cannot be migrated. Document what was lost and what was reconstructed. This is an institutional crisis and should be treated accordingly in the decision log.

Evolution Path

  • Years 0-5: The memory system is small enough that accessibility is relatively easy. The index is manageable. Cross-references are few. Use this period to establish good habits: index everything, cross-reference everything, write plainly. These habits are the foundation for accessibility at scale.

  • Years 5-15: The memory system grows. The index becomes substantial. Search capabilities become more important as manual browsing becomes less feasible. Evaluate whether the search system meets the five access patterns. Conduct the first "new user test" to evaluate whether entry points are effective.

  • Years 15-30: Accessibility at scale is the central challenge. The memory system may contain thousands of records. Automated indexing and search become essential. The glossary of institutional terms requires active maintenance. Cross-reference validation may need to be partially automated.

  • Years 30-50: The memory system spans multiple generations of contributors. Language drift has occurred. Records from the founding era may use terms and concepts that are unfamiliar to current users. The retrospective annotation process becomes essential. Accessibility across generational boundaries is the ultimate test of this article's principles.

Founder Commentary

This section is reserved for dated entries by current and future operators.

2026-02-17 -- Founding Entry: This article exists because I have seen too many archives that are technically complete but practically useless. The records exist. They are intact. They are accurate. And nobody can find anything in them, because they were stored without indexing, without cross-references, and without any thought for the reader who would come later.

The plain language requirement (R-D20-09-02) will be the hardest rule to enforce. Technical people write in technical language. Institutional insiders use insider jargon. Both habits are natural and both make records opaque to future readers. Every time you write a memory record, imagine it being read by someone 30 years from now who has never used this system and knows nobody who has. Write for that person.

The "Start Here" guide is, in my opinion, the single most important accessibility feature. It is the first thing a new maintainer encounters. If it is good, they will find their way through the memory system. If it is bad, they will give up and rely on guesswork. Keep it good. Test it regularly. Update it whenever someone reports confusion.
