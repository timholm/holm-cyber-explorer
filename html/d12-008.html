D12-008 — Data Corruption and Integrity Failure Recovery

Document ID
D12-008
Domain
12 -- Disaster Recovery
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
D12-001, D12-004, D12-005, D12-006, D6-002, D6-003, SEC-003
Depended Upon By
D12-012 (cascade analysis for corruption propagation), D12-002 (drill Scenario 2), D12-013 (post-incident review of corruption events).

Purpose

This article defines how the institution detects and recovers from data corruption -- the silent, insidious failure class that D12-004 identifies as the most dangerous threat to long-term institutional integrity. Hardware failures are dramatic and immediately obvious. Data corruption is neither. A file corrupted by bit rot looks the same size, has the same name, sits in the same directory. It is only when the file is read -- perhaps months or years after the corruption occurred -- that the damage becomes apparent.

This article addresses silent data corruption (bit rot), filesystem damage, database and index inconsistency, configuration file corruption, and the special case of cryptographic key loss. For each corruption type, it defines detection methods, scope assessment procedures, recovery steps, and post-recovery verification.

The operating principle of this article is: assume corruption exists. The question is not whether data has been corrupted but whether the corruption has been detected. Every procedure in this article begins from the assumption that corruption may be more extensive than the initial detection suggests, and it includes steps to assess the full scope before beginning recovery.

Scope

In scope:

  • Detection and recovery for: silent bit rot, filesystem corruption, database/index inconsistency, configuration file corruption, and cryptographic key loss or corruption.
  • Scope assessment procedures: determining how far corruption has spread.
  • Recovery from backup: selecting the correct backup generation when recent backups may also be corrupted.
  • Prevention measures: integrity monitoring that catches corruption early.

Out of scope:

  • Hardware-caused data loss (D12-007 handles the hardware; this article handles the data consequences).
  • Accidental deletion by operator error (D6-002 backup restoration).
  • Primary data integrity monitoring design (D6-003 defines the monitoring; this article defines recovery when monitoring detects a problem).

Background

3.1 The Nature of Silent Corruption

Silent data corruption occurs when stored data changes without any error being reported by the storage system. The causes include: cosmic ray events flipping bits in memory or on disk, firmware bugs in storage controllers that write data incorrectly, electrical interference during write operations, and gradual physical degradation of storage media (magnetic domains weakening on HDDs, charge leakage in SSDs).

Studies by CERN, Google, and others have demonstrated that silent data corruption occurs at measurable rates even in enterprise-grade storage. Over decades of operation, the institution should expect it to occur. The only question is whether the corruption will be detected before it propagates into backups and becomes unrecoverable.

3.2 The Propagation Problem

Silent corruption becomes catastrophic through propagation. A file is corrupted on the primary storage. The next backup faithfully copies the corrupted file to Location A. The weekly backup copies it to Location B. The monthly backup copies it to Location C. Within one backup cycle, every copy of that file is corrupted. If the corruption is not detected before the retention window closes on the last clean backup, the clean version is permanently lost.

This is D12-004's Compound C scenario, and it is the primary justification for both the retention policy in D12-005 and the integrity monitoring in D6-003. This article defines what to do when propagation has occurred despite those defenses.

3.3 Filesystem vs. File-Level Corruption

Filesystem corruption and file-level corruption are different problems requiring different responses. Filesystem corruption (damaged inodes, corrupted directory entries, journal inconsistencies) affects the structure that organizes files, not necessarily the file contents. File-level corruption (bit rot, truncation) affects the data itself. A filesystem check (fsck) can repair structural damage but cannot detect or repair corrupted file contents. Checksum verification can detect file-level corruption but cannot repair it -- only backup restoration can do that.

System Model

4.1 Detection Methods

Scheduled integrity scans (proactive): D6-003 defines a scheduled integrity verification process that computes checksums of critical files and compares them against recorded baselines. Any mismatch triggers an alert. This is the primary detection mechanism for silent corruption.

Application-level detection (incidental): An application fails to parse a file it previously handled correctly. A document renders with garbled characters. An archive cannot be extracted. These incidental discoveries often catch corruption that escaped scheduled scans.

Filesystem-level detection: The operating system reports I/O errors, filesystem inconsistencies, or journal recovery events during boot. These indicate structural corruption that may or may not involve data corruption.

4.2 Scope Assessment Procedure

When corruption is detected, the first action is not recovery -- it is assessment. Premature recovery from backup can overwrite evidence needed to determine the corruption's extent and cause.

  1. Isolate the affected system. If corruption may be ongoing (e.g., a failing controller is actively corrupting writes), power down the system to prevent further damage.
  2. Identify the affected files. Run a full integrity scan (D6-003): cd /data && sha256sum -c /data/integrity/baseline.sha256 2>&1 | grep FAILED
  3. Determine the corruption timeline. When was the last known-good state? Check the most recent backup at each location: is the backup copy also corrupted? Work backward through backup generations until a clean copy is found.
  4. Assess propagation. Has the corruption spread to backups? Check Location A, Location B, and Location C in sequence. For each location, verify the checksum of the affected files against the known-good baseline.
  5. Identify the cause if possible. Is the corruption associated with a specific drive (SMART errors), a recent event (power failure, software change), or apparently random (possible bit rot)?
  6. Document the assessment. Record: affected files, corruption timeline, propagation status, probable cause, and the oldest known-good backup.

4.3 Recovery: File-Level Corruption

  1. Identify the recovery source. Select the most recent backup that contains a verified-clean copy of the affected files. This may be the most recent backup (if corruption was caught before propagation) or an older generation (if corruption propagated).
  2. Restore the affected files. Extract only the corrupted files from the backup, preserving all other current data: tar -xzf /path/to/backup.tar.gz -C /tmp/restore/ [path/to/affected/file] cp /tmp/restore/[path/to/affected/file] /data/[path/to/affected/file]
  3. Verify the restored files. Compute checksums and compare against the known-good baseline.
  4. Update the integrity baseline. If the restored files are verified clean, update the baseline checksums.
  5. Assess data loss. If the clean backup is older than the corruption, any changes made between the clean backup and the corruption discovery are lost. Document this loss per ETH-001 Principle 6.
  6. Create a new backup immediately to capture the corrected state.

4.4 Recovery: Filesystem Corruption

  1. Unmount the affected filesystem (or boot from installation media if the root filesystem is affected).
  2. Run filesystem check: fsck -y /dev/[partition]. Review the output carefully. Note any files that were repaired, moved to lost+found, or deleted.
  3. Mount the filesystem and assess. Check for files in lost+found. Verify that critical files are intact (run integrity check).
  4. Restore any lost files from backup.
  5. Investigate the cause. Filesystem corruption is often caused by hardware issues (failing drive, power loss during write). Check SMART data and system logs.

4.5 Recovery: Cryptographic Key Loss

Key loss is the most severe data integrity failure because it renders encrypted data permanently inaccessible unless the key can be recovered from backup.

  1. Do not panic. D12-002 Scenario 5 drills this exact situation. Follow the drilled procedure.
  2. Consult the key inventory (SEC-003 Section 4.1). Identify all backup copies of the lost key.
  3. Retrieve the backup key from the appropriate tier (Tier B from Zone 1, or Tier C from Zone 3 if Tier B is also unavailable).
  4. Decrypt the backup key media.
  5. Restore the lost key.
  6. Verify access to the encrypted data using the recovered key.
  7. Create new key backups per SEC-003 (since the backup media was opened during recovery).
  8. If no backup key exists or all backups are corrupted: the encrypted data is permanently lost. This is a catastrophic failure. Document it. Assess what can be reconstructed from non-encrypted sources. Update key backup procedures to prevent recurrence.

4.6 Prevention: Integrity Monitoring Schedule

Data TypeScan FrequencyMethod
Tier 1 (docs, logs, governance)WeeklyFull SHA-256 checksum verification
Tier 2 (configs, projects, automation)MonthlyFull SHA-256 checksum verification
Tier 3 (reference, archives)QuarterlySpot-check (10% random sampling) + full annual
System configuration filesWeeklyComparison against known-good baseline

Rules & Constraints

  • R-D12-08-01: Integrity monitoring must be conducted on the schedule defined in Section 4.6. Missed scans must be rescheduled within 7 days.
  • R-D12-08-02: When corruption is detected, scope assessment (Section 4.2) must be completed before recovery begins. Premature recovery risks masking the extent of the problem.
  • R-D12-08-03: Any corruption event that has propagated into backups triggers an immediate review of the retention policy (D12-005) and the integrity monitoring frequency (Section 4.6) to determine whether they are adequate.
  • R-D12-08-04: Data loss caused by corruption must be documented honestly per ETH-001 Principle 6, including an assessment of what was lost and whether it can be reconstructed.
  • R-D12-08-05: Cryptographic key loss or corruption is a Tier 3 governance event and must be recorded in the decision log regardless of whether the key was successfully recovered.

Failure Modes

  • Undetected propagation. Corruption escapes detection for longer than the backup retention window. All copies are corrupted. The clean version is permanently lost. Mitigation: frequent integrity scans (Section 4.6), long retention windows (D12-005 Section 4.3), and multiple independent checksum baselines.
  • False positive corruption detection. The integrity baseline itself is wrong (updated with a corrupted file's checksum). The monitoring system reports no corruption because the baseline matches the corrupted data. Mitigation: maintain multiple independent baselines. Compare file state against the backup copy's checksum, not just the stored baseline.
  • Overcorrection. The operator, alarmed by corruption, restores more data than necessary from backup, overwriting recent valid changes with older backup data. Mitigation: the scope assessment procedure (Section 4.2) identifies exactly which files are affected. Restore only the affected files.
  • Cause misidentification. The operator treats the corruption as random bit rot when it is actually caused by a failing storage controller that will continue corrupting data. The restored data is re-corrupted. Mitigation: the scope assessment includes cause investigation. If the cause is hardware-related, fix the hardware (D12-007) before restoring data.

Recovery Procedures

  1. If integrity monitoring has lapsed: Conduct an immediate full scan of all tiers. Compare against both the stored baseline and the most recent backup. Address any discrepancies discovered. Resume the monitoring schedule.
  2. If corruption has propagated to all backup locations: Check Location C's retention archive for older generations. If a clean generation exists within the retention window, restore from it. Accept the data loss for the period between that generation and the present. If no clean generation exists, the data is permanently corrupted or lost. Document the loss. Assess whether the data can be reconstructed from other sources.
  3. If the integrity baseline itself is corrupted: Rebuild the baseline from the most recent verified-clean backup. Scan all current data against the rebuilt baseline. Address any discrepancies.

Evolution Path

  • Years 0-5: Establish integrity baselines for all institutional data. Conduct the first full-corpus integrity scan. Discover and address any pre-existing corruption. Refine the monitoring schedule based on observed scan times and false positive rates.
  • Years 5-15: The integrity monitoring system is mature. Accumulating scan history reveals any patterns (certain files or directories more prone to corruption, certain media more reliable). Use this data to focus monitoring resources.
  • Years 15-30: Media transitions create a heightened risk period for corruption. Data copied between media types must be verified at both source and destination. The integrity monitoring system must adapt to new storage architectures.
  • Years 30-50+: The accumulated history of corruption events (or the reassuring absence of them) is the best guide to future monitoring strategy. New integrity verification technologies may be available; evaluate them through Domain 14 (Research).

Founder Commentary

This section is reserved for dated entries by current and future operators.

2026-02-17 -- Founding Entry: Data corruption is the failure mode that keeps me up at night. Not because it is the most likely -- hardware failure is more frequent -- but because it is the most treacherous. A fire is honest: it destroys everything and you know it. Bit rot is dishonest: it corrupts one file and lets you believe everything is fine for months, while your backups faithfully replicate the damage.

The integrity monitoring schedule is the article's most important contribution. Everything else -- the recovery procedures, the scope assessment, the filesystem repair -- is reactive. The monitoring is proactive. It is the early warning system that converts a catastrophic corruption event into a manageable one by catching it before it propagates.

I have set the monitoring frequency aggressively for Tier 1 data (weekly scans). This will consume time and compute resources. I accept the cost. The documentation corpus is the institution's irreplaceable core. Everything else can be rebuilt from documentation. Documentation cannot be rebuilt from nothing.
