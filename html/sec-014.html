SEC-014 — Security Documentation and Classification

Document ID
SEC-014
Domain
3 -- Security & Integrity
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
SEC-002 (Access Control), SEC-008 (Security Audit Framework)
Depended Upon By
None (internal process document)

Purpose

This article defines how the institution handles security-sensitive information within its own documentation. Not all information in the institution is equally sensitive. The philosophical principles in Domain 1 are public by design -- their strength comes from transparency, not secrecy. But cryptographic key identifiers, specific hardware configurations, backup locations, and vulnerability assessments are sensitive and must be handled with appropriate care.

The challenge in a single-operator institution is that the operator has access to everything. Classification does not restrict access in the traditional sense. Instead, it serves three purposes. First, it informs the operator about which information requires special handling -- what should not be written on paper left in public, what should not be discussed casually, what should not be included in documents that might leave the institutional perimeter. Second, it informs a future successor about which information carries elevated sensitivity, since the successor may not have the founding operator's intuitive understanding of what matters. Third, it determines how information is stored, backed up, and protected within the institution's zone architecture.

This article establishes the classification system, defines the handling requirements for each classification level, and describes how classification decisions are made and reviewed.

Scope

This article covers:

  • The classification levels used within the institution.
  • The criteria for assigning each classification level.
  • Handling requirements for each level: storage, access, backup, and disposal.
  • The process for classifying and reclassifying information.
  • The relationship between classification and the zone architecture (SEC-009).
  • Classification of the security documentation itself (a meta-question this article must address).

This article does not cover access control mechanisms (SEC-002), cryptographic protection of classified data (SEC-010), or the physical security of storage locations (SEC-006). Those articles provide the tools; this article defines the policy that determines when and how those tools are applied.

Background

3.1 The Paradox of Security Documentation

Security documentation exists in tension with itself. The documentation must be detailed enough to be useful -- a successor must be able to understand and maintain the institution's security systems using only the documentation. But the documentation must also be careful about what it reveals -- specific key identifiers, vulnerability assessments, and implementation details could be exploited if they fell into the wrong hands.

The institution resolves this tension through the Kerckhoffs principle (SEC-010): the security of the institution must not depend on the secrecy of its documentation. The algorithms are public. The procedures are public. The architecture is public. Only the keys are secret. This principle means that most security documentation can be classified at the lowest sensitivity level, because reading it does not compromise the institution. The small number of documents that contain actual secret material -- key inventories, specific vulnerability assessments, succession credentials -- require elevated classification.

3.2 Classification in a Single-Operator Context

In a conventional organization, information classification controls who can access what. In this institution, where a single operator has access to everything, classification serves a different function. It serves as a labeling system that tells the operator (and future successors) how to handle information appropriately. A document classified as "restricted" does not become inaccessible to the operator. It becomes a document that the operator knows must be stored in Zone A, must not be copied to Zone B without encryption, must not be included in any material that crosses the institutional perimeter, and must be destroyed securely when it is no longer needed.

Classification is therefore a set of handling instructions encoded as a label. The label triggers a set of behaviors. The behaviors protect the information. This indirect mechanism -- label triggers behavior, behavior provides protection -- works only if the operator has internalized the handling requirements for each label. This article defines those requirements.

3.3 Over-Classification as a Failure Mode

The temptation is to classify everything as highly sensitive. This is a common failure in classification systems, and it is counterproductive for two reasons. First, if everything is classified as sensitive, nothing is distinguishable from anything else, and the operator cannot prioritize protection efforts. The key inventory and the style guide receive the same handling, which means either the style guide is over-protected (wasteful) or the key inventory is under-protected (dangerous). Second, over-classification creates operational friction that leads to classification fatigue, where the operator begins to ignore classification labels because they are perceived as meaningless.

The classification system defined here is deliberately simple and deliberately conservative in what it elevates to higher sensitivity levels. Most institutional documentation is unclassified. A small number of documents contain sensitive operational details. An even smaller number contain secret material. This pyramid shape is intentional.

System Model

4.1 Classification Levels

The institution uses three classification levels:

Level 1: Open. Information that can be known by anyone without compromising the institution's security. This includes: all constitutional and philosophical documents (Domain 1), all governance procedures (Domain 2), all security framework documents (Domain 3, except as noted), all operational procedures that do not reveal specific implementation vulnerabilities, and all educational and training materials (Domain 9). The Kerckhoffs principle means that the vast majority of institutional documentation is Open. The institution's security does not depend on the secrecy of its architecture, algorithms, or procedures.

Level 2: Restricted. Information that would provide a meaningful advantage to an adversary or whose exposure would create operational risk. This includes: specific hardware configurations and their security properties (e.g., which USB ports are disabled and how), specific software versions and their known limitations, vulnerability assessments and penetration testing results, physical security details (specific lock types, sensor placements, alarm configurations), incident response logs that reveal specific weaknesses, and self-audit records that document procedural deviations. Restricted information must be stored in Zone A or Zone B (never Zone C), must be encrypted at rest, and must not cross the institutional perimeter.

Level 3: Secret. Information whose exposure would directly compromise institutional security. This includes: cryptographic key material and key identifiers, passphrases and authentication credentials, succession credentials and emergency access procedures, the specific locations of geographically distributed backups, and any information that would allow unauthorized access to institutional systems. Secret information must be stored only in Zone A, must be encrypted with the strongest available algorithms, must be backed up only to locations with equivalent physical security, and must be destroyed using secure deletion procedures when no longer needed.

4.2 Classification Criteria

The classification of a piece of information is determined by answering one question: what is the worst thing that could happen if this information were known by an adversary?

If the answer is "nothing significant" -- the adversary gains no actionable advantage -- the information is Open.

If the answer is "the adversary gains knowledge that would make attacking the institution easier, but does not directly enable an attack" -- the adversary learns where to look, but not how to get in -- the information is Restricted.

If the answer is "the adversary could directly compromise the institution's security" -- the adversary gains credentials, key material, or access paths that would permit unauthorized access -- the information is Secret.

When in doubt, classify at the higher level and reclassify downward after review. The cost of brief over-classification is minor inconvenience. The cost of under-classification is potential exposure.

Storage zones Any zone Zone A or B only Zone A only
Encryption at rest Per zone policy Required Required (strongest available)
May cross perimeter Yes (per procedures) No No

4.4 Classification of Security Documentation

This section addresses the meta-question: what is the classification of the Domain 3 security documents themselves?

Per the Kerckhoffs principle, the security framework documents -- including this one -- are classified as Open. They describe the institution's security architecture, threat model, and procedures. They do not contain key material, credentials, or specific vulnerability details. An adversary who reads these documents learns how the institution thinks about security, but gains no direct avenue of attack.

Specific annexes or companion documents that contain implementation details (hardware configurations, vulnerability scan results, key inventories) are classified at Level 2 or Level 3 as appropriate. These annexes are referenced from the framework documents but stored and handled according to their own classification.

The classification of each document is stated in its metadata header. If no classification is stated, the document is assumed to be Open. This default-open stance is deliberate: it reflects the Kerckhoffs principle and prevents the over-classification failure mode described in Section 3.3.

4.5 Reclassification

Information may be reclassified in either direction. Downward reclassification (from Secret to Restricted, or from Restricted to Open) occurs when the reason for the original classification no longer applies -- for example, when a cryptographic key has been rotated and the old key identifier is no longer sensitive. Upward reclassification occurs when new information reveals that a previously Open document contains details that are more sensitive than initially assessed.

Reclassification decisions are documented in the provenance record of the affected document. The reclassification record includes: the previous classification, the new classification, the date of reclassification, and the reasoning. This record ensures that the classification history of every document is traceable.

Rules & Constraints

  • R-SC-01: Every document in the institution must have an explicit classification. If no classification is assigned, the document defaults to Open. This default is by design, not by oversight.
  • R-SC-02: Classification decisions must be based on the worst-case adversary impact test described in Section 4.2. Classification must not be based on the document's perceived importance, prestige, or the effort invested in creating it. Importance and sensitivity are different properties.
  • R-SC-03: Secret information must never exist in Zone B or Zone C, even temporarily. If Secret material must be accessed during operations in Zone B, it must be accessed by reference (consulting Zone A) rather than by copy (moving data to Zone B).
  • R-SC-04: When a document contains information at multiple classification levels, the document as a whole must be classified at the highest level of any information it contains. Alternatively, the sensitive portions may be extracted into a separate, higher-classified annex, and the main document may remain at its lower classification with a reference to the annex.
  • R-SC-05: Classification reviews must be conducted annually for Restricted documents and semi-annually for Secret documents. The review must assess whether the classification remains appropriate or whether reclassification (in either direction) is warranted.
  • R-SC-06: Secure deletion of Restricted and Secret material must follow documented procedures that ensure the data is not recoverable. For magnetic media, this means overwriting. For solid-state media, this means using the media's secure erase command followed by physical destruction if the media is being decommissioned. Standard file deletion (which merely removes the directory entry) is insufficient for Restricted and Secret material.
  • R-SC-07: The classification system must be simple enough that the operator can apply it consistently without constant reference to this document. Three levels are sufficient. Adding additional levels requires a formal amendment to this article with justification that the existing three levels are insufficient.
  • R-SC-08: Classification does not restrict the operator's access. The operator has access to all classification levels. Classification determines handling requirements, not access permissions. This distinction is critical in a single-operator institution and must be understood by any successor.

Failure Modes

  • Over-classification. Too many documents are classified at Restricted or Secret, diluting the signal value of higher classifications and creating operational burden. Mitigation: the default-Open stance, the adversary impact test, and R-SC-07's simplicity requirement all push against over-classification. The annual classification review provides opportunities to downgrade documents that were initially over-classified.
  • Under-classification. Genuinely sensitive information is classified as Open and handled without appropriate safeguards. Mitigation: the adversary impact test is designed to catch this. When in doubt, classify higher. The semi-annual review of Secret documents and annual review of Restricted documents provide checkpoints.
  • Classification drift. Over time, the operator stops assigning classifications to new documents, and the system degrades. Mitigation: the default-Open rule means that unclassified documents are treated as Open, which is safe for the majority of documents. The audit framework (SEC-008) includes classification compliance as a checkpoint.
  • Handling non-compliance. Documents are correctly classified but handled incorrectly -- Secret material is copied to Zone B for convenience, Restricted documents are not securely deleted. Mitigation: the self-audit (SEC-013) includes handling compliance. The zone separation architecture (SEC-009) provides structural barriers against casual handling errors.
  • Succession confusion. A successor does not understand the classification system or its handling requirements, and mishandles sensitive material. Mitigation: this document must be among the first that a successor reads during the onboarding process defined in GOV-001. The classification labels on documents serve as persistent reminders of handling requirements.

Recovery Procedures

  1. If Secret material is found in an inappropriate zone: Immediately delete the copy from the inappropriate zone using secure deletion. Verify that no further copies exist outside Zone A. Assess whether the material was exposed to any context where it could have been compromised (e.g., if Zone C hardware had access to the material, treat it as a potential exposure). Document the incident. Review the circumstances that led to the misplacement and update procedures to prevent recurrence.
  2. If a document is discovered to be under-classified: Immediately reclassify it at the appropriate level. Apply the handling requirements for the new classification retroactively: move the document to the appropriate zone, ensure encryption, verify that no copies exist in unauthorized locations. If the document was handled as Open when it should have been Restricted or Secret, assess the exposure risk and treat as an incident if the exposure could have been harmful.
  3. If the classification system has fallen into disuse: Conduct a comprehensive classification review of all institutional documents. This is a labor-intensive process, but it is necessary to restore the system's integrity. Prioritize the review by starting with documents most likely to contain sensitive material (key management records, audit logs, incident reports) and working outward. Re-establish the classification discipline going forward.
  4. If secure deletion was not performed for decommissioned material: If the material is still physically accessible, perform secure deletion retroactively. If the material has left the operator's physical control (e.g., a decommissioned drive was discarded without secure deletion), assess the worst-case exposure. If Secret material may have been on the media, treat it as a potential key compromise and consider key rotation.

Evolution Path

  • Years 0-2: Establish the classification system and classify all existing documents. The initial classification pass will be the most labor-intensive. Most documents will be Open. A small set will be Restricted. A very small set will be Secret. Get the initial classification right; it is much harder to reclassify hundreds of documents later than to classify them correctly now.
  • Years 2-5: The classification system should be routine. New documents are classified as they are created. Classification reviews happen on schedule. The three-level system should prove adequate for the institution's needs. If it does not, assess why before adding complexity.
  • Years 5-15: As the institution's document corpus grows, the classification review may require more time. Consider developing a prioritized review schedule that focuses on documents most likely to need reclassification (e.g., documents referencing specific technology that has changed, incident reports whose findings are no longer sensitive).
  • Years 15-30: Much of the early Secret material (initial key identifiers, original hardware configurations) may have been reclassified downward as keys were rotated and hardware was replaced. The classification system's value lies increasingly in managing currently sensitive material rather than protecting historical secrets.
  • Years 30-50+: The classification system should be one of the most stable components of the institution's security architecture. Three levels. Clear criteria. Documented handling requirements. The system's simplicity is its strength: it is easy to teach to successors, easy to maintain, and resistant to the complexity creep that undermines more elaborate classification schemes.

Founder Commentary

2026-02-17 -- Founding Entry: I debated whether this article was necessary for a single-operator institution. If I have access to everything, what does classification accomplish? The answer came when I thought about succession. Someday, someone will inherit this institution and will need to know, immediately and unambiguously, which documents contain material that must be protected with extra care. They will not have time to read every document and assess its sensitivity from scratch. They need labels. They need a system. They need this article.

I also realized that classification serves me now, not just my successor later. When I am handling a document classified as Secret, the label reminds me to be careful. Not to leave it displayed on screen when I step away. Not to include its contents in notes that might leave the institutional perimeter. Not to let it exist in a zone where it does not belong. The label is a trigger for discipline, and discipline is a perishable commodity that needs regular reinforcement.

Three levels. No more. The moment you have more than three levels, you spend more time arguing about classification than protecting information. Open, Restricted, Secret. If you cannot decide between them, use the adversary impact test. If you still cannot decide, classify higher and move on. The system works by being simple enough to use every time, not by being sophisticated enough to handle every edge case.
