D18-013 — Import Logging & Audit Trail

Document ID
D18-013
Domain
18 -- Import & Quarantine
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
D18-002 (Validation Pipeline), D18-005 (Provenance Tracking), D20-002 (Decision Log Architecture)
Depended Upon By
D18-014 (Operator Training), D19-005 (Audit Procedure Manual), D20-005 (Timeline Construction)

Purpose

This article defines what is recorded about every import event, the format of those records, how long they are retained, and how they are made accessible for audit, analysis, and institutional memory. D18-001 established that the import pipeline must be auditable (R-D18-05). This article specifies what "auditable" means in practice: what data is captured, how it is structured, and how it can be queried by anyone with legitimate need.

The import log is the institutional memory of the institution's boundary. It records every interaction between the institution and the outside world: what came in, what was turned away, who made the decision, and why. Over decades, this log becomes an invaluable historical resource. It reveals patterns in content acquisition, changes in the institution's risk posture, the evolution of format landscapes, and the quality trajectories of recurring sources.

This article is addressed to import operators who create log entries, to QA auditors who review them, to institutional historians who mine them for patterns, and to future maintainers who must ensure the logging system survives technology transitions while maintaining a continuous, unbroken record.

Scope

This article covers:

  • The import log record structure: mandatory and optional fields.
  • Log entry lifecycle: creation, immutability, correction, and retention.
  • Log storage: where logs are stored, how they are protected, and how they are backed up.
  • Log accessibility: who can read the log, how queries are performed, and what indexes are maintained.
  • The relationship between the import log and Domain 20 institutional memory systems.
  • Audit procedures: how the log is reviewed for completeness, consistency, and accuracy.

This article does not cover:

  • The specific validation checks whose results are logged (see D18-002 and related articles).
  • Provenance data, which is captured by D18-005 and referenced from the import log.
  • The general institutional decision log (see D20-002); the import log feeds into that system but is maintained independently.
  • System-level technical logs (infrastructure monitoring, error logs); these are covered by infrastructure domains.

Background

Why Logging Is Not Optional

Every other article in Domain 18 references the import log. Provenance tracking records are stored there. Trust score assessments are recorded there. Rejection reasons are documented there. Integrity verification results are stored there. The import log is the connective tissue that makes all other import processes auditable and accountable. Without the log, the import pipeline operates in the dark: decisions are made but not recorded, patterns emerge but are not detected, and accountability is impossible.

Append-Only as a Design Principle

The import log is append-only. Entries cannot be modified or deleted after creation. If an entry contains an error, a correction entry is appended that references the original and explains the correction. The original entry remains in the log, unaltered. This principle is borrowed from accounting (journals are never erased) and from D20-008 (Memory Integrity & Anti-Revisionism). The log must be a faithful record of what actually happened, including mistakes.

The Dual Audience Problem

The import log serves two audiences with different needs. Import operators need structured, searchable data that supports daily operations: "Show me all pending items," "What is the trust score for this source?" QA auditors and institutional historians need analytical data that supports pattern detection: "What is the rejection rate trend over the past year?" "How has the format distribution of imports changed?" The log structure must serve both audiences without requiring either to wade through irrelevant data.

                      quarantine rebuild

Index Structure

The following indexes MUST be maintained for efficient querying:

  • By date: All import events in chronological order.
  • By source: All import events from a given source.
  • By operator: All import events processed by a given operator.
  • By outcome: All admissions, all rejections, all holds.
  • By format: All import events containing a given format.
  • By rejection code: All rejections of a given category.
  • By batch ID: All records associated with a given batch.
  • By node: All inter-node import events from a given node.

Rules & Constraints

  • R-D18-L01: Every Import Event Logged. Every import event -- whether it results in admission, rejection, or hold -- MUST produce a log record with all mandatory fields populated. An import event without a log record is a procedural violation equivalent to an undocumented rejection (prohibited by D18-001, R-D18-04).

  • R-D18-L02: Append-Only. The import log MUST be append-only. No log record may be modified or deleted after creation. Corrections are made by appending a correction record that references the original by Import ID and explains the correction.

  • R-D18-L03: Log Retention Is Permanent. Import log records MUST be retained for the lifetime of the institution. There is no sunset period for import logs. They are institutional memory and are subject to D20-008 (Memory Integrity & Anti-Revisionism).

  • R-D18-L04: Log Integrity Protection. The import log MUST be integrity-protected. At minimum, a daily hash of the log's cumulative state MUST be computed and stored separately from the log itself. Any unauthorized modification of the log will be detected by hash comparison.

  • R-D18-L05: Log Accessibility. The import log MUST be queryable by: import operators (for operational use), QA auditors (for compliance review per D19-005), and institutional historians (for pattern analysis per D20). Access controls SHOULD prevent unauthorized modification but SHOULD NOT prevent authorized reading.

  • R-D18-L06: Quarterly Summary Reports. A quarterly summary report MUST be generated from the import log. The report includes: total import events, admission rate, rejection rate by category, format distribution, average processing time, and notable incidents. This report is submitted to Domain 19 (QA) and Domain 20 (Institutional Memory).

  • R-D18-L07: Log Format Portability. The import log MUST be stored in a format that is human-readable without specialized software. Structured plain text (CSV with headers), JSON, or SQLite are acceptable. Proprietary database formats are not acceptable as the primary store. The log must survive platform transitions.

Failure Modes

  • Incomplete log entries. Operators create log records with mandatory fields left blank, reducing the log's value for audit and analysis. Likelihood: Moderate, especially under time pressure. Severity: Medium -- individual entries are degraded but the log structure is intact. Detection: Automated completeness checking at the time of entry. QA audit of log completeness.

  • Log corruption. The log file itself is corrupted by storage failure, software bug, or quarantine compromise. Likelihood: Low with proper backup and integrity protection. Severity: High if the primary and backup are both affected. Detection: Daily hash verification. Backup comparison.

  • Log becomes unwieldy. Over decades, the log grows to millions of records. Queries become slow. Operators avoid consulting the log because it is too large to search efficiently. Likelihood: High over a 50-year horizon. Severity: Medium. Detection: Query performance monitoring. Mitigation: Proper indexing, periodic index rebuilding, and archive segmentation (older records moved to archival storage with on-demand access).

  • Log format becomes unreadable. The institution transitions platforms and the log format is not properly migrated. Old log entries become inaccessible. Likelihood: Low if R-D18-L07 (format portability) is followed. Higher if proprietary formats are used. Severity: High -- institutional memory is lost. Detection: Log accessibility testing during every platform transition.

  • Logging becomes a bottleneck. The logging system slows down the import pipeline because log writes are synchronous and the storage is slow. Operators begin to resent logging as overhead. Likelihood: Low with modern storage. Higher with older or constrained hardware. Severity: Medium. Detection: Import processing time monitoring. Mitigation: Asynchronous log writes with guaranteed durability; log writes should never block the operator.

Recovery Procedures

  1. If log entries are found to be incomplete: QA auditors identify incomplete entries during periodic review. For recent entries (within 30 days), the operator is asked to complete the missing fields from memory or from quarantine station records. For older entries, the missing fields are annotated as "unavailable -- not recorded at time of import." A correction entry is appended. The root cause (time pressure, unclear requirements, tooling deficiency) is addressed to prevent recurrence.

  2. If the log is corrupted: Restore from the most recent verified backup. If the backup is also corrupted, reconstruct what is possible from the quarantine copy, from production system admission records, and from operator memory. Document the gap. Every log corruption event triggers a review of the backup and integrity protection procedures.

  3. If the log is too large to query efficiently: Implement archival segmentation: move log records older than a threshold (e.g., 5 years) to an archival tier with on-demand access. Rebuild indexes on the active tier. If the active tier is still too large, consider reducing the threshold. The archival tier must remain queryable, but slower query times are acceptable for historical data.

  4. If a log format migration fails: Restore the original-format log from backup. Do not destroy the original until the migration is verified. Test the migrated log by running a set of standard queries and comparing results against the original. If discrepancies are found, debug the migration process rather than patching individual records.

Evolution Path

  • Years 0-5: The log begins as a simple structured file (CSV or JSON lines). The record structure may be adjusted as operational experience reveals missing fields or unnecessary ones. Indexes are basic. Quarterly reports are compiled manually. The focus is on establishing the discipline of complete, timely logging.

  • Years 5-15: The log migrates to a more robust storage system (SQLite or equivalent) as volume grows. Automated completeness checking is implemented. Quarterly report generation is automated. The log begins to reveal patterns: seasonal import cycles, common rejection reasons, source quality trends. These patterns inform pipeline optimization.

  • Years 15-30: The log is a decade-spanning dataset. Longitudinal analysis becomes possible: how has the institution's import posture changed over time? Which sources have improved? Which have degraded? The log's analytical value may exceed its operational value. Consider creating derived datasets (anonymized, aggregated) for broader institutional analysis.

  • Years 30-50+: The log spans the institution's history. It is one of the most complete records of the institution's relationship with the outside world. Future historians mining the log will discover the institution's early import decisions, the sources it relied on, and the threats it faced. The log is no longer merely an operational tool -- it is a primary source for institutional history.

Commentary Section

This section is reserved for dated entries by current and future operators.

2026-02-17 -- Founding Entry: The decision to require permanent retention (R-D18-L03) will be questioned. "Do we really need to keep a log record for a single rejected text file from 2026 in perpetuity?" The answer is yes. The cost of storing a log record is negligible. The cost of not having it -- when a QA audit needs to reconstruct import history, when a provenance question arises decades later, when an institutional historian traces the origin of a piece of content -- is significant. Storage is cheap. Memory is priceless.

The choice of human-readable formats (R-D18-L07) over database formats was deliberate and will cause operational friction. A relational database would be faster to query, easier to index, and more familiar to technical operators. But databases require specific software to read. Over fifty years, that software may not exist. A CSV file can be read with any text editor, on any platform, in any era. The log must be the most resilient data store in the institution because it documents everything else.

One compromise: the institution may use a database as an operational layer on top of the canonical text-format log. The database is a convenience, not the source of truth. The source of truth is the text-format log. If the database is lost or corrupted, it can be rebuilt from the log. If the log is lost, the database cannot substitute because it may have been compacted, indexed, or otherwise transformed in ways that do not preserve the original record.
