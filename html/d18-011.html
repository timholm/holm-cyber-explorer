D18-011 — Inter-Node Import Specialization

Document ID
D18-011
Domain
18 -- Import & Quarantine
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
D18-002 (Validation Pipeline), D18-008 (Trust Scoring), D17-001 (Federation Philosophy), D17-002 (Node Identity)
Depended Upon By
D18-014 (Operator Training), D17-004 (Synchronization Protocols)

Purpose

This article defines how content arriving from federated nodes differs from fully external content in the import process, and what special procedures apply. D18-001 established that all external content passes through the import pipeline without exception (R-D18-01), including content from federated nodes. However, content from a known, authenticated federated node is categorically different from content arriving from an unknown external source, and the import pipeline should recognize this difference without compromising its protective function.

Federated nodes are part of the same institution. They operate under the same governance framework (D17-008), use the same canonical formats (or should), and maintain their own import pipelines with the same standards. Content that has been admitted to one node has already passed one instance of the institution's validation process. Re-validating it from scratch at the receiving node is redundant in the best case and insulting to the sending node in the worst.

Yet trust between nodes is not absolute. A node may have been compromised. Its validation pipeline may have drifted from institutional standards. The physical transport medium may have been tampered with during transit. The receiving node must verify what it can, trust what it has reason to trust, and maintain the ability to detect problems that the sending node missed.

This article defines the balance point: how to be a good federation partner without being naive.

Scope

This article covers:

  • How inter-node sync packages are received and processed through the import pipeline.
  • The trust relationship model between federated nodes and how it maps to the Trust Scoring Framework (D18-008).
  • Expedited validation: which validation checks may be reduced or skipped for trusted node content, and which may never be skipped.
  • Sync package integrity verification: verifying that the package was not altered in transit.
  • Node trust lifecycle: how trust is established, maintained, and revoked between nodes.
  • The handoff protocol between Domain 17 (which defines sync package format and transport) and Domain 18 (which processes the package contents).

This article does not cover:

  • The synchronization protocol itself (see D17-004, Synchronization Protocols).
  • The physical transport of sync packages (see D17-012, Physical Transport Protocols).
  • Conflict resolution when imported content conflicts with existing content (see D17-005, Consistency & Conflict Resolution).
  • Node identity and authentication (see D17-002, Node Identity & Lifecycle).
  • Batch import procedures, which apply when the sync package is large (see D18-010; this article defines what is specific to inter-node batches).

Background

The Federated Trust Spectrum

In the federation model defined by Domain 17, nodes are not equally trusted. A node that has been part of the federation since founding, has regular sync exchanges, and has never had a quality incident holds a different position than a node that joined recently, has synced only once, or has had quality problems in the past. The inter-node import process must reflect this spectrum rather than treating all federated content identically.

The Handoff Problem

When a sync package arrives at the institution's boundary, it crosses from Domain 17's jurisdiction (transport, format, addressing) to Domain 18's jurisdiction (validation, admission). The handoff point must be clearly defined: at what moment does a sync package stop being a "federation delivery" and become an "import event"? This article, jointly informed by both domains, defines that boundary.

Why Full Bypass Is Never Permitted

It would be efficient to allow content from highly-trusted nodes to bypass quarantine entirely. This article explicitly prohibits this. The reason is defense in depth: if a trusted node is compromised and its content bypasses quarantine, the compromised content enters production directly. By requiring all federated content to pass through quarantine -- even if the quarantine process is expedited -- the institution maintains a detection point for compromised-node scenarios. The cost is processing time. The benefit is a safety net that may never be needed but would be invaluable if it were.

Mandatory Checks (Never Skipped)

Regardless of node trust level, the following checks are always performed on inter-node imports:

  • Package authentication: The sync package must be from an authenticated, recognized node.
  • Integrity verification: Every item's checksum must match the manifest. This catches transit corruption and tampering.
  • Format identification: Every item must be in a recognized format per D18-007. Federated nodes should use canonical formats, so unrecognized formats from a node are a warning sign.
  • Duplicate detection: Check for content that already exists in the receiving node to avoid redundant admission.

Checks That May Be Expedited

For nodes at trust level NT-3 or NT-4, the following checks may be reduced based on sampling rather than applied to every item:

  • Content-level validation (internal consistency, structural completeness).
  • Deep format analysis (structural parsing beyond magic byte identification).
  • Trust scoring of individual items (the node's trust level serves as a proxy).

Checks That Are Never Expedited

Even for NT-4 founding nodes:

  • Malware scanning (per D18-003) is never skipped. A compromised node may distribute malware unknowingly.
  • Integrity verification (checksums) is never skipped. Transit damage is independent of node trust.
  • Format identification is never skipped. Format drift between nodes must be detected.

Rules & Constraints

  • R-D18-N01: All Federated Content Enters Quarantine. Content from federated nodes MUST pass through the quarantine zone. No federated content may be admitted directly to production, regardless of the sending node's trust level. The quarantine process may be expedited but not bypassed.

  • R-D18-N02: Node Trust Levels Are Maintained in the Trust Registry. Every federated node MUST have an entry in the trust registry with its current trust level (NT-1 through NT-4). The entry MUST record the basis for the trust level, the date of last assessment, and the history of trust level changes.

  • R-D18-N03: Trust Level Review on Every Sync. The receiving node's operator MUST review the sending node's trust level at the beginning of each sync package processing. If there is reason to question the trust level (e.g., the previous sync had quality issues), the operator may temporarily downgrade the trust level for this sync.

  • R-D18-N04: Incident Reporting to Sending Node. If content from a federated node fails validation, the receiving node MUST report the failure to the sending node in the next sync exchange. The report includes: item identification, failure type, and recommended corrective action. This is not punitive; it is collaborative quality assurance.

  • R-D18-N05: Trust Level Downgrade Triggers. A node's trust level MUST be reduced by one level if: (a) more than 5% of items in a sync package fail validation for content or format reasons, or (b) any item fails integrity verification (suggesting transit tampering), or (c) any item is found to contain malware. Trust restoration requires a review period and successful sync exchanges at the reduced trust level.

  • R-D18-N06: Handoff Point Definition. The handoff from Domain 17 to Domain 18 occurs when the sync package's external integrity has been verified (package-level checksum confirmed) and the sending node's identity has been authenticated. At that point, the package contents become import items subject to Domain 18 jurisdiction. Domain 17 retains jurisdiction over the package structure, transport metadata, and sync protocol compliance.

Failure Modes

  • Compromised node distributes malware. A trusted node's systems are compromised, and its sync packages contain malicious content. The high trust level causes reduced scrutiny, potentially allowing some malware to pass sampling. Likelihood: Low but catastrophic. Severity: Critical. Detection: Mandatory malware scanning (never expedited). Anomaly detection in content patterns from the node. Mitigation: Even at NT-4, malware scanning is mandatory.

  • Trust level stagnation. A node achieves NT-4 status and is never re-evaluated. Gradual quality drift at the node is not detected because the trust level prevents adequate scrutiny. Likelihood: Moderate over long time periods. Severity: High. Detection: Mandatory trust level review on every sync (R-D18-N03). Periodic full-validation audits of content from NT-4 nodes (random full-validation despite trust level).

  • Federation diplomatic incident. A node's trust level is downgraded, and the node perceives this as an insult or political act rather than a quality response. Federation cohesion is damaged. Likelihood: Moderate, especially in a distributed governance model. Severity: Medium (social/political rather than technical). Mitigation: Trust level changes must be communicated with full documentation of the triggering incident. The process must be perceived as objective and procedural, not arbitrary.

  • Transit tampering undetected. The sync package's integrity is verified at the package level, but individual items within the package were modified. This is possible if the package-level checksum was recomputed after tampering. Likelihood: Low if per-item checksums are verified. Severity: Critical if tampering introduces malicious content. Detection: Per-item integrity verification, not just package-level. Cross-verification against the sending node's manifest if available.

  • Sync package format divergence. Over years, nodes evolve their internal formats independently. A sync package arrives in a format or structure that the receiving node's import pipeline cannot efficiently process. Likelihood: Moderate over decades. Severity: Medium -- sync processing is delayed or fails. Detection: Format identification during import. Mitigation: Federation governance should enforce periodic format compatibility reviews (D17 responsibility).

Recovery Procedures

  1. If a compromised node has been distributing contaminated content: Immediately downgrade the node to NT-1 (untrusted). Identify all content received from the node since the suspected compromise date. Re-validate all identified content with full-scrutiny settings. Quarantine any items that fail re-validation. Notify the federation governance of the compromise. Assist the compromised node with remediation if requested. Do not restore trust level until the node has been independently audited.

  2. If a sync package fails authentication: Do not process the package. Isolate the physical media. Notify federation governance. Investigate: Was the media tampered with? Was it mislabeled? Is the sending node's authentication key compromised? Do not destroy the media immediately -- it may be needed for forensic investigation.

  3. If trust level downgrade causes diplomatic friction: Provide the sending node with complete documentation of the triggering incident(s). Offer to conduct a joint review of the findings. Propose a concrete remediation plan with defined criteria for trust restoration. Escalate to federation governance (D17-008) if bilateral resolution fails. Do not restore the trust level to resolve the diplomatic issue -- trust is earned through performance, not negotiation.

  4. If format divergence blocks sync processing: Hold the sync package in quarantine. Notify the sending node of the format compatibility issue. Work with Domain 17 governance to establish a format reconciliation process. If the divergence is minor, the receiving node may develop a conversion pathway (per D18-004). If the divergence is fundamental, it indicates a federation-level problem that must be addressed at the governance level.

Evolution Path

  • Years 0-5: The federation is small, possibly a single node. Inter-node import procedures are defined but may not be exercised immediately. When the first federated node joins, the procedures receive their first real test. Expect adjustments as theory meets practice. The first sync exchanges establish operational patterns.

  • Years 5-15: Multiple nodes are exchanging content regularly. Node trust levels are established and tested. The trust level system is calibrated against actual experience: Is NT-4 expedited validation adequate? Do the downgrade triggers fire at the right thresholds? The diplomatic aspects of trust level management become as important as the technical aspects.

  • Years 15-30: Nodes that were founding members may have undergone leadership transitions. The trust levels of these nodes reflect institutional history, not personal relationships. The inter-node import system must function without relying on interpersonal trust between specific operators at different nodes. Documentation and procedure replace personal familiarity.

  • Years 30-50+: The federation may include nodes that were established long after the founding. The trust level system must accommodate nodes with no shared history with the original institution. The inter-node import procedures, tested and refined over decades, provide a reliable framework for integrating new nodes while protecting existing institutional integrity.

Commentary Section

This section is reserved for dated entries by current and future operators.

2026-02-17 -- Founding Entry: The decision to never allow full quarantine bypass for federated content was contentious during drafting. The argument for bypass: trusted nodes are part of the institution, and re-validating their content is redundant overhead that discourages sync exchanges. The argument against: a compromised node that bypasses quarantine has a direct path into production systems. We chose defense over efficiency. If this proves to be an unacceptable operational burden, the right response is to make quarantine faster (better tooling, more automation), not to remove it.

The node trust level system (NT-1 through NT-4) is intentionally simpler than the content-level trust scoring framework (D18-008). Nodes are assessed holistically; their trust level reflects the overall quality of their import pipeline and governance. Applying the full four-dimensional trust model to nodes was considered and rejected as over-complex. A node is not a piece of content; it is a partner. The trust relationship is bilateral and evolves over years of interaction, not minutes of examination.

The handoff point definition (R-D18-N06) resolves a genuine ambiguity: when does a sync package stop being a "federation operation" and start being an "import operation"? The answer -- after package authentication and integrity verification -- was chosen because those are the checks that establish the package is what it claims to be. Everything after that is about whether the contents are acceptable, which is squarely an import concern.
