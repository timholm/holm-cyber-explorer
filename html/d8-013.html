D8-013 — Scheduled vs. Event-Driven Automation

Document ID
D8-013
Domain
8 -- Automation & Agents
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
ETH-001, CON-001, SEC-001, OPS-001, D8-001, D8-002, D8-005, D8-007, D8-008, D8-009, D8-011, D8-012
Depended Upon By
D10 operational scheduling procedures. All automations that must choose a triggering pattern. D8-009 scheduled task architecture.

Purpose

This article provides the decision framework for choosing between two fundamental patterns of automation execution: scheduled (time-based) and event-driven (condition-based). Every automation in the institution must be triggered by something. Either it runs because a clock says it is time to run, or it runs because a condition in the environment has changed. These two patterns have profoundly different characteristics in reliability, observability, resource consumption, failure modes, and comprehensibility -- and the wrong choice can create an automation that is fragile, wasteful, or invisible to its operator.

The choice between scheduled and event-driven execution is not a technical preference. It is an architectural decision with decades-long consequences. A scheduled backup that runs every night at 3 AM is predictable, testable, and easy to verify: either it ran or it did not. An event-driven backup that triggers whenever a file changes is responsive but harder to reason about: how many times did it run today? Did it miss a change? What happens if a thousand files change at once? The scheduled approach may miss changes that happen between runs. The event-driven approach may overwhelm the system during periods of high activity.

Neither pattern is inherently superior. Each has characteristics that make it appropriate for specific classes of tasks and inappropriate for others. This article provides the criteria for making that choice, the design patterns for implementing each approach correctly, and the hybrid patterns that combine elements of both when neither pure approach is adequate.

Scope

This article covers:

  • The characteristics, strengths, and weaknesses of scheduled automation.
  • The characteristics, strengths, and weaknesses of event-driven automation.
  • The decision matrix for choosing between them.
  • Design patterns for reliable scheduled automation.
  • Design patterns for reliable event-driven automation.
  • Hybrid patterns that combine scheduled and event-driven elements.
  • Failure mode comparison: how each pattern fails and what makes failures detectable.
  • Power and resource implications for an off-grid institution.

This article does not cover:

  • The specific scheduling infrastructure (cron, systemd timers, or equivalent) -- see D8-009 for scheduled task architecture.
  • Agent design principles beyond triggering pattern selection -- see D8-002.
  • Task delegation decisions (whether to automate at all) -- see D8-005 and D8-001.
  • Data integrity requirements for either pattern -- see D8-012.
  • Monitoring and observability requirements -- see D8-008 and D8-011.

Background

3.1 Two Philosophies of Time

Scheduled automation embodies a philosophy of regularity. The institution imposes a rhythm on its automated work: certain things happen at certain times, regardless of whether they are needed at that moment. This rhythm is inherently wasteful -- the backup runs even if nothing changed, the integrity check runs even if no data was modified -- but the waste buys predictability. The operator knows when each automation will run, can plan around it, and can verify completion by checking a schedule against a log.

Event-driven automation embodies a philosophy of responsiveness. Work happens when it is needed, not before and not on a schedule. This is inherently efficient -- no wasted cycles on unnecessary work -- but the efficiency costs predictability. The operator cannot predict when the automation will run, how often it will run, or how much system resource it will consume in a given period. The automation's behavior is determined by the environment, not by the institution's plan.

In a small, off-grid, air-gapped institution, predictability is usually more valuable than efficiency. Power budgets are constrained. System resources are finite. The operator's attention is limited. Knowing that the backup runs at 3 AM is operationally simpler than knowing that the backup runs whenever something changes -- because the former can be verified with a glance at a log, while the latter requires understanding what changed, when, and whether the backup captured all of it.

3.2 The Comprehensibility Dimension

D8-001 establishes comprehensibility as a foundational requirement for all automation. Scheduled automation is inherently more comprehensible than event-driven automation because its behavior can be predicted from its schedule alone. A future maintainer reading a cron table can understand when every scheduled automation runs without understanding what triggers it -- because the trigger is simply the clock. A future maintainer reading an event-driven automation's configuration must understand the event source, the event conditions, the filtering logic, and the potential for event storms or missed events. The comprehension burden is higher.

This does not mean event-driven automation should be avoided. It means that event-driven automation incurs a higher comprehension debt per D8-001 and must justify that debt with a correspondingly higher benefit.

3.3 Historical Context

The earliest automation systems were exclusively scheduled: cron jobs running at fixed times. Event-driven automation emerged as systems grew more complex and the latency between an event and the response to it became operationally important. Modern systems routinely combine both patterns. This institution's preference for simplicity (D8-001, D8-002) means that scheduled automation is the default and event-driven automation requires additional justification. But the institution acknowledges that some tasks are genuinely event-driven in nature and forcing them into a schedule creates its own problems.

System Model

4.1 Scheduled Automation: Characteristics

A scheduled automation runs at predetermined times according to a fixed schedule. The schedule may be simple (every day at 3 AM) or complex (every 15 minutes on weekdays, every hour on weekends, not during the power conservation window defined in D4-003). The trigger is always the clock.

Strengths:

  • Predictability. The operator knows exactly when each automation will run. Resource consumption can be planned. Conflicts between automations can be resolved by scheduling them at different times.
  • Verifiability. Checking whether a scheduled automation ran is trivial: compare the schedule to the log. A missing log entry at the expected time is an immediate, obvious signal.
  • Bounded resource consumption. A scheduled automation runs at most once per schedule interval. Resource consumption is bounded and predictable. Power budgets can account for it.
  • Simplicity of implementation. The triggering mechanism (cron, systemd timer, or equivalent) is well-understood, well-tested, and universally available. No custom event infrastructure is required.
  • Independence from environment. The automation runs regardless of what else is happening in the system. It does not depend on events being generated, event queues being available, or event processing infrastructure being healthy.

Weaknesses:

  • Latency. Events that occur between schedule intervals are not addressed until the next scheduled run. If the schedule is daily, the maximum latency is 24 hours. For time-sensitive tasks, this may be unacceptable.
  • Wasted work. The automation runs even if there is nothing to do. A nightly integrity check that finds no changes still consumes power, CPU time, and disk I/O.
  • Missed events. If multiple events occur between scheduled runs, the automation processes them all in a batch. Event ordering and individual event context may be lost.
  • Schedule sprawl. As automations accumulate, the schedule becomes crowded. Conflicts arise when multiple resource-intensive automations are scheduled concurrently. Managing the schedule becomes a task in itself.

4.2 Event-Driven Automation: Characteristics

An event-driven automation runs in response to a specific condition or change in the environment. The trigger is an event: a file is created, a threshold is exceeded, a service changes state, a user performs an action. The automation activates only when the event occurs.

Strengths:

  • Responsiveness. The automation reacts immediately (or near-immediately) to the triggering condition. Latency is minimal. Time-sensitive tasks are addressed promptly.
  • Efficiency. The automation runs only when needed. No wasted cycles on unnecessary execution. Power consumption correlates with actual work.
  • Event fidelity. Each event is processed individually, preserving event ordering and context. The automation can respond differently to different events.
  • Natural fit for reactive tasks. Some tasks are inherently event-driven: "when a new file arrives in the import directory, process it." Forcing such tasks into a schedule is artificial and introduces latency without benefit.

Weaknesses:

  • Unpredictable resource consumption. The automation runs as often as events occur. During periods of high activity, it may run continuously, consuming resources that were budgeted for other purposes. Resource consumption is unbounded unless explicitly constrained.
  • Event storms. A burst of events can trigger a burst of automation executions, overwhelming the system. Without rate limiting, event-driven automation can consume all available CPU, memory, or disk I/O.
  • Silent inactivity. If no events occur, the automation does not run. The operator cannot distinguish "the automation is idle because no events occurred" from "the automation is broken and not receiving events." Verifying correctness requires verifying both the automation and the event source.
  • Infrastructure dependency. Event-driven automation requires an event delivery mechanism: filesystem watchers, message queues, signal handlers, or equivalent. This infrastructure must itself be reliable, monitored, and maintained. It is an additional dependency that scheduled automation does not have.
  • Complexity of reasoning. The automation's behavior depends on external events. Predicting what the automation will do requires predicting what events will occur. Testing requires generating realistic event sequences. Debugging requires reconstructing the event history.

4.3 The Decision Matrix

The following matrix provides criteria for choosing between scheduled and event-driven automation. Apply each criterion to the task and sum the recommendations. If the result is mixed, prefer scheduled (as the simpler, more comprehensible default per D8-001).

Criterion Favors Scheduled Favors Event-Driven
Latency tolerance Task can tolerate delays of minutes to hours between event and response. Task requires response within seconds of the triggering condition.
Event frequency Events occur at predictable, moderate rates. Batching is acceptable. Events are sporadic and unpredictable. Each must be processed individually.
Resource budget Power and compute budget is tight. Predictable consumption is essential. Sufficient headroom exists for variable consumption.
Verification need Operator needs simple verification: did it run? Check the schedule. Operator can verify through event logs and output inspection.
Task nature Task is maintenance, housekeeping, or periodic verification. Task is reactive: processing input, responding to conditions.
Failure visibility A missed run is immediately visible (expected log entry absent). Acceptable if silent inactivity is addressed by independent monitoring.
Comprehensibility Future maintainer can understand behavior from the schedule alone. Future maintainer must understand event sources and triggering logic.
Event infrastructure No event delivery infrastructure exists or is warranted. Reliable event delivery infrastructure is already in place.

Default: When the analysis is inconclusive, use scheduled automation. The comprehensibility and verifiability advantages of scheduled automation align with the institution's restraint doctrine (D8-001). Event-driven automation should be chosen only when the latency, efficiency, or task-nature criteria strongly favor it.

4.4 Design Patterns for Scheduled Automation

Pattern 1: Fixed-interval execution. The automation runs at a fixed interval (hourly, daily, weekly). This is the simplest and most common pattern. Use for: backups, log rotation, integrity verification, report generation, disk space monitoring. Implementation: cron or systemd timer with a single schedule entry. The automation's specification (D8-007) documents the interval and the rationale for choosing it.

Pattern 2: Staggered execution. Multiple automations are scheduled at different times to avoid resource contention. The institution maintains a master schedule (part of D8-009) that shows all scheduled automations and their execution windows. New scheduled automations must be placed in the master schedule without conflicting with existing ones. Use for: any environment with multiple scheduled automations, which is the normal case.

Pattern 3: Conditional execution within a schedule. The automation runs on schedule but checks a condition before performing work. If the condition is not met, the automation logs that no work was required and exits. This combines the verifiability of scheduled automation (the log confirms it ran) with the efficiency of event-driven automation (it does not perform unnecessary work). Use for: tasks where the condition changes infrequently but must be checked regularly. Example: a scheduled check for files in an import directory; if no files are present, log "no imports pending" and exit.

Pattern 4: Catch-up execution. If a scheduled automation misses its window (because the system was powered down, or the automation was disabled), it runs as soon as possible after the missed window. The automation detects missed runs by comparing the current time against the last recorded execution time. If the gap exceeds one schedule interval, it executes immediately. Use for: critical tasks where a missed run must be recovered, such as backups or integrity checks. Implementation: the automation checks its own last-execution timestamp (from its metrics file per D8-011) at startup.

4.5 Design Patterns for Event-Driven Automation

Pattern 1: File-system watcher. The automation monitors a directory for new, modified, or deleted files. When a change is detected, the automation processes the affected file. Implementation: inotifywait (Linux), fswatch, or polling at sub-minute intervals. Use for: import processing, real-time log analysis, configuration change detection. Caution: filesystem watchers have platform-specific limitations. Some do not detect changes on network-mounted filesystems. All consume system resources proportional to the number of watched files and directories. Document these limitations in the specification.

Pattern 2: Threshold watcher. The automation monitors a metric (disk usage, CPU temperature, error rate) and triggers when the metric exceeds a defined threshold. Implementation: a lightweight polling loop that reads the metric at a fixed interval and compares it to the threshold. Use for: resource alerts, health checks, capacity warnings. This is technically a scheduled poll with event-driven response -- a common and effective hybrid.

Pattern 3: Signal-based trigger. The automation listens for an explicit signal from another automation or from the operator. Implementation: Unix signals, named pipes, flag files (a file whose presence signals the automation to act). Use for: inter-automation coordination, operator-initiated automation, post-deployment triggers. Caution: signal mechanisms must be documented in the automation's specification and in the triggering automation's specification, creating a documented dependency chain.

Pattern 4: Rate-limited event processing. The automation receives events but processes them at a controlled rate, queuing excess events for later processing. This prevents event storms from overwhelming the system. Implementation: an event queue (which may be as simple as a directory of pending files) with a processing loop that handles one event at a time with a configurable delay between events. Use for: any event-driven automation where the event rate may exceed the system's processing capacity.

4.6 Hybrid Patterns

Hybrid 1: Scheduled sweep with event-driven fast path. The automation runs on a schedule to process accumulated work, but also responds to events for urgent items. The schedule ensures that nothing is missed; the event trigger ensures that urgent items are processed promptly. Example: an import processor that checks the import directory every hour (scheduled) but also responds to a flag file that signals an urgent import (event-driven). The scheduled run processes everything the event-driven path missed.

Hybrid 2: Event-driven with scheduled verification. The automation runs in response to events, but a separate scheduled automation verifies that the event-driven automation is functioning correctly. The verifier checks that the event source is active, that events are being generated, and that the event-driven automation is processing them. This addresses the silent inactivity failure mode. Example: a filesystem watcher processes imports as they arrive, while a nightly scheduled check verifies that the watcher is running and that no unprocessed files are languishing in the import directory.

Hybrid 3: Event accumulation with scheduled processing. Events are detected and logged in real time, but processing occurs on a schedule. The event detection mechanism writes events to a queue; the scheduled processor reads the queue and handles accumulated events in batch. This combines the responsiveness of event detection (nothing is missed) with the predictability of scheduled processing (resource consumption is bounded and predictable). Example: filesystem changes are logged in real time to a change journal; a nightly scheduled automation reads the journal and performs integrity verification on all changed files.

Rules & Constraints

  • R-D8-13-01: Every automation's specification (D8-007) must declare its triggering pattern: scheduled, event-driven, or hybrid. The declaration must include the rationale for the choice, referencing the decision matrix criteria in Section 4.3.
  • R-D8-13-02: Scheduled automation is the institutional default. Event-driven automation requires documented justification demonstrating that the task's latency requirements, event characteristics, or inherent nature make scheduled execution inadequate.
  • R-D8-13-03: All scheduled automations must be registered in the institution's master schedule (D8-009). The master schedule must be reviewed quarterly to identify conflicts, gaps, and scheduling inefficiencies.
  • R-D8-13-04: All event-driven automations must implement rate limiting (Pattern 4 in Section 4.5, or equivalent). The maximum event processing rate must be specified in the automation's specification and must be within the system's resource capacity under worst-case conditions.
  • R-D8-13-05: All event-driven automations must have a corresponding scheduled verification mechanism (Hybrid 2 in Section 4.6, or equivalent) that confirms the event source is active, events are being delivered, and the automation is processing them. Silent inactivity must be detectable.
  • R-D8-13-06: Scheduled automations must not be scheduled during the power conservation window defined in D4-003, except for critical automations (Tier 1 data protection, security monitoring) that have explicit power budget allocations.
  • R-D8-13-07: Event-driven automations must log every received event, whether processed or deferred, per D8-011. The log must distinguish between events processed immediately, events queued for later processing, and events discarded due to rate limiting.
  • R-D8-13-08: Scheduled automations must implement the catch-up pattern (Pattern 4 in Section 4.4) for all tasks classified as critical in the automation's specification. A missed scheduled run of a critical automation must be recovered at the earliest opportunity.
  • R-D8-13-09: The choice of triggering pattern must be re-evaluated during the annual automation re-justification (D8-001, R-D8-07). If the task's characteristics have changed (event frequency has increased or decreased, latency requirements have changed, resource budget has changed), the triggering pattern must be reconsidered.
  • R-D8-13-10: Hybrid automations must document both their scheduled and event-driven components separately in the specification, including the interaction between them and the failure modes that arise from the combination.

Failure Modes

  • Event storm overwhelming event-driven automation. A burst of events (thousands of file changes during a large import, a cascade of error conditions) triggers the automation faster than it can process. The system's CPU, memory, or disk I/O is exhausted. Other automations and manual operations are starved of resources. Mitigation: R-D8-13-04 mandates rate limiting. The automation queues excess events and processes them at a sustainable rate. The monitoring system (D8-008) tracks resource consumption and alerts when thresholds are approached.

  • Silent inactivity of event-driven automation. The event source fails (filesystem watcher crashes, signal handler is overwritten, flag file mechanism breaks). The automation sits idle, receiving no events, processing no work. No one notices because the automation does not log inactivity -- it logs nothing. Work accumulates unprocessed. Mitigation: R-D8-13-05 mandates a scheduled verification mechanism that independently checks whether the event-driven automation is functioning. The heartbeat mechanism (D8-008) detects inactivity.

  • Schedule drift. A scheduled automation's execution time gradually shifts due to accumulated execution time. An automation scheduled at 3:00 AM takes 5 minutes to run. The next night it takes 7 minutes. Over weeks, the completion time drifts, eventually overlapping with the next scheduled automation's window. Mitigation: the master schedule (R-D8-13-03) allocates execution windows, not points in time. Each automation has a start time and a maximum execution duration. Overlap detection is part of the quarterly schedule review.

  • Missed scheduled run without catch-up. The system is powered down during a scheduled automation's window. When power returns, the automation does not run because its schedule was in the past. The automation waits for the next scheduled window, leaving a gap. For critical tasks (backups, integrity checks), this gap is unacceptable. Mitigation: R-D8-13-08 mandates catch-up execution for critical automations. The automation checks its last execution time at startup and runs immediately if a window was missed.

  • Event ordering violation. Events arrive out of order (due to filesystem timing granularity, concurrent event sources, or event queue reordering). The automation processes them in the wrong sequence, producing incorrect results. Mitigation: event-driven automations that are sensitive to ordering must include sequence validation. Events are timestamped at creation, and the automation verifies ordering before processing. Out-of-order events are logged and either reordered or flagged for human review.

  • Hybrid pattern incoherence. In a hybrid automation, the scheduled and event-driven components produce conflicting results. The event-driven path processes a file; the scheduled sweep processes the same file again because it does not know the event-driven path already handled it. Mitigation: R-D8-13-10 requires documenting the interaction between components. Hybrid automations must maintain a shared state record (processed-items log) that both components consult before processing.

  • Triggering pattern mismatch. A task is implemented with the wrong pattern. A naturally event-driven task is forced into a daily schedule, creating unacceptable latency. Or a naturally periodic task is implemented as event-driven, creating unnecessary complexity and fragility. Mitigation: R-D8-13-09 requires annual re-evaluation of the triggering pattern. The decision matrix (Section 4.3) provides objective criteria for reassessment.

Recovery Procedures

  1. If an event storm is detected: Check whether the rate limiter is functioning. If it is, the storm is being managed -- verify that queued events are processing correctly and that the queue is not growing without bound. If the rate limiter has failed, halt the event-driven automation. Process accumulated events manually or through a controlled batch run. Investigate the event source to determine whether the event rate is genuinely high or whether the event detection mechanism is generating spurious events. Adjust the rate limit or the event filtering logic. Restart the automation with the corrected configuration.

  2. If silent inactivity is detected by the verification mechanism: Determine whether the event source has failed, the event delivery mechanism has failed, or the automation itself has failed. Restart the failed component. Check for unprocessed events that accumulated during the inactivity period. Process them manually or through a catch-up run. Review the verification mechanism's check frequency -- if the inactivity persisted for longer than expected, the verification interval may need to be shorter.

  3. If a missed scheduled run is discovered: For critical automations: execute the missed run immediately per the catch-up pattern. Verify that the output is consistent with what would have been produced at the scheduled time. For non-critical automations: document the miss, verify that no data was left in an inconsistent state, and allow the next scheduled run to proceed normally. Investigate why the run was missed: system downtime, scheduler failure, or automation error.

  4. If schedule conflicts are causing resource contention: Review the master schedule. Identify overlapping execution windows. Stagger automations to eliminate overlap. If the schedule is too crowded to avoid overlap, prioritize by criticality: critical automations retain their windows, lower-priority automations are rescheduled. If no acceptable schedule exists, consider reducing automation frequency, optimizing execution duration, or deferring lower-priority automations to off-peak periods.

  5. If a hybrid pattern produces inconsistent results: Halt both components. Determine which component's output is authoritative. Reconcile the data using the authoritative source. Fix the shared state mechanism that was supposed to prevent duplicate processing. Test the fix by simulating the condition that caused the inconsistency. Document the incident and update the specification to address the interaction failure.

Evolution Path

  • Years 0-5: The institution's automation inventory is small. Nearly all automation should be scheduled. The master schedule is easy to manage. Event-driven automation should be limited to cases with a clear, strong justification: perhaps real-time import processing or critical threshold monitoring. The priority in this period is establishing the discipline of documenting triggering patterns and maintaining the master schedule, even when the schedule is simple.

  • Years 5-15: As the automation inventory grows, the master schedule becomes more crowded. Schedule conflicts require attention. Some tasks that were originally scheduled may benefit from event-driven triggers as the institution's operational patterns become clearer. The decision matrix in Section 4.3 should be applied rigorously during the annual re-justification. Hybrid patterns may emerge for tasks that need both responsiveness and reliability.

  • Years 15-30: The master schedule is a historical document: it shows how the institution's automated work has evolved over time. Schedule changes should be documented with rationale. Event-driven automations should have mature, well-tested event infrastructure. The verification mechanisms (R-D8-13-05) should have caught and prevented multiple instances of silent inactivity by this point, validating their necessity. Hardware changes may require revisiting power conservation windows and schedule timing.

  • Years 30-50+: A successor inheriting the automation will find in the master schedule a complete picture of the institution's automated rhythm. The schedule is a map of what the institution does automatically and when. Event-driven automations, being less immediately visible, require their verification mechanisms and their specifications to be especially well-documented. The successor should be able to reconstruct the institution's automated behavior from the master schedule and the event-driven automation specifications without reading source code.

  • Signpost for revision: If the institution finds itself with more event-driven automations than scheduled ones, revisit whether the institutional default (R-D8-13-02) is still appropriate. The default exists because scheduled automation is simpler and more comprehensible, but if the institution's operational reality is predominantly event-driven, the default may be creating artificial complexity by forcing event-driven tasks into schedules. Conversely, if event-driven automations are causing frequent incidents (storms, silent inactivity, ordering violations), the institution may need to convert some back to scheduled patterns with accepted latency trade-offs.

Commentary Section

This section is reserved for dated entries by current and future operators.

2026-02-17 -- Founding Entry: The most practical advice in this article is: when in doubt, use cron. A cron job that runs every five minutes and checks whether there is work to do is simpler, more comprehensible, and more debuggable than a filesystem watcher that triggers on inotify events. The cron job's maximum latency is five minutes. For most tasks in this institution, five minutes of latency is not a meaningful cost, and the gain in simplicity is substantial.

I am particularly concerned about the silent inactivity failure mode (Section 6). In my experience, event-driven systems fail most dangerously when their event sources fail -- because the system's behavior when receiving no events is indistinguishable from its behavior when functioning correctly with no events to process. The verification requirement (R-D8-13-05) addresses this, but only if the verification mechanism itself is maintained and monitored. A verification mechanism that no one checks is not a verification mechanism.

On the subject of hybrid patterns: Hybrid 3 (event accumulation with scheduled processing) is, in my judgment, the most robust pattern for tasks that seem to need event-driven responsiveness but where processing latency is tolerable. It captures events in real time so nothing is lost, but processes them on a predictable schedule so resource consumption is bounded. It is the best of both worlds at the cost of a small increase in complexity. I expect it to become the standard pattern for most tasks that outgrow pure scheduled execution.

A note on power conservation (R-D8-13-06): in an off-grid institution, the power conservation window is not optional. Scheduling automations during that window is not merely inefficient -- it may deplete batteries, reduce system availability during critical hours, or force the institution to choose between automation and essential services. Respect the power budget. The automation can wait.

Future maintainers: the master schedule is one of the most operationally important documents in the institution. Keep it current. Review it quarterly. When you add a new scheduled automation, check the master schedule first. When you decommission an automation, remove it from the master schedule. An accurate master schedule is a window into the institution's automated heartbeat. An inaccurate one is worse than none, because it creates false confidence.
