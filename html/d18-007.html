D18-007 — Format Recognition & Inventory

Document ID
D18-007
Domain
18 -- Import & Quarantine
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
D18-002 (Validation Pipeline), D18-001 (Import Philosophy)
Depended Upon By
D18-004 (Format Conversion), D18-008 (Trust Scoring), D18-010 (Batch Import)

Purpose

This article defines how the institution identifies the format of incoming content during the import process and how it maintains a registry of known formats. Format recognition is the gateway to every subsequent validation step: you cannot validate what you cannot identify. You cannot convert what you do not understand. You cannot assess the risk of a format you have never seen before.

In an air-gapped institution that must endure for decades, format recognition is not a static capability. Formats evolve. New formats emerge. Old formats become obscure. Content may arrive on media that was last written twenty years ago in a format that has since been forgotten by the wider world. The institution must maintain its own knowledge of formats because it cannot query the internet for format specifications when an unknown file appears on a quarantine station.

This article is addressed to import operators who must identify incoming content, to the maintainers of the format registry, and to future operators who will encounter formats that do not exist today. The format recognition system must be self-contained, extensible, and robust against the passage of time.

Scope

This article covers:

  • The methods by which incoming file formats are identified (magic bytes, file extensions, header analysis, structural analysis).
  • The institution's Format Registry: its structure, contents, and maintenance procedures.
  • The classification of formats by trust level (open vs. proprietary, documented vs. undocumented, static vs. executable).
  • Procedures for handling unrecognized formats.
  • The relationship between format recognition and the validation pipeline.
  • Format obsolescence tracking and long-term format viability assessment.

This article does not cover:

  • Conversion of identified formats to canonical formats (see D18-004, Format Conversion Standards).
  • Malware detection within recognized formats (see D18-003, Dangerous Content Handling).
  • Trust scoring based on format identification (see D18-008; this article provides the format data that trust scoring consumes).
  • Canonical format definitions for long-term preservation (see Content Domain specifications).

Background

Why Format Recognition Is Hard

File extensions are unreliable. A file named report.pdf may not be a PDF. A file named data.txt may contain binary data. A file with no extension at all may be perfectly valid content in a well-known format. Format recognition must go deeper than the filename.

Magic bytes -- the first few bytes of a file that identify its format -- are more reliable but not universal. Many formats share magic bytes. Some formats have no distinctive magic bytes. Container formats (ZIP, TAR) may contain files in many different formats, requiring recursive identification.

Structural analysis -- examining the internal structure of a file to determine if it conforms to a known format specification -- is the most reliable method but the most expensive. It requires the institution to possess the format specification and the tools to parse it.

In practice, format recognition uses all three methods in combination, starting with the cheapest (filename, magic bytes) and escalating to the most expensive (structural analysis) only when cheaper methods are inconclusive.

The Format Proliferation Problem

The number of file formats in active use worldwide is enormous and growing. The institution cannot know all of them. It does not need to. It needs to know the formats it is likely to encounter, the formats it has encountered in the past, and the formats that are important for its mission. The Format Registry is a curated knowledge base, not an exhaustive encyclopedia. Its completeness is judged by operational utility, not by theoretical coverage.

The Long-Horizon Format Problem

Over fifty years, formats will be born, mature, decline, and die. Content imported in 2026 in a then-common format may be unreadable by 2056 if the format has been abandoned and the institution has not preserved the ability to interpret it. Format recognition must therefore include not just current identification capability but also historical preservation of format knowledge. The Format Registry is itself an archival resource.

Format Registry Structure

The Format Registry is a structured data store with one entry per recognized format. Each entry contains:

FieldDescription
Format IDUnique internal identifier (e.g., FMT-0042)
Common NameHuman-readable name (e.g., "Portable Document Format")
File ExtensionsKnown file extensions associated with this format (e.g., .pdf, .PDF)
Magic BytesByte sequences that identify this format (e.g., %PDF- at offset 0)
MIME TypeStandard MIME type if one exists (e.g., application/pdf)
Format ClassClassification: TEXT, IMAGE, AUDIO, VIDEO, DOCUMENT, ARCHIVE, EXECUTABLE, DATABASE, OTHER
OpennessOPEN (published specification), PARTIALLY_OPEN, PROPRIETARY, UNKNOWN
ExecutabilitySTATIC (cannot execute code), CAPABLE (may contain executable elements), EXECUTABLE (is itself executable)
Specification ReferencePointer to locally archived format specification, if available
Conversion TargetPointer to D18-004 conversion pathway for this format, if one exists
Trust TierDefault trust tier for this format: HIGH (open, static), MEDIUM (open but capable), LOW (proprietary or executable), UNASSESSED
Viability StatusCURRENT, DECLINING, OBSOLETE, HISTORICAL_ONLY
First EncounteredDate this format was first encountered during import
Last EncounteredDate this format was most recently encountered during import
NotesOperator notes, known issues, special handling requirements

Format Classification Tiers

Formats are classified into trust tiers that influence how they are treated in the validation pipeline:

Tier 1 -- High Trust: Open-specification, non-executable, well-understood formats. Examples: plain text (UTF-8), PNG, JPEG, PDF/A, CSV, HTML (static), SVG. These formats receive the lightest validation burden.

Tier 2 -- Medium Trust: Open or partially open formats that may contain active elements or embedded content. Examples: standard PDF (may contain JavaScript), EPUB, DOCX/ODT (may contain macros), HTML with embedded scripts. These formats require additional inspection for active content.

Tier 3 -- Low Trust: Proprietary formats, formats with poor documentation, or formats known to be vectors for malicious content. Examples: proprietary database formats, compiled binaries, formats requiring specific vendor software to read. These formats receive the most rigorous validation and may require conversion before admission.

Tier 4 -- Unrecognized: Formats that do not match any entry in the Format Registry. These trigger the Unknown Format Procedure and cannot proceed through the standard validation pipeline until identified.

Rules & Constraints

  • R-D18-F01: Multi-Method Identification. Format identification MUST NOT rely solely on file extension. At minimum, magic byte analysis MUST be performed. For Tier 2 and Tier 3 formats, structural analysis SHOULD be performed to confirm the format identification.

  • R-D18-F02: Registry Is Authoritative. The Format Registry is the institution's authoritative source of format knowledge. If a format is not in the registry, it is unknown to the institution regardless of how well-known it may be in the outside world. The registry must be maintained as a living document.

  • R-D18-F03: Unknown Formats Are Not Admitted. Content in an unrecognized format MUST NOT be admitted to production systems until the format has been identified, assessed, and (if appropriate) added to the Format Registry. Content may be held in quarantine while format research is conducted.

  • R-D18-F04: Executable Formats Require Enhanced Scrutiny. Any format classified as CAPABLE or EXECUTABLE MUST undergo additional validation checks as defined in D18-003 (Dangerous Content Handling) before admission. Static-format classification does not guarantee safety, but executable classification guarantees additional scrutiny.

  • R-D18-F05: Registry Review Cadence. The Format Registry MUST be reviewed at least every six months. Review includes: updating viability status for all formats, adding any formats newly encountered, reviewing trust tier assignments for accuracy, and archiving format specifications for any format whose viability status has changed to DECLINING or OBSOLETE.

  • R-D18-F06: Format Specifications Must Be Locally Archived. For every format in the registry with viability status CURRENT or DECLINING, the institution SHOULD possess a locally archived copy of the format specification. For Tier 1 formats, this is REQUIRED. The institution cannot rely on external access to format documentation.

  • R-D18-F07: Container Format Recursion. When a container format (ZIP, TAR, ISO, etc.) is identified, format recognition MUST be performed recursively on the contents. A ZIP file containing only text files is different from a ZIP file containing executables, and the validation pipeline must know the difference.

Failure Modes

  • Misidentification. A file is identified as one format but is actually another -- potentially a more dangerous one. For example, a file that appears to be a static image but contains executable payload disguised behind valid image headers. Likelihood: Low for common formats, higher for obscure or maliciously crafted files. Severity: High if the misidentification causes the file to bypass security checks. Detection: Structural analysis as a secondary check. Anomaly detection (file size inconsistent with claimed format, internal structure inconsistent with format specification).

  • Registry staleness. The Format Registry falls behind the formats actually being encountered. Operators encounter unknown formats frequently, causing processing delays and frustration. Likelihood: Moderate over years, especially if registry maintenance is neglected. Severity: Medium. Import throughput degrades but safety is maintained (unknown formats are held, not admitted). Detection: Track the frequency of Unknown Format Procedure activations. Rising frequency indicates registry staleness.

  • Format specification loss. The locally archived specification for a format is lost or corrupted, and the format is still encountered in imports. The institution can identify the format (magic bytes match) but can no longer verify structural validity or perform informed conversion. Likelihood: Low if specifications are stored with redundancy. Higher over decades as storage media degrades. Severity: Medium. Detection: Periodic integrity checks on archived specifications.

  • Over-classification. The trust tier assignments become overly conservative, classifying too many formats as Low Trust and creating excessive validation burden. Import throughput suffers. Likelihood: Moderate, especially after a security incident prompts a reactionary tightening. Severity: Medium. The institution becomes too cautious, echoing the "immune overreaction" failure mode from D18-001. Detection: Monitor the distribution of trust tier assignments over time. A healthy registry has content in all tiers.

  • Polyglot files. A file that is simultaneously valid in multiple formats (e.g., a file that is both a valid JPEG and a valid ZIP). Such files may pass format recognition as a benign format while containing content interpretable as a different, more dangerous format. Likelihood: Low but non-zero; polyglot files are a known attack vector. Severity: High. Detection: Check for multiple valid format identifications. Flag any file that positively identifies as more than one format for enhanced scrutiny.

Recovery Procedures

  1. If a format misidentification led to admission of dangerous content: Identify all content in production that was admitted under the same format identification. Re-examine using corrected identification methods. Update the Format Registry to prevent recurrence -- add structural analysis rules, flag the problematic magic byte pattern, or reclassify the format to a lower trust tier. Document the incident in the registry notes and in Domain 20.

  2. If the Format Registry has become stale: Conduct an emergency registry review. Analyze the quarantine log for all Unknown Format Procedure activations in the past six months. For each, research the format, add it to the registry, and process any held content. Assign a dedicated maintainer to the registry if one does not exist. Increase the review cadence temporarily until the backlog is cleared.

  3. If a format specification is lost: Attempt to recover from backup or redundant copies. If no copy is available, attempt to reconstruct the specification from existing content in the institution that uses the format (reverse engineering). If the format is still encountered, escalate the need to acquire the specification during the next import from an external source that might carry it. Document the gap in the registry.

  4. If a polyglot file has been admitted: Identify the file in production. Examine it under all identified format interpretations. If any interpretation reveals dangerous or unwanted content, quarantine the file retroactively. Update the format identification pipeline to detect polyglot patterns for the combination discovered. Add polyglot detection rules to the registry entries for the involved formats.

Evolution Path

  • Years 0-5: The Format Registry is seeded with the formats the institution expects to encounter most frequently. Expect the registry to grow rapidly as actual imports reveal formats not initially anticipated. Format identification tooling is basic: file extension, magic byte lookup, and manual inspection. Focus on building the registry discipline -- every new format encounter results in a registry entry.

  • Years 5-15: The registry stabilizes as the common format landscape becomes well-mapped. Attention shifts to format viability monitoring: which formats are declining? Which specifications should be archived before they become unavailable? The format identification tooling matures, with automated structural analysis for high-frequency formats.

  • Years 15-30: Historical format knowledge becomes increasingly valuable. Content may arrive from federated nodes or archival media in formats that are decades old. The registry's historical entries -- formats marked OBSOLETE or HISTORICAL_ONLY -- become operational necessities, not archival curiosities. The institution's ability to recognize and process old formats is a competitive advantage.

  • Years 30-50+: The Format Registry may be one of the institution's most unique assets: a self-contained, locally maintained knowledge base of file format specifications spanning decades. Formats that the outside world has forgotten may still be understood within the institution. This long memory is part of the institution's value proposition.

Commentary Section

This section is reserved for dated entries by current and future operators.

2026-02-17 -- Founding Entry: The initial Format Registry should be seeded pragmatically, not ambitiously. Start with the formats the institution actually uses and expects to receive: plain text (UTF-8), PDF/A, PNG, JPEG, HTML, CSV, JSON, SQLite, ZIP, TAR. Add formats as they are encountered. Do not attempt to pre-populate the registry with every format known to computing -- that way lies an unmaintainable database of entries that have never been operationally tested.

The trust tier system is deliberately coarse (four tiers). There will be pressure to add more granularity -- sub-tiers, numerical scores, weighted assessments. Resist this until operational experience demonstrates that four tiers are insufficient. Complexity in classification systems tends to grow faster than the ability to maintain it. The trust tier is a starting point for the validation pipeline, not a final judgment.

One dissenting note for the record: there is an argument that the Format Registry should not include a trust tier at all -- that trust should be assessed entirely by the Trust Scoring Framework (D18-008) and that baking trust into the format registry creates a redundant and potentially conflicting trust assessment. The counter-argument is that format-level trust is a property of the format itself (open vs. proprietary, static vs. executable) and is distinct from content-level or source-level trust. Both arguments have merit. Future maintainers should evaluate whether the dual-trust approach creates confusion.
