D13-008 — Data Format Migration and Preservation

Document ID
D13-008
Title
Data Format Migration and Preservation
Domain
13 -- Evolution & Adaptation
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
ETH-001, CON-001, D13-001, D13-002, D13-006, D13-007, D6-001, D6-002
Depended Upon By
D13-009, D13-013. Referenced by any article involving data storage, archival, or format decisions.

Purpose

This article defines how data is migrated from one format to another without loss of content, structure, or meaning. Data outlives every other component of the institution. Hardware is replaced every five to ten years. Software is replaced every three to seven years. But data -- the actual records, documents, configurations, and institutional knowledge -- must survive for the entire lifetime of the institution. The format in which data is stored is the vessel that carries it through time, and when that vessel becomes fragile, the data must be transferred to a new vessel without spilling a drop.

D13-001 R-EVL-04 mandates that all data must be stored in formats that can be read without the originating software. D13-002 Section 4.4 provides the operational procedure for format conversions. This article goes deeper: it addresses how to assess format obsolescence risk before it becomes acute, how to select canonical formats that maximize longevity, how to verify that a conversion preserved everything that matters, and how to maintain a format genealogy that traces the history of every data set through its format transformations.

Data format migration is the highest-risk category of evolution activity because it touches the institution's irreplaceable asset: its data. A failed hardware swap can be retried with another piece of hardware. A failed software migration can be rolled back. A failed data format conversion that goes undetected can corrupt data permanently. This article establishes the safeguards that prevent that outcome.

Scope

In scope: - Format obsolescence risk assessment: how to determine when a format is approaching end of viability. - Canonical format selection criteria: how to choose formats that maximize longevity and interoperability. - The format conversion pipeline: the step-by-step process for converting data between formats. - Conversion verification methods: how to confirm that converted data is equivalent to the original. - The format genealogy record: tracking the history of format transformations for each data set. - The format watch list: monitoring formats in use for signs of approaching obsolescence. - Round-trip conversion testing as a verification technique.

Out of scope: - Specific format recommendations (these change with the technology landscape). - Generic data backup procedures (D6-002). - Documentation format migration (D13-009 handles documentation-specific concerns). - Data integrity verification for non-migration purposes (D6-001).

Background

3.1 The Format Mortality Problem

Data formats die. Not suddenly, like hardware, but gradually, like languages. First, the primary authoring tool stops being developed. Then, alternative tools stop supporting the format. Then, the specification becomes hard to find. Then, the only readers left are specialized archival tools that may or may not handle every feature of the format. Eventually, the format is effectively dead: the data it contains is theoretically accessible but practically unreachable without extraordinary effort.

The institution cannot afford to let its data follow this trajectory. Format mortality must be anticipated and preempted. Data must be migrated to living formats while the dying format can still be read reliably, not after reading it has become a forensic exercise.

3.2 The Fidelity Challenge

Not all format conversions are lossless. When a rich format is converted to a simpler one, information may be lost. When a structured format is converted to a flat one, relationships may be broken. When metadata-rich formats are converted to metadata-poor formats, context disappears. The fidelity challenge is to understand, before conversion, exactly what will be preserved and what will not -- and to ensure that what is lost is either unimportant or preserved through alternative means.

3.3 The Canonical Format Principle

Rather than maintaining data in whatever format it was originally created in, the institution designates canonical formats for each data category. Canonical formats are selected for durability, openness, and self-description. All data is stored in or converted to canonical formats as soon as practicable after acquisition. This reduces the institution's format surface area -- the number of distinct formats it must support -- and concentrates preservation effort on a manageable set of well-understood formats.

System Model

4.1 Format Obsolescence Risk Assessment

Every format in active use is assessed for obsolescence risk on an annual basis during the institutional review (OPS-001). The assessment uses five indicators:

IndicatorLow RiskMedium RiskHigh Risk
Specification availabilityOpen standard, freely availablePublished but maintained by single entityProprietary, undocumented, or no longer published
Tool supportMultiple active tools read/write formatFew tools, declining communitySingle tool or no actively maintained tools
Community adoptionWidely used, active communityNiche use, stable communityDeclining use, shrinking community
Institutional dependencyUsed for Tier 3 data onlyUsed for Tier 2 dataUsed for Tier 1 data
Conversion path availabilityWell-documented conversion to alternativesConversion possible but complexNo known reliable conversion path

Formats scoring high risk on any two indicators, or high risk on institutional dependency alone, are placed on the format watch list and a migration plan is initiated.

4.2 Canonical Format Selection Criteria

When selecting a canonical format for a data category, the following criteria apply in order of priority:

  1. Openness. The format specification must be publicly available and not encumbered by patents, licenses, or proprietary restrictions that could limit future use.
  2. Self-description. The format must carry enough metadata within itself that a reader unfamiliar with the institution can determine what the data represents. Formats that depend on external schemas or context are less durable than self-describing formats.
  3. Simplicity. Simpler formats are more durable because they are easier to implement, easier to verify, and less likely to have edge cases that cause problems. Between two formats that meet other criteria equally, prefer the simpler one.
  4. Tool independence. The format must be readable by multiple independent tools. A format that can only be read by one software package is a single point of failure.
  5. Human readability. Preferred but not required. Formats that can be read by a human with a text editor (plain text, CSV, HTML, XML) are more durable than binary formats because they can be partially recovered even without specialized tools.
  6. Structural fidelity. The format must be capable of representing the data's structure, relationships, and metadata without loss. A format that meets all other criteria but cannot represent the data faithfully is unsuitable.

4.3 The Format Conversion Pipeline

Every format conversion follows this pipeline:

Stage 1: Pre-conversion inventory. Generate a complete manifest of all files to be converted: file paths, sizes, checksums, and format-specific metadata. Record the total count. This manifest is the baseline against which conversion success is measured.

Stage 2: Conversion tool validation. Select the conversion tool. Test it on a representative sample (minimum 5% of files or 50 files, whichever is greater). Verify output against the original using the verification methods in Section 4.4. If the tool fails validation, select an alternative or investigate the failures before proceeding.

Stage 3: Full conversion. Convert all files. For each file, record: source path, destination path, source checksum, destination checksum, conversion status (success, warning, failure), and any conversion notes. Produce a conversion log.

Stage 4: Verification. Verify the converted files using at least two of the methods in Section 4.4. Record verification results in the conversion log.

Stage 5: Retention and archival. Retain the original files for the stabilization period defined in D13-002 (90 days for Tier 1 data, 30 days for Tier 2, 7 days for Tier 3). After stabilization, archive the originals -- do not delete them. Originals are retained as permanent archival copies per D13-001 R-EVL-07.

4.4 Conversion Verification Methods

At least two of the following methods must be used for every conversion:

  • Content comparison. Compare the text content of the source and destination files. For text-based formats, this can be automated with diff tools. For binary formats, extract content to text and compare.
  • Structural validation. Verify that the converted file is a valid instance of the target format. Use format-specific validation tools where available.
  • Round-trip conversion. Convert the data from format A to format B, then back from format B to format A. Compare the round-tripped file with the original. Differences indicate potential fidelity loss.
  • Visual or functional inspection. Open the converted file in a tool that reads the target format and verify that it appears and functions correctly. This catches semantic errors that automated tools may miss.
  • Metadata comparison. Verify that metadata (creation date, author, title, tags, relationships) has been preserved in the conversion.

4.5 The Format Genealogy Record

For each data set, the institution maintains a format genealogy record that tracks every format transformation the data has undergone. The record includes: original format, each intermediate format, current format, dates of each transformation, tools used for each transformation, and verification results for each transformation. The genealogy record is stored alongside the data it describes and is itself stored in a canonical format (plain text).

Rules & Constraints

  • R-DFM-01: All institutional data must be stored in canonical formats or have a documented, tested conversion path to a canonical format. Data in non-canonical formats must be converted within 12 months of acquisition per D13-001 R-EVL-04.
  • R-DFM-02: Format obsolescence risk assessment must be conducted annually for every format in active use. Formats at high risk must have an active migration plan within 6 months of the assessment.
  • R-DFM-03: Every format conversion must follow the conversion pipeline (Section 4.3) and must use at least two verification methods (Section 4.4). Conversions that fail verification must not overwrite or replace the original files.
  • R-DFM-04: Original files must be retained as permanent archival copies after conversion. No original data may be deleted after format migration. This is an absolute rule with no exceptions for Tier 1 data and a Tier 2 exception process for Tier 2 and Tier 3 data.
  • R-DFM-05: The format genealogy record must be maintained for every data set that has undergone format conversion. The record must be stored alongside the data and must be updated with each conversion.
  • R-DFM-06: Canonical format selection must be reviewed every 5 years or when a format's obsolescence risk assessment changes from low to medium risk, whichever comes first.
  • R-DFM-07: Conversion tools are Tier 2 data artifacts. They must be preserved in the institutional archive alongside their documentation. They must be tested annually to verify they still function correctly.

Failure Modes

  • Silent conversion error. The conversion tool produces output that appears valid but contains subtle errors -- truncated fields, lost metadata, reformatted content, broken references. The error is not detected during verification and is discovered only later when the data is actually used. Mitigation: using multiple verification methods (R-DFM-03) reduces the chance that all methods miss the same error. Round-trip conversion is particularly effective at detecting subtle errors.
  • Format watch list neglect. The annual format review is performed perfunctorily or skipped. A format that is approaching obsolescence is not placed on the watch list. By the time the obsolescence is recognized, reliable conversion tools are no longer available. Mitigation: R-DFM-02 mandates annual assessment. The format watch list review is a checklist item in the annual institutional review (OPS-001).
  • Canonical format failure. A format designated as canonical proves less durable than expected. The format itself begins to show signs of obsolescence. Mitigation: R-DFM-06 mandates periodic reassessment of canonical formats. The canonical format selection criteria (Section 4.2) favor formats with the broadest possible support base, reducing the likelihood of this failure.
  • Conversion tool dependency. The institution relies on a single conversion tool that becomes unavailable. No alternative tool can perform the conversion. Mitigation: R-DFM-07 preserves conversion tools in the archive. Tool independence is a canonical format selection criterion (Section 4.2 item 4), reducing the likelihood that only one tool can perform a given conversion.
  • Genealogy record loss. The format genealogy record for a data set is lost or not maintained. The operator cannot determine the provenance of the data or the transformations it has undergone. Mitigation: R-DFM-05 mandates genealogy record maintenance. Storing the record alongside the data ensures that the record and the data travel together through all institutional operations.

Recovery Procedures

  1. If a silent conversion error is discovered: Retrieve the original files from the permanent archive. Re-execute the conversion with a corrected tool or procedure. Verify the new conversion rigorously. If the original files are not available, assess the extent of the error and determine whether the data can be recovered from the corrupted conversion. Document the error and update conversion procedures to prevent recurrence.
  2. If a format has become obsolete without a migration plan: Assess whether any conversion tools still exist, even experimental or unmaintained ones. If tools exist, test them immediately and begin conversion of all data in the obsolete format. If no tools exist, investigate whether the format specification is available and whether a conversion tool can be developed locally. Prioritize Tier 1 data for recovery. Document the failure and its causes.
  3. If a canonical format is failing: Begin an emergency canonical format selection process. Identify the best available replacement. Initiate conversion of all data from the failing canonical format to the new one. This is a high-priority evolution activity that should override non-critical work.
  4. If the genealogy record for a data set is missing: Reconstruct what can be reconstructed from conversion logs, file metadata, and institutional records. Mark the reconstructed record as "RECONSTRUCTED -- original genealogy not maintained." Going forward, maintain the genealogy record rigorously for this data set.

Evolution Path

  • Years 0-5: The institution's data formats are current and well-supported. Use this period to establish canonical formats, perform the first format risk assessments, and build the genealogy records. Most data will be in its original format. The first conversions may be pre-emptive -- converting data from proprietary formats to canonical open formats as a hygiene measure.
  • Years 5-15: The first format obsolescence events are detected. Some formats chosen at founding are showing signs of decline. The conversion pipeline is tested in earnest. The genealogy records begin to accumulate transformation histories. Lessons from the first conversions inform refinements to the verification methods.
  • Years 15-30: Multiple format generations have been navigated. Some data has been converted twice -- from its original format to the first canonical format, then to a revised canonical format when the first one aged. The genealogy records are substantive documents that trace each data set's journey through format changes. The format watch list is a critical early-warning system that has prevented multiple crises.
  • Years 30-50+: Data format preservation is a mature practice. The institution's data has survived multiple format paradigm shifts. The genealogy records are among the most valuable institutional artifacts, documenting not just what the data is but how it has been carried through decades of technological change. The principle that data outlives formats has been proven repeatedly by experience.

Founder Commentary

This section is reserved for dated entries by current and future operators.

2026-02-17 -- Founding Entry: Data formats feel permanent when they are new. I cannot imagine a world where UTF-8 text files or HTML documents are unreadable. But I also know that someone in 1990 could not imagine a world where WordStar documents or Lotus 1-2-3 spreadsheets were unreadable, and yet that world arrived. The canonical format principle is my hedge against this inevitability: by concentrating institutional data into a small number of well-chosen formats, I reduce the surface area that needs protection. And by maintaining genealogy records, I ensure that even if a format does become unreadable, the history of what it was and how it was transformed is never lost. The most important safeguard in this article is the permanent retention of originals after conversion. It is storage-expensive but invaluable. If a conversion turns out to be flawed, the original is the only path back. Deleting originals after conversion is saving pennies and risking the library.
