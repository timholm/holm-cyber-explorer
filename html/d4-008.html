D4-008 — Internal Network Topology

Document ID
D4-008
Domain
4 -- Infrastructure & Power
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
D4-001, D4-007, SEC-001, SEC-003
Depended Upon By
Domain 5 (platform relies on network connectivity), D4-013 (documentation standards for network diagrams)

Purpose

This article defines the physical and logical network topology of the holm.chat Documentation Institution's internal network. The network is the nervous system of the institution: it connects computing nodes, enables data replication, supports monitoring, and facilitates every inter-system communication. Unlike the networks of conventional organizations, this network has no gateway, no router to the outside world, and no wireless segments. It is an island by design, and this article defines the shape of that island.

The purpose of this article is threefold. First, it establishes the design principles that govern network topology decisions, ensuring that every connection serves a documented purpose and that the absence of a connection is as deliberate as its presence. Second, it provides the concrete specifications for cabling, addressing, segmentation, and documentation that the operator follows when building or modifying the network. Third, it creates the authoritative reference against which the physical network is verified during every inspection.

A network that is not documented is a network that cannot be verified. A network that cannot be verified cannot be trusted. This article ensures the institutional network is always documented, always verifiable, and always trusted.

Scope

This article covers:

  • Physical network design: topology, cabling standards, switch selection, port assignments, and cable routing.
  • Logical network design: IP addressing scheme, network segmentation, DNS or host resolution, and service discovery.
  • The quarantine isolation boundary: how the quarantine machine remains physically separated from the institutional network.
  • Network documentation requirements: diagrams, cable schedules, port maps, and change logs.
  • Network expansion principles: how to add nodes, segments, or switches without compromising the design.
  • Physical cable management: routing, labeling, protection, and maintenance.

This article does not cover:

  • Logical network services (firewalls, traffic shaping, application-level protocols) -- see Domain 5.
  • Network hardware lifecycle and replacement -- see D4-004.
  • Network security policy and threat modeling -- see SEC-001 and SEC-003.
  • The content import pathway from quarantine to institutional systems -- see D18-001 and D18-002.

Background

Most institutional networks are designed to connect an organization to the outside world. The design priorities are bandwidth to the internet, segmentation for security against external threats, and scalability for growing user populations. None of these priorities apply here.

The institution's network exists solely to connect a small number of machines to each other. There is no internet connection. There are no external users. There is no need for a firewall because there is no boundary between inside and outside -- there is only inside. The network's design priorities are therefore different: simplicity, reliability, verifiability, and absolute physical isolation from any external network.

D4-001 establishes the foundational network principles: wired only, no wireless, physically isolated, documented. This article takes those principles and translates them into a complete network design. SEC-001 and SEC-003 define the trust boundaries that the network must implement physically -- most critically, the absolute separation between the institutional network and any non-institutional system, including the quarantine machine.

The simplicity of an air-gapped internal network is its greatest strength. There are no routing tables to misconfigure, no firewall rules to accidentally open, no VPN tunnels to be exploited. There is cable, there are switches, and there are machines. If a cable is not plugged in, the machine is not on the network. This physical simplicity makes the network auditable by visual inspection -- a property that no software-defined network can match.

System Model

8.1 Physical Topology

The institutional network uses a star topology centered on one or more Ethernet switches. Every machine connects to a switch port via a dedicated cable run. There are no daisy-chains, no hubs, and no bus topologies. The star topology provides several properties critical to the institution:

  • Fault isolation: A cable failure or port failure affects only one machine. All other machines continue to communicate.
  • Auditability: Every connection is a single cable between a known machine and a known switch port. The network can be verified by tracing cables.
  • Manageability: Machines can be added or removed by plugging or unplugging a single cable, with no impact on other connections.

Switch architecture: For installations of ten machines or fewer, a single managed switch is sufficient. For larger installations, multiple switches are connected via dedicated uplink cables (trunk connections). The switch fabric should provide sufficient bandwidth for the institution's workload, which is dominated by backup and replication traffic. Gigabit Ethernet (1 Gbps) is the minimum standard; 10 Gbps uplinks between switches are recommended if multi-switch topologies are deployed.

Cable specifications: All cables are shielded or unshielded twisted pair Ethernet, Category 6 (Cat6) or Category 6a (Cat6a) minimum. Cat6 supports Gigabit Ethernet at runs up to 100 meters. Cat6a supports 10 Gigabit Ethernet at runs up to 100 meters. All cables are solid-core copper for permanent runs and stranded-core copper for patch cables. Fiber optic is permitted for inter-switch uplinks in multi-room installations where copper's 100-meter limit is a constraint, but copper is preferred for simplicity and repairability.

8.2 Logical Topology

IP addressing: The network uses the private address space 10.0.0.0/16. This provides 65,534 usable addresses, which far exceeds any plausible institutional scale. All addresses are statically assigned. No DHCP server is used. Static addressing ensures that every machine's address is known, documented, and permanent.

The address space is divided into segments by function:

SegmentAddress RangePurpose
Core Services10.0.1.0/24Primary servers: main document server, primary backup server, monitoring server
Storage10.0.2.0/24Dedicated storage nodes, NAS devices, archive servers
Workstations10.0.3.0/24Operator workstations, administrative machines
Infrastructure10.0.4.0/24Switches (management interfaces), UPS network cards, environmental sensors with network capability
Reserved10.0.10.0/24 through 10.0.255.0/24Reserved for future expansion

Name resolution: A local DNS server running on the primary server provides name resolution for the institutional network. The DNS server contains only entries for institutional machines. It has no forwarders, no root hints, and no mechanism to resolve external names. As a fallback, each machine maintains a static /etc/hosts file with entries for all institutional machines. The hosts file is updated whenever a machine is added or removed.

8.3 Network Segmentation

For installations of five machines or fewer, a flat network (all machines on a single switch and single subnet) is acceptable. The overhead of VLAN configuration outweighs the benefit at this scale.

For installations of six machines or more, logical segmentation via VLANs is recommended. Each segment in the addressing table above maps to a VLAN. Inter-VLAN traffic is handled by the switch's Layer 3 capability or by a dedicated routing instance on the primary server. Segmentation provides:

  • Traffic isolation: backup traffic on the storage segment does not contend with operator workstation traffic.
  • Access control: if VLAN access control lists are configured, workstations cannot directly access storage nodes except through the document server.
  • Monitoring: per-VLAN traffic statistics simplify performance monitoring and anomaly detection.

8.4 The Quarantine Boundary

The quarantine machine (per D18-002) is never connected to any institutional network segment by any means. It has no Ethernet cable connected to any institutional switch. It is not in any VLAN. It has no IP address in the institutional address space. The quarantine machine is a physically isolated system that communicates with the institutional network only via the operator carrying a dedicated USB transfer medium between the two.

The quarantine machine should ideally be in a different physical area of the equipment room, with its cable runs and power connections clearly distinct from the institutional network's infrastructure. Visual distinction prevents accidental connection.

8.5 Cable Management

All cable runs are labeled at both ends with the source machine name and the destination switch port. Labels use printed adhesive labels or permanent marker on cable ties; handwritten labels directly on cables are not acceptable as they fade and smear. Cable runs follow dedicated pathways (cable trays, conduit, or wall-mounted clips). Cables do not run across floors, drape over equipment, or share pathways with power cables (to avoid electromagnetic interference). Cable pathways are documented on the network diagram with route descriptions.

Rules & Constraints

  • R-D4-20: Wired Only. All network connections within the institution are wired Ethernet. No wireless access points, no Bluetooth bridges, no infrared links, no powerline networking adapters. Any device found to be creating a wireless network signal is immediately disconnected and investigated.
  • R-D4-21: Static Addressing. All IP addresses are statically assigned and documented. No DHCP server operates on the institutional network. If a machine is observed with an undocumented IP address, it is immediately disconnected and investigated.
  • R-D4-22: Network Changes Are Documented Before Execution. Any change to the network topology -- adding a machine, removing a machine, changing a cable run, changing an IP address, adding a switch -- must be documented in the network change log before the change is physically made. The network diagram is updated within 24 hours of any change.
  • R-D4-23: Quarantine Isolation Is Physical. The quarantine machine's isolation from the institutional network is enforced by the absence of any physical connection, not by software configuration. During every quarterly infrastructure inspection, the operator physically verifies that no cable connects the quarantine machine to any institutional switch.
  • R-D4-24: Minimum Cable Standard. All new cable installations use Cat6 or better. Existing Cat5e installations may remain in service until replaced, but new runs must meet the Cat6 minimum.

Failure Modes

  • Switch failure. The central switch fails, severing all network communication. In a single-switch topology, this halts all inter-machine operations. Mitigation: maintain a cold spare switch. Configure and document the spare so it can be deployed by swapping cables. The spare switch should be tested annually by temporarily substituting it for the production switch.
  • Cable degradation. Over years, cables degrade: insulation cracks from heat, UV, or mechanical stress; connectors corrode in humid environments; strain reliefs fail from repeated movement. Network errors increase gradually, causing intermittent connectivity failures that are difficult to diagnose. Mitigation: monthly visual cable inspection. Replace cables at the first sign of physical damage. Test cables annually with a cable tester to verify they meet Cat6 specifications.
  • Accidental external connection. A cable is accidentally connected from an institutional switch to a non-institutional device (a personal laptop, an external network, or the quarantine machine). Mitigation: all switch ports not in active use are administratively disabled on managed switches. All cables are labeled. The quarterly audit physically traces every connected cable from end to end.
  • IP address conflict. Two machines are assigned the same IP address, causing intermittent network failures for both. Mitigation: the IP address register is the authoritative source. Before assigning any address, the register is consulted. The register is maintained in a single location with a defined update process.
  • Network documentation drift. The network diagram and cable schedule fall out of date as changes are made without updating documentation. The operator can no longer verify the network by comparing documentation to physical reality. Mitigation: R-D4-22 requires documentation before changes are made. The quarterly audit compares the physical network to the documented network and flags discrepancies.

Recovery Procedures

  1. If the primary switch fails: Retrieve the cold spare switch from inventory. Transfer the configuration documentation for the failed switch. Reconnect all cables to the spare switch, following the port assignment documentation. Verify connectivity to each machine. Update the network diagram to reflect the new switch. Order a replacement spare switch within 7 days.
  2. If a cable fails: Identify the failed cable by tracing the reported connectivity problem to a specific cable run. Replace the cable with a spare of the same category and length. Label both ends. Test the new cable. Update the cable schedule if the cable route has changed. Retain the failed cable for inspection to determine the cause of failure.
  3. If an unauthorized connection is discovered: Disconnect the unauthorized cable immediately. Determine what was connected. If the connection was to a non-institutional device or external network, treat this as a potential security incident per SEC-001: assess what data may have been exposed, determine the duration of the unauthorized connection, and follow incident response procedures. Document the incident and implement measures to prevent recurrence (disable unused ports, improve labeling, add physical port locks).
  4. If the network diagram is found to be inaccurate: Conduct an immediate full physical audit: trace every cable, verify every connection, verify every IP address. Rebuild the network diagram from scratch based on physical reality. Identify and document all discrepancies between the old diagram and the physical network. Investigate the root cause of the documentation failure and address it.

Evolution Path

  • Years 0-5: The network is small -- likely three to five machines connected to a single switch. The topology is simple enough to hold in one's head, but it must be documented anyway, because the person who inherits this institution may not be the person who built it. Establish the documentation habits now that will scale later.
  • Years 5-15: The network may grow as additional services, backup nodes, or archive systems are added. The transition from a single flat network to a segmented VLAN topology may occur during this period. Plan the transition carefully: document the target state before making changes, migrate one segment at a time, verify connectivity after each step.
  • Years 15-30: Ethernet standards may evolve. 10 Gbps may become the baseline, and new cable categories may replace Cat6a. The institution transitions cable standards gradually: new installations use the current standard; existing cable remains in service until it fails or until a segment is being rebuilt for other reasons. The topology principles -- star, wired-only, documented -- do not change even as the physical layer evolves.
  • Years 30-50: The network may have been rebuilt entirely one or more times as hardware is replaced and the physical site evolves. The addressing scheme, the segmentation model, and the documentation standards in this article survive those transitions. A future operator reading this article in 2076 should find the principles as applicable as they are today, even if every specific cable and switch has been replaced multiple times.

Founder Commentary

This section is reserved for dated entries by current and future operators.

2026-02-17 -- Founding Entry:

There is a temptation to dismiss the network topology article as unnecessary for an air-gapped institution with a handful of machines. After all, you plug cables into a switch and it works. What is there to document?

Everything. The network is the one component that connects all the others. A power failure affects the machines on that circuit. A drive failure affects the data on that drive. A network failure affects everything, because nothing can talk to anything else. And network failures are often subtle -- a flaky cable does not produce a dramatic failure, it produces intermittent errors that masquerade as application bugs, disk problems, or mysterious timeouts.

Document the network obsessively. Label every cable. Map every port. Record every IP address. When something goes wrong at two in the morning and you are tracing a connectivity problem, you will not remember which cable goes where. But if you labeled them, you will not need to remember. The documentation remembers for you. That is its purpose.
