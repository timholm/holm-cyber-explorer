D20-011 — Memory Taxonomy & Classification

Document ID
D20-011
Domain
20 — Institutional Memory
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
D20-001 (Philosophy)
Depended Upon By
D20-005 (Oral History), D20-003 (Timeline), D20-009 (Accessibility)

Purpose

This article defines the classification system for all institutional memory records. It provides a structured taxonomy of memory types, event categories, significance levels, and retention priorities that enables the institution to organize, prioritize, and navigate its memory systematically rather than as an undifferentiated mass of records.

Without a taxonomy, institutional memory is a pile. With a taxonomy, it is an organized archive. The difference is the difference between a library with a catalog and a warehouse with no inventory. Both contain the same books; only one allows you to find the book you need.

The taxonomy defined here serves multiple purposes: it guides the creation of new memory records (by telling the creator what type of record to create and what fields to fill), it structures the index (by providing the classification dimensions that D20-009 uses for indexing), it informs retention decisions (by specifying which categories of memory have which retention requirements), and it supports quality assessment (by providing Domain 19 with a framework for evaluating whether the memory system is complete).

Scope

This article covers:

  • The classification of memory records by type (what kind of record).
  • The classification of recorded events by category (what kind of event).
  • Significance levels for memory records (how important).
  • Retention tiers and their relationship to the three-tier architecture in D20-001.
  • The process for classifying new records.
  • The process for reclassifying existing records when circumstances change.
  • Alignment with the QA taxonomy in Domain 19.

This article does not cover:

  • The storage mechanism for records (covered by Infrastructure Domains).
  • The retention policies and sunset criteria themselves (see D20-012).
  • The indexing and search system that uses this taxonomy (see D20-009).
  • The content or format of individual record types (covered by their respective articles: D20-002 for decision logs, D20-005 for oral histories, etc.).

Background

3.1 Why Classification Matters

An institution generates many different kinds of records: decisions, events, observations, corrections, oral histories, lessons learned, audit results, personnel changes, system changes, and more. Each kind of record has different characteristics: different creation triggers, different audiences, different retention requirements, and different value curves over time.

Without classification, all records are treated equally. A routine configuration change is stored alongside a foundational governance decision. A minor maintenance note occupies the same status as an oral history from a departing founder. This equality is false and wasteful. It makes the important records harder to find by burying them among the routine ones. It applies inappropriate retention rules to records with different lifespans. It prevents the institution from prioritizing its memory maintenance efforts.

3.2 The Three-Tier Architecture Revisited

D20-001 introduced the three-tier memory architecture: Permanent (Tier 1), Active (Tier 2), and Transient (Tier 3). This article defines the criteria for assigning records to tiers, which D20-001 described only in general terms. The tier assignment is one dimension of classification; this article adds additional dimensions that allow finer-grained organization within each tier.

3.3 Alignment with Domain 19

Domain 19 (Quality Assurance) uses its own taxonomy for quality events. Memory classification must align with the QA taxonomy so that quality events can be properly stored, indexed, and retrieved as institutional memory. The alignment is described in Section 4.4 of this article.

System Model

4.1 Record Type Taxonomy

Record TypeDefinitionDefault TierExamples

Operational LogA routine record of system or process operationTier 2 or 3Backup logs, maintenance records, access logs

CategoryDescriptionTypical Significance

4.3 Significance Levels

Every memory record is assigned a significance level that influences its retention tier, its synchronization priority (D20-010), and its prominence in search results (D20-009):

LevelDefinitionRetentionExamples
FoundationalRecords that define the institution's identity, purpose, or structureTier 1, permanent, no exceptionsRoot documents, founding decisions, constitutional changes
SignificantRecords that materially affect institutional operations or directionTier 1, permanentMajor policy decisions, system architecture changes, personnel succession events
OperationalRecords that support ongoing operations but do not define institutional directionTier 2, reviewed annuallyRoutine audit results, maintenance logs, configuration changes
RoutineRecords created as part of normal operation with limited long-term valueTier 3, defined expirationDaily backup logs, cache states, temporary configurations

4.4 QA Taxonomy Alignment

Domain 19 classifies quality events using its own taxonomy (defects, audit findings, compliance results). These events must be classifiable within the Domain 20 taxonomy as well. The mapping is:

  • QA defect reports map to Record Type: Event, Category: Quality, Significance: variable (depends on defect severity).
  • QA audit results map to Record Type: Audit Result, Category: Quality, Significance: Operational or Significant depending on scope.
  • QA compliance findings map to Record Type: Event, Category: Quality or Governance, Significance: Significant.

This dual classification ensures that quality events are properly stored in institutional memory while remaining findable through both the QA-specific interface and the general memory search.

Rules & Constraints

  • R-D20-11-01: Every Record Is Classified. No memory record may be stored without classification along three dimensions: Record Type, Event Category, and Significance Level. Unclassified records are indexed as "UNCLASSIFIED" and are flagged for classification within 30 days.

  • R-D20-11-02: Classification Determines Retention. The retention tier of a record is determined by its significance level per the mapping in Section 4.3. Records may be promoted to a higher tier (Tier 3 to Tier 2, or Tier 2 to Tier 1) at any time. Demotion (moving a record to a lower tier) requires a documented decision per D20-012.

  • R-D20-11-03: The Taxonomy Is Extensible. New record types, event categories, or significance levels may be added to the taxonomy through a governance decision recorded in the decision log. Existing categories may not be removed (they may be deprecated with a successor mapping). The taxonomy is append-only, like the memory records it classifies.

  • R-D20-11-04: Reclassification Is Documented. If a record's classification changes (for example, a routine operational log entry is reclassified as significant because it later proved relevant to a major decision), the reclassification must be documented: original classification, new classification, reason for change, and who authorized the change.

  • R-D20-11-05: Classification Is Reviewable. Domain 19 (Quality Assurance) may audit the classification of memory records to verify that they are correctly classified. Systematic misclassification -- for example, significant decisions being classified as routine to avoid permanent retention -- is a quality defect and must be corrected.

Failure Modes

  • Classification neglect. Records are created but not classified. They accumulate in the "UNCLASSIFIED" category, making them unfindable by type, category, or significance. Severity: moderate. Likelihood: high (classification is easy to skip). Detection: the 30-day classification deadline and periodic audits. Mitigation: build classification into the record creation workflow so it is part of the same step, not a separate task.

  • Significance inflation. Everything is classified as "Foundational" or "Significant" to ensure permanent retention. The taxonomy becomes meaningless because it does not distinguish between truly important and merely cautious. Severity: moderate (degrades search and navigation). Likelihood: moderate. Detection: periodic distribution analysis of significance levels. Mitigation: the significance definitions in Section 4.3 are designed to be discriminating. If more than 20% of records in any period are classified as Foundational, the classification criteria may be being misapplied.

  • Significance deflation. Important records are classified as routine to reduce the burden of permanent retention. Critical decisions are lost when routine records expire. Severity: high. Likelihood: low (but consequences are severe). Detection: D19 audit of classification accuracy. Mitigation: R-D20-11-05 mandates reviewability; the classification review includes sampling of routine records to verify they are truly routine.

  • Taxonomy rigidity. The taxonomy does not evolve to accommodate new types of events or records. Records that do not fit existing categories are either forced into inappropriate categories or left unclassified. Severity: moderate. Likelihood: moderate over long time periods. Detection: accumulation of "UNCLASSIFIED" records or misfit classifications. Mitigation: R-D20-11-03 mandates extensibility; the taxonomy review is part of the annual memory system review.

Recovery Procedures

  1. If a backlog of unclassified records has accumulated: Conduct a classification sprint. Prioritize by creation date (oldest first) and by likely significance (records associated with major events or decisions first). Accept that some records may be difficult to classify retroactively; classify them to the best of current knowledge and note the late classification date.

  2. If significance inflation has been detected: Review the classification criteria. Provide training or examples that illustrate the distinction between Foundational, Significant, Operational, and Routine. Reclassify inflated records downward where appropriate, documenting the reclassification per R-D20-11-04.

  3. If significance deflation has caused loss: Determine what was lost by reviewing records that were retired under Tier 3 sunset policies. If the lost records can be restored from backup, restore them and reclassify. If they cannot be restored, document the loss as a KNOWLEDGE GAP in the decision log and implement safeguards (such as a review step before any Tier 3 records are retired).

  4. If the taxonomy needs extension: Propose the new categories through the governance process. Document the rationale for the new categories, provide examples of records that would be classified under them, and update this article's taxonomy tables. Reclassify any existing records that fit the new categories better than their current classification.

Evolution Path

  • Years 0-5: The taxonomy is new and being tested against real records. Some categories may prove too broad or too narrow. Refine the taxonomy based on the actual distribution of records. The goal is a taxonomy that feels natural to users -- where the correct classification is obvious for most records.

  • Years 5-15: The taxonomy stabilizes. New categories are added occasionally to accommodate new types of institutional activity. The relationship between classification and retention (D20-012) is tested as the first records reach their review dates.

  • Years 15-30: The taxonomy may need generational review. Terms that were clear to the founders may be opaque to current users. Ensure the taxonomy descriptions are written in plain language per D20-009's accessibility standards. Consider whether the significance scale is still calibrated correctly: does "Foundational" mean the same thing it meant 20 years ago?

  • Years 30-50: The taxonomy is a mature classification system with decades of records organized under it. Its value is proven by its utility: can a researcher find what they need by navigating the taxonomy? If so, it has succeeded. If not, it needs revision -- but revision must preserve backward compatibility, since decades of records are classified under the existing scheme.

Founder Commentary

This section is reserved for dated entries by current and future operators.

2026-02-17 -- Founding Entry: Taxonomy is one of those things that seems bureaucratic until you need it. The first time you search for "all decisions related to federation governance" and the system can answer that query instantly because every decision is classified by type and category, the taxonomy pays for itself.

I have tried to keep the taxonomy simple. The temptation to create a deeply nested hierarchical classification with dozens of sub-categories is real, but it would be unused. People will not classify records if the classification requires reading a manual. The three dimensions (Record Type, Event Category, Significance Level) should be sufficient for most purposes. If they are not, the taxonomy can be extended per R-D20-11-03, but the extension should be motivated by actual need, not theoretical completeness.

A note on significance levels: I expect "Operational" to be the most common classification, with "Routine" second, "Significant" third, and "Foundational" rare. If the distribution looks different from this, something is wrong -- either the definitions are not clear or the classifiers are not applying them correctly. Monitor the distribution and investigate anomalies.
