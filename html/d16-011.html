D16-011 — Offline & Degraded-Mode Interface Behavior

Document ID
D16-011
Domain
16 -- Interface & Navigation
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
D16-009 (Error States, Feedback & System Communication)
Depended Upon By
D16-013 (User Onboarding), D17-009 (Partition Tolerance)

Purpose

This article defines how the institution's interface behaves when the underlying system is degraded, when hardware has partially failed, when storage is limited, when processing power is constrained, or when the system is operating in conditions far below its design parameters. In an air-gapped, off-grid institution, degraded operation is not a theoretical edge case. It is a recurring reality. Power may be intermittent. Storage devices may fail. Hardware may age beyond its original specifications. The interface must remain useful under all of these conditions.

D16-001 establishes graceful degradation as a core principle (R-D16-06). This article operationalises that principle by defining, for each category of system degradation, what the interface must continue to provide, what it may reduce, and what it must communicate to the user about the system's current limitations. The user must never be left wondering whether the system is broken or merely constrained. Degraded operation must be visible, honest, and as functional as conditions permit.

This article is addressed to every person who designs interface behaviour that depends on system resources -- which is to say, every person who designs any interface behaviour at all. It is also addressed to the operator who must use the institution during an infrastructure emergency and needs to know what to expect.

Scope

This article covers:

  • Definition and classification of degraded-mode states.
  • Minimum viable interface functionality under each degradation class.
  • User communication requirements during degraded operation.
  • Interface adaptation strategies for resource-constrained environments.
  • The relationship between system health monitoring and interface state.
  • Paper-based fallback interface specifications.
  • Recovery transition: how the interface returns to normal after degradation ends.

This article does not cover:

  • The technical infrastructure that detects and reports system degradation (covered by Infrastructure domains).
  • Federation-level partition behaviour (covered by D17-009), though this article addresses the local interface implications of partition.
  • Disaster recovery procedures at the system level (covered by Domain 12).
  • Error message content and format (covered by D16-009; this article defines when degraded-mode messages appear, not how they are written).

Background

Why Degraded Mode Is Normal

In a connected, cloud-backed system, degradation is rare and brief. Redundant servers, automatic failover, and professional operations teams keep systems running near peak capacity almost all the time. The institution has none of these luxuries. It operates on hardware that may be old, on power that may be intermittent, with storage that may be partially failed, and with no external support infrastructure. Degraded operation is not an exception; it is a regular operating condition that may persist for days, weeks, or longer.

An interface designed only for optimal conditions will, in practice, be non-functional for a significant fraction of the institution's operational lifetime. Designing for degraded mode is not pessimism. It is realism. It is the same realism that leads a ship to carry lifeboats and a building to have emergency exits. The institution's interface must carry its own lifeboats.

The Principle of Progressive Enhancement

The degraded-mode strategy follows the principle of progressive enhancement: start with the minimum viable interface (which must work under the most constrained conditions), and add capabilities as resources permit. This is the inverse of the common approach in commercial software, which starts with the full-featured interface and tries to degrade gracefully. Progressive enhancement is more reliable because the baseline is tested and guaranteed, and additional features are additive rather than subtractive.

The Paper Fallback

D16-001 Principle R-D16-08 (Printability) anticipates the ultimate degradation scenario: a digital interface that is completely non-functional. The institution must have a paper-based fallback that allows basic navigation and content access. This is not a theoretical concern. Hardware fails. Power systems fail. In a fifty-year institution, there will be periods when the only functioning interface is printed documentation. This article defines what that fallback must include.

System Model

DEGRADATION CLASSIFICATION
============================

Class    Condition                    Interface Response
-----    ---------                    ------------------
D0       Normal operation             Full interface, all features
D1       Reduced processing power     Disable non-essential animations,
                                      reduce search result previews,
                                      defer background operations
D2       Limited storage              Read-only mode for content;
                                      disable import wizards;
                                      warn before any write operation
D3       Partial data loss            Navigate available content;
                                      clearly mark unavailable items;
                                      search returns partial results
                                      with explicit "incomplete" notice
D4       Minimal system               Static HTML rendering only;
                                      no search; manual navigation
                                      via table of contents and index;
                                      all dynamic features disabled
D5       Digital system offline       Paper-based fallback: printed
                                      table of contents, article index,
                                      and critical reference documents


PROGRESSIVE ENHANCEMENT STACK
================================
(Each layer adds to the one below it)

Layer 5: Full features     -- Discovery, serendipity, related content
Layer 4: Search            -- Full-text search, faceted navigation
Layer 3: Dynamic nav       -- Interactive navigation, breadcrumbs,
                              history, bookmarks
Layer 2: Static rendering  -- HTML rendering with typographic hierarchy,
                              basic hyperlinking, table of contents
Layer 1: Raw content       -- Plain text with structural markers
                              (headings as "# HEADING", lists as "- item")
Layer 0: Paper             -- Printed index + critical documents


SYSTEM HEALTH INDICATOR
=========================
  [Normal]  [Reduced]  [Limited]  [Degraded]  [Minimal]  [Offline]
     D0        D1         D2         D3          D4          D5

The current degradation class MUST be visible on every screen
at all times when the class is D1 or above.

Rules & Constraints

  • R-D16-11-01: Content Always Accessible. Regardless of degradation class, available content MUST be accessible through the interface. The method of access may change (from search to manual navigation to printed index), but the user MUST be able to reach any piece of content that physically exists in the storage system. No degradation mode may hide content that is present.

  • R-D16-11-02: Degradation Must Be Visible. The current degradation class MUST be displayed on every interface screen when the system is operating at D1 or above. The indicator MUST use both text and colour (to meet accessibility requirements). The indicator MUST link to a page that explains what the current degradation means, what features are affected, and what the user can still do.

  • R-D16-11-03: Read Before Write. In degraded modes (D2 and above), the interface MUST prioritise read operations over write operations. If resources are scarce, the ability to read existing content takes precedence over the ability to create or modify content. The user MUST be warned before any write operation that the system may not be able to complete it reliably.

  • R-D16-11-04: No Data Loss from Interface Degradation. Interface degradation MUST NOT cause data loss. If the interface reduces functionality, any user input that was accepted MUST be preserved. If an operation cannot be completed due to resource constraints, the user MUST be informed before the operation begins, not after they have invested effort in it.

  • R-D16-11-05: Honest Partiality. If search results, navigation listings, or content displays are incomplete due to degradation, the interface MUST explicitly state that the results are partial. "Showing 47 of approximately 230 results (search index partially unavailable)" is required. Showing 47 results with no indication that 183 are missing is prohibited.

  • R-D16-11-06: Paper Fallback Maintenance. The paper-based fallback materials (table of contents, article index, and critical reference documents) MUST be regenerated and reprinted whenever the institution's content changes significantly. "Significantly" is defined as: any addition, removal, or reorganisation of top-level sections in the information architecture, or the addition of more than 10% new content since the last printing. The printed materials MUST include the print date and a notice that digital content may be more current.

  • R-D16-11-07: Graceful Recovery. When the system recovers from a degraded state, the interface MUST transition back to higher capability levels without requiring user action. The user MUST be notified that the system has recovered. Any operations that were deferred or limited during degradation MUST be indicated as now available.

  • R-D16-11-08: Degraded-Mode Testing. Each degradation class (D1 through D4) MUST be testable in a controlled environment. The interface MUST support simulated degradation for testing purposes. Quarterly accessibility audits (per D16-006) MUST include at least one degraded-mode test scenario.

Failure Modes

  • Degradation denial. The system is degraded but the interface does not reflect this. Features that cannot function correctly are still presented as available. Users attempt actions that silently fail or produce incorrect results. Likelihood: moderate (especially when degradation is gradual). Severity: critical (users make decisions based on false information). Detection: health monitoring integration; automated comparison of available resources against feature requirements.

  • Catastrophic degradation skip. The system does not degrade gradually through D1-D2-D3 but jumps directly from D0 to D4 or D5 due to a sudden hardware failure. The intermediate degradation levels, which allow the user to adapt, are never experienced. The user goes from full functionality to minimal functionality instantaneously. Likelihood: moderate (sudden failures are common in aging hardware). Severity: high. Detection: this is a scenario that must be tested during D16-11-08 testing; the interface must handle non-sequential degradation transitions.

  • Paper fallback is out of date. The printed materials have not been updated since the last significant content change. The user consulting the paper fallback finds that the index does not match the current content organisation. Likelihood: high (printing is easy to defer). Severity: high during a D5 event (the paper is the only interface and it is wrong). Detection: automated tracking of content changes since last print date; alert when the threshold is exceeded.

  • Recovery oscillation. The system alternates between degraded and normal states (e.g., due to intermittent power). Each transition triggers notifications and interface reconfiguration. The user is bombarded with recovery and degradation messages. Likelihood: moderate in off-grid environments. Severity: medium (causes alert fatigue and disrupts workflow). Detection: detection of rapid state changes; dampening logic that requires a degradation or recovery state to persist for a minimum duration before the interface reconfigures.

  • Feature creep in degraded mode. Over time, features are added to the degraded-mode interface until the "degraded" mode consumes nearly as many resources as the normal mode, defeating the purpose. Likelihood: moderate. Severity: medium (the degraded mode becomes unreliable under the conditions it was designed for). Detection: periodic resource-usage profiling of each degradation level; comparison against defined resource budgets.

Recovery Procedures

  1. If the interface fails to indicate degradation: Audit the connection between system health monitoring and the interface degradation indicator. Verify that all degradation-triggering conditions are correctly mapped to degradation classes. If the mapping is incorrect, fix it and test against simulated degradation. If the health monitoring itself is not reporting degradation, escalate to Infrastructure domains.

  2. If the interface crashes during degradation instead of degrading gracefully: Identify the feature or component that crashed. Determine whether it was correctly configured for the current degradation class. If it was attempting to operate at a capability level above what the current resources support, fix the degradation mapping so it is disabled at the correct class. If it crashed due to a bug, fix the bug and add a degraded-mode test case for the specific scenario.

  3. If the paper fallback is needed and is out of date: Use whatever digital capabilities remain to generate an updated table of contents and article index. If no digital capabilities remain, use the outdated paper materials but annotate them manually with known discrepancies. Once digital capabilities are restored, regenerate and reprint the paper fallback immediately.

  4. If recovery oscillation is occurring: Implement a dampening threshold: the interface does not transition between degradation states unless the new state has been stable for at least 60 seconds. During oscillation, the interface should remain at the more degraded level and display a status message indicating that system stability has not yet been confirmed.

  5. If the system has been at D4 or D5 for an extended period: Document the extended outage in the D20 decision log. Conduct a post-outage review to determine whether the degraded-mode interface served its purpose. Identify any content that was inaccessible during the outage and determine whether it should be prioritised for the paper fallback. Update this article's Commentary Section with lessons learned.

Evolution Path

  • Years 0-5: Build the progressive enhancement stack from Layer 0 up. Test each degradation class independently. Generate the first paper fallback materials. Establish the simulated degradation testing framework. This is the most important period for degraded-mode design, because the patterns established now will govern how the interface behaves during every future crisis.

  • Years 5-15: Real degradation events will have occurred. Use each one as a learning opportunity. Update the degradation classification if real-world conditions reveal gaps. Refine the paper fallback materials based on any D5 events. Verify that the progressive enhancement stack still functions correctly as the full-featured interface evolves.

  • Years 15-30: Hardware will have been replaced at least once. Each hardware transition is a test of the degraded-mode design. If the new hardware has different failure characteristics, the degradation classification may need revision. The paper fallback will have been reprinted multiple times. Ensure the printing process is well-documented and can be executed by someone who did not design it.

  • Years 30-50: The degraded-mode interface should be one of the most battle-tested parts of the system. Its simplicity is its strength: the lower layers of the progressive enhancement stack have fewer dependencies and fewer things that can break. If the institution survives fifty years, it will be in part because the degraded-mode interface kept it accessible during the crises that inevitably occurred along the way.

Founder Commentary

This section is reserved for dated entries by current and future operators.

2026-02-17 -- Founding Entry: I am writing this article in an environment where degradation is not yet a concern. The hardware is new. The storage is ample. The power is stable. This is exactly the wrong time to design for degradation, because it is the time when degradation feels most theoretical. But it is also the only time when the design can be done thoughtfully, without the pressure of an ongoing emergency.

The progressive enhancement stack is the most important concept in this article. If the institution adopts it from the beginning -- building the static HTML layer first, then adding dynamic navigation, then adding search, then adding discovery features -- then each layer can be independently disabled under resource pressure without affecting the layers below it. If the institution instead builds a monolithic interface where all features are intertwined, degradation becomes all-or-nothing: either the full interface works, or nothing works.

The paper fallback will seem absurd to some readers. It is not absurd. I have personally experienced situations where the only usable documentation was a printed binder, because the digital systems that should have provided access were unavailable. In a fifty-year institution, this will happen more than once. The printed binder does not need to be comprehensive. It needs to be sufficient to locate and access critical information while the digital systems are being restored.
