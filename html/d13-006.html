D13-006 — Version Management and Compatibility Policy

Document ID
D13-006
Title
Version Management and Compatibility Policy
Domain
13 -- Evolution & Adaptation
Version
1.0.0
Date
2026-02-17
Status
Ratified
Depends On
ETH-001, CON-001, GOV-001, OPS-001, D13-001, D11-001, D11-002
Depended Upon By
D13-007, D13-008, D13-009, D13-010, D13-011, D13-012, D13-013. Referenced by all domains that track versioned artifacts.

Purpose

This article establishes the version numbering scheme, version tracking procedures, and forward and backward compatibility requirements for all systems, software, data formats, and documentation within the holm.chat Documentation Institution. Versioning is the language of change. Without a consistent version scheme, it is impossible to know what state a system is in, what changed between states, what is compatible with what, or when something was last updated.

D13-001 establishes the philosophy of managed change. This article provides the infrastructure for that management. Every component in the institution -- from the operating system to the documentation framework to individual data format specifications -- must have a version identifier that communicates its lifecycle status. The version scheme must be simple enough that a single operator can maintain it without overhead, robust enough to survive fifty years of continuous use, and expressive enough to distinguish between trivial changes and breaking ones.

This article also defines the compatibility policy: the rules governing whether a new version of a component is required to work with the old version's data and interfaces, and for how long. Compatibility is the bridge that makes evolution possible without disruption. Without compatibility guarantees, every version change is a potential crisis. With them, version changes become orderly transitions.

Scope

In scope: - The institutional version numbering scheme (major.minor.patch) and its semantic meaning. - Version lifecycle stages: active, deprecated, sunset, archived. - Version tracking: where version information is recorded and how it is maintained. - Forward compatibility: the obligation for new versions to read old data. - Backward compatibility: the obligation for old systems to handle new data gracefully. - Compatibility windows: how long old versions must be supported after a new version is released. - Version conflict resolution: what happens when components at different versions must interact. - Version numbering for documentation, configuration, and data formats.

Out of scope: - Specific migration procedures between versions (D13-002). - Hardware lifecycle tracking (D13-003, D11-002). - Document content revision tracking (D20-001 handles institutional memory; this article handles the numbering scheme). - Software sunset procedures (D13-004).

Background

3.1 The Problem of Unnamed Change

In personal computing, version management is typically informal. Files are overwritten. Configurations are changed in place. Software is updated and the old version vanishes. This works -- until it does not. When a problem appears, the operator cannot determine what changed. When a backup is restored, the operator cannot determine what version it represents. When a data file fails to open, the operator cannot determine whether the file or the software is at fault.

This institution cannot afford unnamed change. Every change must be traceable to a version increment. Every state of every significant component must be identifiable by its version number. The version number is the institution's way of asserting: "I know exactly what state this is, and I can distinguish it from every other state."

3.2 Semantic Versioning as a Foundation

The industry standard approach to version numbering -- semantic versioning (major.minor.patch) -- provides a solid foundation. The major version increments when backward-incompatible changes are introduced. The minor version increments when new functionality is added in a backward-compatible way. The patch version increments for backward-compatible fixes. This institution adopts semantic versioning with institutional-specific extensions for lifecycle status and compatibility guarantees.

3.3 Compatibility as an Engineering Discipline

Compatibility is not an afterthought. It is a design constraint that must be considered at the time a component is created, not when a new version is released. The principle from D13-001 -- data sovereignty, interface stability, modularity -- are the architectural foundations of compatibility. This article makes those principles concrete by specifying what compatibility means, how long it lasts, and what happens when it cannot be maintained.

System Model

4.1 The Version Numbering Scheme

All versioned artifacts in the institution use the format: MAJOR.MINOR.PATCH

MAJOR (integer, starting at 1): Incremented when a change breaks backward compatibility. A consumer of version 1.x.x data or interfaces cannot assume it will work with version 2.0.0 without migration. Major version increments are Tier 2 decisions under GOV-001.

MINOR (integer, starting at 0): Incremented when new capabilities are added without breaking existing capabilities. A consumer of version 1.2.x data should be able to consume version 1.3.0 data, though it may not use the new capabilities. Minor version increments are Tier 3 decisions.

PATCH (integer, starting at 0): Incremented for bug fixes, corrections, and clarifications that do not change capabilities. A consumer of version 1.2.3 should see no functional difference from version 1.2.4. Patch increments are operational decisions requiring no governance approval beyond documentation.

4.2 Version Lifecycle Stages

Every versioned artifact passes through four lifecycle stages:

Active. The current recommended version. Receiving updates (minor and patch increments). All new work should target this version. There is exactly one active major version for each artifact at any time.

Deprecated. A version that is still functional and supported but is no longer the recommended version. A newer major version has been released. The deprecated version receives only critical patches (security fixes, data integrity fixes). No new features. Deprecation is announced with a sunset date -- the date after which the deprecated version will no longer be supported. The minimum deprecation period is 12 months for major system components and 6 months for minor components.

Sunset. A version that is no longer supported. It may still function but will not receive patches. Data in this version's format should be migrated to the active version. Sunset is a defined event recorded in the institutional timeline (D20-003).

Archived. A version that has been fully retired. Its documentation is preserved in the institutional archive. Its data has been migrated. It no longer runs in any production capacity. Archive is the final state.

4.3 The Version Registry

The version registry is a document maintained as part of the asset registry (D11-002). It lists every versioned artifact in the institution with its current version number, lifecycle stage, and key dates (creation date, deprecation date if applicable, sunset date if applicable). The registry is updated whenever a version changes. It is reviewed during the annual institutional review (OPS-001).

The registry format is a plain-text table (to ensure durability per D13-001 R-EVL-04):

ARTIFACT                  CURRENT VERSION  STAGE       ACTIVE SINCE  DEPRECATED  SUNSET
operating-system          24.04.2          Active      2024-04-01    --          --
documentation-framework   1.0.0            Active      2026-02-16    --          --
backup-format             1.2.0            Active      2026-01-15    --          --
encryption-scheme         1.0.0            Active      2026-02-16    --          --

4.4 Forward Compatibility Requirements

Forward compatibility means that newer versions can consume data and interfaces produced by older versions. The rule is strict: every new version must be able to read data produced by any prior version within the same major version line. Version 1.5.0 must be able to read data produced by version 1.0.0. If a new minor or patch version cannot read old data, it is by definition a major version increment.

Across major version boundaries, forward compatibility is strongly recommended but not strictly required. Version 2.0.0 should include a migration path from version 1.x.x data, but it is not required to read version 1.x.x data natively. Instead, a conversion tool or procedure must be documented as part of the major version release.

4.5 Backward Compatibility Requirements

Backward compatibility means that older versions can handle data produced by newer versions gracefully. Full backward compatibility is not required -- an older version may not understand new features. But it must not crash, corrupt data, or silently lose information when encountering data from a newer minor version. At minimum, old versions must recognize that the data is from a newer version and report this clearly rather than failing silently.

Across major version boundaries, backward compatibility is not required but failure must be graceful: clear error messages, no data corruption, no silent degradation.

4.6 Compatibility Windows

When a new major version is released, the previous major version enters the deprecated stage. The compatibility window is the period during which both versions are supported and data can flow between them (via conversion tools). Minimum compatibility windows:

Artifact TypeMinimum Compatibility Window
Data formats (Tier 1 data)24 months
Data formats (Tier 2/3 data)12 months
System configurations12 months
Documentation formats18 months
Interface specifications12 months

During the compatibility window, conversion tools and migration procedures must be maintained and tested. At the end of the compatibility window, the deprecated version enters sunset.

Rules & Constraints

  • R-VMP-01: All versioned artifacts must use the MAJOR.MINOR.PATCH numbering scheme defined in Section 4.1. No alternative schemes are permitted without Tier 2 approval and documented justification.
  • R-VMP-02: Major version increments are Tier 2 decisions under GOV-001. They must be accompanied by a migration path from the previous major version and documented compatibility windows.
  • R-VMP-03: Forward compatibility within a major version line is mandatory. A newer minor or patch version must read data produced by any older version in the same major line.
  • R-VMP-04: The version registry must be maintained and reviewed at least annually. Every version change must be recorded within 7 days of the change.
  • R-VMP-05: Deprecated versions must have an explicit sunset date established at the time of deprecation. The sunset date must respect the minimum compatibility windows in Section 4.6.
  • R-VMP-06: No versioned artifact may transition directly from active to sunset. The deprecation stage is mandatory to provide transition time.
  • R-VMP-07: Archived versions must have their documentation preserved in the institutional archive. The documentation must include: version number, date range of active use, reason for archival, and location of any migration tools or procedures.
  • R-VMP-08: When version conflicts arise between components, the resolution must preserve data integrity above all other concerns. Performance degradation and feature loss are acceptable; data loss is not.

Failure Modes

  • Version drift. The version registry falls out of sync with actual deployed versions. The operator believes a component is at version 1.2.0 when it is actually at 1.3.1. Compatibility assumptions based on the registry are incorrect. Mitigation: R-VMP-04 mandates timely registry updates. The annual review includes a physical verification of deployed versions against the registry.
  • Silent incompatibility. A minor version update introduces a subtle incompatibility that is not detected because the change was classified as backward-compatible when it was not. Data produced by the new version cannot be read correctly by the old version. Mitigation: R-VMP-03 mandates forward compatibility testing. Pre-migration testing (D13-002) must include compatibility verification.
  • Compatibility window violation. A deprecated version is sunset before the compatibility window expires, leaving data stranded in the old format. Mitigation: R-VMP-05 mandates explicit sunset dates that respect minimum windows. R-VMP-06 prevents skipping the deprecation stage.
  • Version number exhaustion. Over decades, version numbers may grow large. Version 47.0.0 is unwieldy but functional. This is an aesthetic concern, not a functional one. The scheme does not require resetting numbers. Large numbers are a badge of institutional longevity, not a problem to solve.
  • Orphaned conversion tools. A conversion tool for migrating from version N to version N+1 is lost or becomes non-functional. Data in version N format cannot be migrated. Mitigation: conversion tools are treated as Tier 2 data artifacts and preserved in the institutional archive. They are tested annually as part of the compatibility window verification.

Recovery Procedures

  1. If the version registry is out of sync: Conduct a full audit. For each versioned artifact, determine the actually deployed version by examining the artifact directly (configuration files, software version commands, data format headers). Update the registry. Document the discrepancies and their causes. Implement procedural safeguards to prevent recurrence.
  2. If a silent incompatibility is discovered: Assess the scope of affected data. If data has been produced by the incompatible version, determine whether it can be converted. If the incompatibility was introduced in a minor version, this represents a version classification error -- the change should have been a major version increment. Reclassify, create a migration path, and update the registry.
  3. If a compatibility window has been violated: If the deprecated version's data still exists, restore the conversion tools from the archive and perform the migration. If the conversion tools are unavailable, treat this as a data format recovery event and use forensic techniques to reconstruct the migration path. Document the violation and enforce compliance going forward.
  4. If a conversion tool has been lost: Check the institutional archive for backup copies. If none exist, attempt to reconstruct the tool from documentation. If reconstruction is not possible, the data in the old format may need to be read by restoring the old version of the software from archive. This is a strong argument for R-VMP-07's requirement to preserve archived version documentation.

Evolution Path

  • Years 0-5: Most artifacts are at version 1.x.x. The version registry is small and easy to maintain. The compatibility policy is theoretical -- no major version transitions have occurred. Use this period to build the discipline of version tracking. Every change, no matter how small, gets a version increment.
  • Years 5-15: The first major version transitions occur. The compatibility windows are tested in practice. The conversion tools are used for the first time. Expect to discover that some compatibility windows are too short and some conversion tools are inadequate. Revise based on experience.
  • Years 15-30: The version registry has grown significantly. Multiple major version lines have been navigated. The archived versions section of the registry tells the story of the institution's technological evolution. Version management is routine.
  • Years 30-50+: Version numbers are large but meaningful. The registry is a comprehensive history of every significant change in the institution. Conversion tools spanning decades of versions are preserved in the archive. The version scheme itself has proven its durability -- or has been revised through the Evolution Decision Framework if it proved inadequate.

Founder Commentary

This section is reserved for dated entries by current and future operators.

2026-02-17 -- Founding Entry: Version management feels like bureaucratic overhead when you are a single operator maintaining a personal system. I know what version everything is because I installed it. But I will not always be the operator. And even if I am, my memory of what version I installed three years ago will fade. The version registry and the version numbering scheme are not for today's operator who remembers everything. They are for tomorrow's operator -- or today's operator in five years -- who needs to know exactly what state the institution is in without relying on memory. The discipline of incrementing a version number for every change is a small tax on the present that pays enormous dividends to the future. I have chosen semantic versioning because it communicates intent, not just sequence. A major version change tells the reader: something important shifted. A patch tells the reader: this is a minor correction. The number itself carries meaning, and meaning survives longer than memory.
